= CUI Labeling
:description: How UMRS implements Controlled Unclassified Information labeling using SELinux MCS categories.

Controlled Unclassified Information (CUI) is the United States government's single system for managing
unclassified information that requires safeguarding or dissemination controls.

Before CUI, agencies used over 100 ad hoc markings -- FOUO, SBU, LES, OPSEC -- with inconsistent
handling rules. Executive Order 13556 (2010) and 32 CFR Part 2002 replaced all of them with one
framework administered by the National Archives and Records Administration (NARA).

This module documents:

* The CUI category hierarchy and the distinction between CUI Basic and CUI Specified
* How UMRS maps CUI categories to MCS sensitivity labels
* The DoD CUI Registry and how it relates to the NARA master registry
* Reconciliation between our catalog and the authoritative DoD source
* The `umrs-label` tool for loading CUI definitions into a running system

== Why Machine-Readable Labels Matter

Most CUI breaches happen because a file ended up somewhere it should not be.
An email attachment forwarded to an unauthorized recipient.
A spreadsheet copied to an uncontrolled share.
A backup written to an unencrypted drive.

Every one of these failures has the same root cause: the protection was procedural, not structural.
The file itself carried no machine-readable indication of its sensitivity.
The filesystem had no mechanism to enforce boundaries.

For documented examples of what happens when these controls are absent — including federal
convictions, regulatory settlements, and environmental disasters — see the
xref:ROOT:foundations/history/case-studies-consolidated.adoc[high-assurance failure case studies].
Each case maps to a specific CUI category and identifies the NIST SP 800-53 controls that
were published, required, and not implemented.

SELinux MCS changes this.
SELinux attaches a security context to every kernel object -- files, processes, network sockets,
ports, and other resources -- not just files on disk.
The enforcement is pervasive: every access decision, on every object, is mediated by the kernel.

When a process attempts to open a file labeled `s0:c0,c90`, the kernel checks both sides:
the file's security context and the process's security context must both carry those categories.
Neither the file alone nor the process alone determines access -- the kernel evaluates the pair.
The check happens on every `open()` call, regardless of DAC permissions.
The label travels with the file through copy, move, and backup operations.

CUI labeling is the bridge between the compliance framework (NARA, DoD, CMMC) and the enforcement
mechanism (SELinux MCS). UMRS builds that bridge.

== Phase 1: Awareness and Custody

The first UMRS release operates under the SELinux targeted policy.
In targeted policy, MCS labels are present and functional, but the `unconfined_t` domain bypasses 
most MAC checks. This means:

* Labels are applied and visible -- `umrs-ls` shows them, `chcat -L` translates them
* Tools read and display label information correctly
* Operators learn the CUI category structure through hands-on vault exploration
* The system demonstrates custody: files are labeled, and the labeling is auditable

What Phase 1 does _not_ do is enforce mandatory separation between CUI categories.
An `unconfined_t` process can still read any file regardless of its MCS label.
This is by design -- targeted policy is the safe starting point for organizations that have never used MCS.

Phase 2 introduces MLS policy, where MCS category enforcement is mandatory and the `unconfined_t` domain does not exist.
Under MLS policy, every process operates within a defined sensitivity range and category set; the escape hatch that targeted policy permits is closed.
That is a different operational posture with different deployment requirements.

== CUI Marking Syntax

The standard CUI marking format uses the double-slash prefix:

----
CUI//Category1
CUI//Category1/Category2
CUI//Category1//LDC
----

The double slash (`//`) separates `CUI` from the first category and separates the last
category from a dissemination control.
A single slash (`/`) separates categories from each other.

Examples of correct CUI markings:

* `CUI//CTI` -- Controlled Technical Information (CUI Basic)
* `CUI//SP-CTI` -- Controlled Technical Information (CUI Specified)
* `CUI//INV` -- Investigation
* `CUI//INV/PRVCY` -- Investigation and Privacy (two independent categories, alphabetized)
* `CUI//SP-CTI/EXPT//NOFORN` -- Controlled Technical Information (Specified), Export Control, No Foreign Nationals

NOTE: Index groups (Law Enforcement, Defense, Intelligence, etc.) are organizational groupings
used for navigation in the NARA registry. They never appear in a banner marking.
`INV` is the Investigation category abbreviation; `LEI` (General Law Enforcement) is a
separate category. Neither is a subcategory of the other.

Categories that carry required dissemination controls append them after a second double slash:

* `CUI//SP-NNPI//NOFORN` -- Naval Nuclear Propulsion Information (Specified), No Foreign Nationals
