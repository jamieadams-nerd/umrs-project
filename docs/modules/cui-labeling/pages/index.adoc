= CUI Labeling
:description: How UMRS implements Controlled Unclassified Information labeling using SELinux MCS categories.

Controlled Unclassified Information (CUI) is the single system for managing unclassified information that requires safeguarding or dissemination controls.
Before CUI, agencies used over 100 ad hoc markings -- FOUO, SBU, LES, OPSEC -- with inconsistent handling rules.
Executive Order 13556 (2010) and 32 CFR Part 2002 replaced all of them with one framework administered by the National Archives and Records Administration (NARA).

UMRS brings CUI labeling into the filesystem.
Every CUI category maps to an SELinux Multi-Category Security (MCS) label.
When a file carries `CUI//LEI/INV`, the kernel enforces access -- not a policy document, not a training reminder, not a sticky note on a monitor.

This module documents:

* The CUI category hierarchy and the distinction between CUI Basic and CUI Specified
* How UMRS maps CUI categories to MCS sensitivity labels
* The DoD CUI Registry and how it relates to the NARA master registry
* Reconciliation between our catalog and the authoritative DoD source
* The `umrs-mcs` tool for loading CUI definitions into a running system

== Why Filesystem Labeling Matters

Most CUI breaches happen because a file ended up somewhere it should not be.
An email attachment forwarded to an unauthorized recipient.
A spreadsheet copied to an uncontrolled share.
A backup written to an unencrypted drive.

Every one of these failures has the same root cause: the protection was procedural, not structural.
The file itself carried no machine-readable indication of its sensitivity.
The filesystem had no mechanism to enforce boundaries.

SELinux MCS changes this.
A file labeled `s0:c100,c200` cannot be read by a process without those categories in its security context.
The check happens in the kernel, on every `open()` call, regardless of DAC permissions.
The label travels with the file through copy, move, and backup operations.

CUI labeling is the bridge between the compliance framework (NARA, DoD, CMMC) and the enforcement mechanism (SELinux MCS).
UMRS builds that bridge.

== Phase 1: Awareness and Custody

The first UMRS release operates under SELinux targeted policy.
In targeted policy, MCS labels are present and functional, but the `unconfined_t` domain bypasses most MAC checks.
This means:

* Labels are applied and visible -- `umrs-ls` shows them, `chcat -L` translates them
* Tools read and display label information correctly
* Operators learn the CUI category structure through hands-on vault exploration
* The system demonstrates custody: files are labeled, and the labeling is auditable

What Phase 1 does _not_ do is enforce mandatory separation between CUI categories.
An `unconfined_t` process can still read any file regardless of its MCS label.
This is by design -- targeted policy is the safe starting point for organizations that have never used MCS.

Phase 2 introduces MLS policy, where MCS enforcement is mandatory and `unconfined_t` does not exist.
That is a different operational posture with different deployment requirements.

== CUI Marking Syntax

The standard CUI marking format uses the double-slash prefix:

----
CUI//[Category Group]/[Category]
----

Examples:

* `CUI//LEI/INV` -- Law Enforcement, Investigation
* `CUI//AGR/AMNT` -- Agriculture, Ammonium Nitrate
* `CUI//INTEL/OPSEC` -- Intelligence, Operations Security

Categories without subcategories use the group abbreviation alone:

* `CUI//FNC` -- General Financial Information
* `CUI//GEO` -- Geodetic Product Information

Categories that carry additional dissemination controls append them:

* `CUI//DEF/NNPI//NOFORN` -- Naval Nuclear Propulsion Information, No Foreign Nationals
