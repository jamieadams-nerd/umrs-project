= Sealed Evidence Cache (SEC)
:toc: left
:description: Seal costly verification results with an ephemeral HMAC and a short TTL so that re-queries can be served from cache without repeating the full pipeline.

// Controls: NIST 800-53 SC-28 (protection at rest), SC-12 (key management),
//           SI-7 (integrity verification), NIST 800-218 SSDF PW.4

[WARNING]
====
This pattern is *required* and *not yet implemented* in the UMRS codebase.
The design is stable. Implementation should proceed before the OS detection pipeline
enters production use. See xref:index.adoc#open[Open — Awaiting Implementation].
====

== Overview

Some verification pipelines are expensive — they open files, check filesystem magic,
query package databases, compute digests, and resolve contradictions across multiple
evidence sources. Running such a pipeline on every re-query is wasteful when the
underlying inputs (kernel version, installed packages, OS release files) change
infrequently.

The Sealed Evidence Cache (SEC) pattern caches the verified result of an expensive
pipeline alongside a cryptographic seal. The seal binds the cached data to a specific
boot session and pipeline invocation. Re-queries within a short TTL verify the seal
and return the cached result — without re-running the pipeline. If the seal fails,
the cache is discarded, the pipeline re-runs, and the anomaly is logged.

The design is intentionally minimal: no persistent storage, no network, one cache
entry per caller context. The sealing key is ephemeral and zeroized on drop.

== The Threat

A verification pipeline that produces a trusted result creates a high-value target.
An adversary who can substitute a cached result can:

* Replace a T4 `IntegrityAnchored` result with a fabricated T3 or T2 result,
  downgrading the trust level of the host OS identification
* Inject a false `OsRelease` that names a different distribution or kernel version,
  causing policy decisions based on OS identity to apply the wrong controls
* Cause stale cached data to persist past an upgrade or configuration change, making
  the system believe it is still in a known-good state when it is not

Re-running the pipeline on every query closes the substitution window, but the
performance cost is prohibitive for a pipeline that reads multiple kernel attributes,
queries the RPM database, and computes SHA-256 digests. The SEC pattern retains the
performance benefit of caching while adding a cryptographic integrity check over the
cached data, preventing substitution without detection.

A second threat is cache persistence across boots. If a sealed result were stored on
disk, an attacker who controlled disk content before a reboot could pre-populate the
cache with a fabricated result that would pass seal verification. The SEC pattern
prevents this by deriving the sealing key from a boot-local value (`boot_id`) and the
process start time — values an attacker cannot know until after the system is running.

== The Pattern

[mermaid]
....
flowchart TD
    A([Re-query request]) --> B{Cache entry\npresent?}

    B -->|No| PIPE[Run detection pipeline]
    PIPE --> SEAL[Compute HMAC seal]
    SEAL --> STORE[Store: data + TrustLevel +\nevidence digest + TTL]
    STORE --> RET([Return result])

    B -->|Yes| TTL{TTL\nexpired?}
    TTL -->|Yes| DISCARD[Discard entry]
    DISCARD --> PIPE

    TTL -->|No| VERIFY{Verify\nHMAC seal}
    VERIFY -->|Valid| RET
    VERIFY -->|Invalid| LOG[Log anomaly\nlog::warn!]
    LOG --> DISCARD
....

=== Sealing Key

The sealing key is derived from two boot-local values:

* `boot_id` — a UUID generated by the kernel at boot, available at
  `/proc/sys/kernel/random/boot_id`. Changes on every reboot.
* Process start time — the monotonic start time of the current process, read from
  `/proc/self/stat`. Different for every process invocation.

Neither value is predictable before the system boots. Their combination produces a key
that is unique per boot-and-process-invocation pair and cannot be pre-computed by an
attacker with only disk access. The key is held in memory, never written to disk, and
zeroized when the cache is dropped (see xref:pattern-zeroize.adoc[Zeroize Sensitive Data]).

=== Seal Construction

The HMAC seal is computed over a canonical byte representation of:

. The serialized `DetectionResult` (or equivalent pipeline output)
. The `TrustLevel` discriminant as a single byte
. A digest of the `EvidenceBundle` (SHA-256 of all `EvidenceRecord` entries, in
  insertion order)

This ensures that downgrading the trust level or substituting an evidence record
invalidates the seal — even if the `DetectionResult` data fields are left unchanged.

=== TTL

The TTL is short by design. The default is 30 seconds. Configuration may lower it;
the implementation must not allow it to be raised above a system-defined ceiling
(suggested maximum: 5 minutes). A short TTL limits the window during which a
compromise of the cache has operational impact, and ensures that package updates and
kernel changes are detected promptly.

=== FIPS Posture

On FIPS 140-2/140-3 systems, the HMAC algorithm must be FIPS-validated. The only
validated HMAC algorithm available to this codebase without an FFI dependency is
HMAC-SHA-256 via a FIPS-validated provider. If no validated provider is available,
the cache must be disabled — the pipeline runs on every query, no caching.

The FIPS check uses the same `ProcfsText` path used in `integrity_check.rs`:
`/proc/sys/kernel/fips_enabled`. This read must occur during cache initialization,
not at seal time, to avoid a TOCTOU window where the FIPS state changes between
initialization and first use.

=== Fail-Closed on Seal Failure

Seal verification failure is not a recoverable error — it is a security event. The
correct response is:

. Discard the cache entry immediately
. Re-run the full pipeline
. Log the anomaly with `log::warn!` (never suppress it)
. Return the fresh result to the caller

This follows the xref:pattern-fail-closed.adoc[Fail-Closed] and
xref:pattern-loud-failure.adoc[Loud Failure] patterns. Returning a stale result on
seal failure is prohibited; the caller must not be able to observe that a seal failure
occurred in a way that leaks the cached value.

== In the UMRS Codebase

The SEC pattern is designed around types that already exist in `umrs-platform`. When
implemented, the cache will wrap the output of the OS detection pipeline.

=== Existing Types the Cache Will Seal

`TrustLevel` (defined in `umrs-platform/src/confidence.rs`) is the primary value
being protected. Its discriminants map directly to the pipeline's evidence strength:

[source,rust]
----
// umrs-platform/src/confidence.rs
pub enum TrustLevel {
    Untrusted,              // T0 — no evidence
    KernelAnchored,         // T1 — kernel attributes only
    EnvAnchored,            // T2 — kernel + environment
    SubstrateAnchored,      // T3 — substrate (package DB) confirmed
    IntegrityAnchored,      // T4 — cryptographic digest verified
}
----

A falsified cache entry that changes `IntegrityAnchored` to `SubstrateAnchored`
would cause policy decisions to treat the host as having weaker provenance. The seal
covers the `TrustLevel` byte, so this substitution is detectable.

`EvidenceBundle` (defined in `umrs-platform/src/evidence.rs`) carries the full
record of what was observed. The evidence chain digest in the seal covers all records
in the bundle, so removing or replacing an `EvidenceRecord` also invalidates the seal.

=== Where `boot_id` Is Already Read

The `kernel_anchor` phase already reads `boot_id` using the correct provenance-checked
path (`ProcfsText` + `SecureReader`):

[source,rust]
----
// umrs-platform/src/detect/kernel_anchor.rs (step 3 of run())
let boot_id_node = ProcfsText::new("/proc/sys/kernel/random/boot_id");
let boot_id_raw = SecureReader::<ProcfsText>::new()
    .read_generic_text(&boot_id_node)?;
----

The cache implementation should reuse this value — it is already in the
`EvidenceBundle` produced by `kernel_anchor`. There is no need to read it again at
seal time.

=== FIPS Gate — Existing Pattern

The FIPS check is already established in `integrity_check.rs`:

[source,rust]
----
// umrs-platform/src/detect/integrity_check.rs
let fips_node = ProcfsText::new("/proc/sys/kernel/fips_enabled");
let fips_raw = SecureReader::<ProcfsText>::new()
    .read_generic_text(&fips_node)?;
----

The cache initialization should gate on the same path to decide whether to enable
HMAC caching or disable it and fall through to the pipeline on every call.

=== Conceptual Cache Structure

The cache is not yet implemented. When it is, the structure will resemble:

[source,rust]
----
// Conceptual — not yet in the codebase
struct SealedCache {
    // Sealed payload: serialized DetectionResult + TrustLevel byte + evidence digest
    sealed_bytes: Vec<u8>,
    // HMAC tag over sealed_bytes
    tag: [u8; 32],
    // Expiry: Instant::now() + Duration::from_secs(ttl)
    expires_at: Instant,
    // Ephemeral sealing key — zeroized on drop
    // key: SecretBox<[u8; 32]>,  ← requires zeroize crate (pattern-zeroize)
}
----

NOTE: The `zeroize` crate is not yet in the workspace. The key type placeholder above
requires xref:pattern-zeroize.adoc[Zeroize Sensitive Data] to be implemented first.
The two patterns have a hard dependency: SEC cannot safely hold its sealing key without
zeroize-on-drop.

== When to Apply

Apply the SEC pattern when all of the following are true:

. *The pipeline is expensive.* Running on every re-query would cause measurable latency
  or resource pressure (multiple file opens, database queries, digest computations).
. *The inputs change infrequently.* Kernel version, OS release, and installed packages
  change rarely at runtime — an upgrade or kernel module load is exceptional.
. *The result drives security decisions.* The output is a `TrustLevel` or equivalent
  classification that controls what the caller is permitted to do next.
. *The caller is in-process.* The cache is in-memory, single-process. Cross-process
  caching requires a separate threat model and is out of scope for this pattern.

Do not apply SEC when:

* The pipeline is cheap (sub-millisecond; no I/O). Overhead of sealing exceeds savings.
* The input changes frequently or unpredictably (network state, user session data).
* FIPS is required and no validated HMAC is available — disable caching rather than
  use an unvalidated primitive.
* The result does not affect authorization or classification decisions.

=== Primary Application Site

`umrs-platform` — OS detection pipeline (`OsDetector::detect()`). The pipeline runs
up to seven phases, queries the RPM SQLite database, and verifies SHA-256 digests of
OS release files. It is the canonical case for SEC.

Future candidates: package inventory queries in `umrs-platform`; any multi-phase
integrity check that resolves to a `TrustLevel`.

== Security Controls

[cols="1,3", options="header"]
|===
| Control | Application

| NIST 800-53 SC-28
| Protection of information at rest. The cached result is a trusted security
  determination; the HMAC seal is the integrity protection mechanism while the
  data resides in the in-memory cache.

| NIST 800-53 SC-12
| Cryptographic key management. The sealing key is ephemeral, derived from
  boot-session-unique values, never persisted, and zeroized on drop.

| NIST 800-53 SI-7
| Software and information integrity. Seal verification detects substitution of
  the cached result. Seal failure triggers re-verification — the pipeline does not
  degrade to returning an unverified result.

| NIST 800-218 SSDF PW.4
| Implement secure coding practices. The pattern enforces fail-closed behavior
  (PW.4.1), avoids unsafe caching (PW.4.4), and applies ephemeral key management
  to limit exposure.

| NIST 800-53 AU-3
| Audit record content. Seal failures are logged with `log::warn!`, producing
  auditable anomaly records without exposing cached data content.
|===

== Summary

The Sealed Evidence Cache (SEC) pattern adds a cryptographic integrity check to
in-memory caching of expensive, security-relevant pipeline results. The seal is bound
to a single boot session by an ephemeral key; a failed seal triggers a full pipeline
re-run and anomaly log rather than returning stale data. The pattern has a hard
dependency on xref:pattern-zeroize.adoc[Zeroize Sensitive Data] — the sealing key must
be zeroized on drop, and that capability must be in place before SEC is implemented.
