= umrs-ls — High-Assurance Directory Listing
:description: Overview of the umrs-ls tool: inode-anchored, SELinux-aware directory \
    listing grouped by security label, with IOV posture markers and encryption detection.
:toc: left
:toclevels: 3

// NIST SP 800-53 AC-3, AC-4, AU-3, SC-28, SI-7
// NSA RTB Non-Bypassability (RAIN), TOCTOU safety

`umrs-ls` lists the contents of a directory and groups entries by their SELinux type
and security marking.
It is designed for operators and auditors who need to verify the security posture of
filesystem objects — not just see their names.

The key distinction from `/bin/ls -Z` is *how the security label is obtained*.
Standard `ls` uses `lstat`, a path-based syscall.
`umrs-ls` attempts to `open()` each object and anchors all subsequent metadata
reads — including the SELinux xattr — to that single file descriptor.
When `open()` is denied, the entry is marked `<restricted>` rather than silently
falling back to an unverified path-based label.

For the complete usage reference including IOV marker details, TPI parse behavior,
and the threat model, see xref:umrs-tools:umrs-ls.adoc[umrs-ls — High-Assurance Directory Listing].

.Data source descriptions
[cols="2,4", options="header"]
|===
| Source | What `umrs-ls` reads from it

| `US-CUI-LABELS.json`
| US CUI catalog loaded at startup for the label detail popup.
  When a user presses Enter on a marking, the popup shows the full catalog entry.
  Searched at candidate paths; silently absent if not found.

| `CANADIAN-PROTECTED.json`
| Canadian Protected catalog for the detail popup.
  Searched at candidate paths; silently absent if not found.

| `secolor.conf`
| Optional SELinux type-to-color mapping file.
  The active policy name is queried from the kernel and used to construct the path.
  The path is never hardcoded.
  Used only when `--color` is specified.
  Falls back to bold/underline formatting if absent or unreadable.

| `/proc/mounts`
| Mount table used to identify the backing device for each mount point entry.
  Read via `ProcfsText` with procfs magic verification.

| `/sys/block/*/dm/uuid` and `/sys/block/*/dm/type`
| Device-mapper attributes used to detect LUKS dm-crypt encrypted backing devices.
  Read via `SysfsText` with sysfs magic verification.
  The lock icon appears only when these attributes confirm LUKS.

| `security.selinux` xattr (per entry)
| SELinux security context for each directory entry.
  Read via `fgetxattr(fd, ...)` — never via path-based `getxattr`.
  Verified by two independent parsers (TPI).

| `security.ima` xattr (per entry)
| IMA measurement hash presence.
  Populates the `V` position of the IOV security-posture marker.

| `ioctl(FSGETXATTR)` (per entry)
| Inode flags including `FS_IMMUTABLE_FL`.
  Populates the `I` position of the IOV security-posture marker.

|===

=== What each layer contributes

`umrs-selinux::SecureDirent` is the shared inspection primitive.
Both `umrs-ls` and `umrs-stat` call `SecureDirent::from_path()` to open a file and
gather its security attributes through the resulting fd.
Neither tool performs raw filesystem I/O directly.

`umrs-selinux::utils::dirlist::list_directory()` handles directory traversal for `umrs-ls`.
It calls `SecureDirent` on each entry and groups the results into a `DirListing` struct.
The `DirListing` contains no display logic, no ANSI codes, and no formatted strings.
`umrs-ls` consumes it and handles all rendering decisions.

`umrs-labels::Catalog` is the shared label catalog abstraction.
`umrs-ls` loads it for the label detail popup.
`umrs-label` loads it for its full registry browser.
The catalog library is the single source of truth for CUI marking data.

`umrs-selinux::fs_encrypt` performs encryption detection.
It is called by both `umrs-ls` (for the mount-point lock icon) and `umrs-stat`
(for the Encryption Source field in Tab 1).

`umrs-selinux::SecureXattrReader` applies the TPI dual-parser gate to every SELinux
label read.
This is not optional and cannot be bypassed by either tool's display layer.
The `<restricted>` and `<unlabeled>` group labels in `umrs-ls` output are data values
set by the library layer, not display decisions made by `umrs-ls`.

== Quick Reference

[source,shell]
----
# List current directory
umrs-ls

# List a specific directory
umrs-ls /var/log

# Enable true-color group headers using secolor.conf palette
umrs-ls --color /var/log

# Show file sizes
umrs-ls --with-size /usr/bin

# Suppress IOV column and modification time
umrs-ls --no-iov --no-mtime /etc
----

For the complete column reference, IOV marker detail, `<restricted>` vs. `<unlabeled>`
distinction, TPI behavior, and threat model, see
xref:umrs-tools:umrs-ls.adoc[umrs-ls — High-Assurance Directory Listing].

== Compliance

[cols="1,3", options="header"]
|===
| Control | How `umrs-ls` satisfies it

| NIST SP 800-53 AC-3
| Entries where access was denied by DAC or MAC appear in the `<restricted>` group —
  visible to the operator, not silently dropped.

| NIST SP 800-53 AC-4
| Information-flow labels (SELinux type and marking) drive the grouping structure.
  Every entry is associated with its verified label, or explicitly marked `<restricted>`
  when the label cannot be verified.

| NIST SP 800-53 AU-3
| Each row captures sufficient fields for an audit record: mode, ownership,
  modification time, and security marking.

| NIST SP 800-53 SC-28
| Encryption posture per mount point is surfaced in the `NAME` column icon zone.
  Absence of the lock icon is a gap indicator — not a guarantee of no encryption.

| NIST SP 800-53 SI-7
| IMA xattr presence reported in the `V` column;
  immutable flag reported in the `I` column.
  Both are read through the open fd.
  SELinux label verified via TPI before acceptance.

| NSA RTB RAIN (Non-Bypassability)
| The `<restricted>` sentinel is set in the data layer and cannot be overridden
  by a display option.
  No flag exists to suppress `<restricted>` entries or substitute a path-derived label.

|===
