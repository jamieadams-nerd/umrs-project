== Why Annotations Exist

Security auditors trace every enforcement decision to a governing standard.
If a function performs an access control check, the auditor needs to know
which NIST 800-53 control that check satisfies. If a type models an MLS
security label, the auditor needs to know which framework defines the
requirements for that label's handling.

Without control citations in the source, the code and the compliance
documentation are separate artifacts that require manual reconciliation.
With citations in rustdoc, `cargo doc` produces an auditable record that
is always in sync with the implementation.

Unannotated security-critical code fails an audit. The auditor cannot
assume that the right control applies; the developer must assert it.

== The Tiered Annotation Requirement

Not every item in the codebase requires an annotation. The requirement
is tiered by item type:

[cols="1,2,1"]
|===
|Item type |Requirement |Reason

|Module (`//!` doc comment)
|Always annotate with the primary governing control(s)
|The module is the unit an auditor reviews first

|Security-critical type (`struct`, `enum`)
|Annotate with the specific control the type satisfies
|Types that model security attributes are direct compliance artifacts

|Security-critical function
|Annotate with the specific control the function enforces
|Functions that make access decisions must be traceable

|Simple accessor (`fn foo(&self) -> &Bar`)
|No annotation required if the parent type is annotated
|The parent annotation covers the accessor's context

|Display / Debug impl
|No annotation required
|Formatting has no security enforcement role
|===

The key question for any function is: *does this function make a security
decision, enforce a boundary, or validate security-relevant input?* If yes,
annotate it. If it merely reads a field that the constructor already
validated, the constructor's annotation covers it.

== Annotation Format

=== Module-Level Annotation

Module annotations appear in the `//!` doc block at the top of the file,
after the description. List each applicable control on its own line.

.Example from `umrs-platform/src/kattrs/mod.rs`
[source,rust]
----
//! UMRS PLATFORM: High-Assurance Kernel Attribute Modeling
//!
//! This module provides a high-assurance substrate for interacting with
//! kernel-resident security attributes on RHEL 10.
//!
//! 1. PROVENANCE VERIFICATION (NIST 800-53 SI-7):
//!    Before any byte is read, the SecureReader verifies the backing filesystem
//!    integrity using fd-anchored fstatfs.
//!
//! 4. NON-BYPASSABILITY (NSA RTB - RAIN):
//!    By leveraging Rust's type system and private execution engines, we ensure
//!    that security checks are always invoked.
----

This format names the control, states the number, and explains in one sentence
what the module does to satisfy it. The explanation is not documentation of
the control standard — it is documentation of what *this code* does.

=== Type-Level Annotation

Type annotations appear in the `///` doc block immediately above the type
definition. State the control number and which property of the type satisfies it.

.Example from `umrs-selinux/src/category.rs`
[source,rust]
----
/// CategorySet — ebitmap Equivalent
///
/// Represents a set of 1024 MCS categories (c0 through c1023).
/// Fixed-size bitmask ensures O(1) dominance math and zero heap allocation.
///
/// NIST 800-53 AC-4: Information Flow Enforcement
/// NSA RTB Requirement: Deterministic execution and bounded memory usage.
#[derive(Clone, PartialEq, Eq, Hash)]
pub struct CategorySet {
    bits: [u64; 16],
}
----

The citation is at the end of the description, not at the beginning. The
description explains what the type does; the citation tells the auditor
which control it satisfies.

.Example from `umrs-selinux/src/context.rs`
[source,rust]
----
/// NIST 800-53 AC-3: Access Enforcement logic depends on this structure.
/// NSA RTB: Minimized TCB via strictly bounded data structures.
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
#[must_use]
pub struct SecurityContext {
    user: SelinuxUser,
    role: SelinuxRole,
    security_type: SelinuxType,
    level: Option<MlsLevel>,
}
----

`SecurityContext` is the central type for access enforcement decisions, so
it carries an AC-3 citation. The `#[must_use]` attribute enforces that
callers cannot discard the result of a security-relevant computation —
this is the NSA RTB non-bypassability principle applied at the type system level.

=== Function-Level Annotation

Function annotations appear in the `///` doc block above the function.
Cite the specific control the function enforces and state what enforcement
action the function takes.

.Example from `umrs-selinux/src/category.rs`
[source,rust]
----
/// Determines MLS dominance.
///
/// NIST 800-53 AC-4: Dominance Check (Lattice Mathematics).
///
/// Evaluates whether `self` (the subject) dominates `other` (the object).
/// In MLS, the subject must have at least all the categories of the object.
///
/// Mathematically: (Subject & Object) == Object
#[must_use]
pub fn dominates(&self, other: &Self) -> bool {
    // ...
}
----

The annotation explains both the control and the mathematical rule being
implemented. An auditor reviewing this function can verify that the
implementation matches the Bell-LaPadula category superset requirement
for AC-4 information flow enforcement.

== Which Control to Cite

For common patterns, the applicable control is usually clear:

[cols="2,1,2"]
|===
|What the item does |Control to cite |Notes

|Enforces or evaluates an access decision
|AC-3
|Access Enforcement — who can do what to what

|Enforces information flow between labels
|AC-4
|Information Flow Enforcement — label-based access; dominance math

|Verifies integrity of a file, kernel attribute, or filesystem source
|SI-7
|Software and Information Integrity — provenance verification

|Produces or structures an audit record
|AU-3
|Audit Record Content — what goes in an audit event

|Protects data at rest; zeroizes sensitive memory
|MP-4, SC-28
|MP-4: media protection; SC-28: data at rest

|Enforces non-bypassability (private constructor, module boundary)
|NSA RTB RAIN
|Not a NIST control number — cite as "NSA RTB RAIN"

|Implements TPI (Two-Path Independence) parsing
|NIST 800-218 SSDF PW.4
|Cite the SSDF practice, not an 800-53 control
|===

If multiple controls apply, list all of them. There is no rule against
citing both AC-4 and SI-7 on the same type if both apply.

== Common Mistakes

*Citing a control on every field, every accessor, and every Display impl.*
This creates noise and obscures the annotations that matter. Accessors
and Display impls do not make security decisions. The parent type's
annotation covers them.

*Using vague language: "security-related", "handles sensitive data".*
Annotations must name a specific control. "Security-related" is not a
control citation; it is a hedge. State the control number and what
property it governs.

*Omitting the control number.*
"NIST 800-53" alone is not a citation. "NIST 800-53 AC-4" names the
specific control family and number. Auditors work from control numbers,
not family names.

*Citing the standard but not explaining what the code does to satisfy it.*
The annotation should state both the control and the enforcement action.
"NIST 800-53 AC-4: Dominance Check (Lattice Mathematics)" is better than
"NIST 800-53 AC-4" alone because it tells the auditor what to look for
in the implementation.

*Annotating items that the crate re-exports but does not implement.*
Annotations belong on the implementation, not on re-exports. A thin
wrapper over an unannotated type is not annotated by the re-export.

== Module-Level vs. Function-Level Precedence

A module-level annotation does not replace function-level annotations for
security-critical functions. If `category.rs` carries a module-level
AC-4 citation and `CategorySet::dominates` also carries an AC-4 citation,
both are correct and both serve a purpose: the module annotation tells
the auditor what the module is about; the function annotation tells the
auditor exactly which function performs the enforcement.

The rule is additive: annotate at the coarsest level where the control
applies, and also at any specific item where the control is directly enforced.

== SSDF PW.4 and CMMC — Developer Summary

=== What CMMC Requires from Developers

CMMC 2.0 Level 2 does not mandate a specific programming language.
It requires evidence that security is built into the development process.
An assessor reviewing UMRS will look for:

* Version-controlled code with restricted write access
* Documented code reviews before code is promoted
* Static analysis results (Clippy, `cargo audit`, `cargo-geiger`)
* Demonstrated input validation at trust boundaries
* Use of approved cryptographic modules
* Evidence that known-vulnerable dependencies are tracked and addressed

UMRS satisfies these through its toolchain (`cargo xtask`), workspace
lint policy, and the coding patterns in this guide.

=== SSDF PW.4 in Rust — Implementation Reference

NIST SP 800-218 SSDF PW.4 defines five secure coding practices.
Each maps directly to a Rust mechanism used in this codebase.

==== PW.4.1 — Input Validation: Parse, Don't Validate

The SSDF requires validating all input from untrusted sources before use.
In Rust, the idiomatic implementation is "parse, don't validate": construct
a typed value at the trust boundary. If construction fails, reject the input
immediately — it never reaches business logic.

[source,rust]
----
// Construct a validated type at the boundary.
// If this succeeds, downstream code has a proven-valid value — not a string.
let ctx = SecurityContext::parse("user_u:user_r:user_t:s0-s3:c0,c1")?;
// ctx is now a SecurityContext — downstream code cannot receive an invalid label.
----

The key rule: constructors return `Result`. A successful construction is a
proof of validity. Fields are private to prevent callers from constructing
types that bypass validation.

*NIST SP 800-218 SSDF PW.4.1 / NIST SP 800-53 SI-10*

==== PW.4.2 — Output Encoding: Error Information Discipline

The SSDF requires encoding output to prevent injection and information leakage.
In UMRS, this applies to error messages: variable data (file paths, label strings,
kernel values) goes to the audit log; caller-visible error strings describe the
*category* of failure without including the sensitive data.

[source,rust]
----
// Correct: path logged, not returned to caller
log::error!("Integrity failure: {} is not on the authorized filesystem", path.display());
return Err(io::Error::new(
    io::ErrorKind::PermissionDenied,
    "Integrity failure: path is not on the authorized filesystem",
));
----

The caller gets the category. The log gets the specifics.
An attacker who triggers the error learns nothing useful from the return value.

*NIST SP 800-218 SSDF PW.4.2 / NIST SP 800-53 SI-12*

==== PW.4.3 — Memory Safety

Rust satisfies PW.4.3 structurally. The borrow checker prevents buffer overflows,
use-after-free, and null-pointer dereferences at compile time.

The UMRS codebase adds a compile-time enforcement layer:

[source,rust]
----
#![forbid(unsafe_code)]
----

This attribute, placed in every crate root, prevents any `unsafe` block from
compiling anywhere in the crate — including in transitively called macros.
It is a mechanical proof verifiable by an auditor without reading the code.

*NIST SP 800-218 SSDF PW.4.3 / NIST SP 800-53 SA-11*

==== PW.4.4 — Secure Arithmetic

Integer overflow in a security-relevant calculation can produce silent
incorrect results. The SSDF requires explicit handling. Rust provides
checked variants on all primitive integer operations.

[source,rust]
----
// Instead of: let new_count = count + 1; // silent overflow possible
let new_count = count.checked_add(1)
    .ok_or(MyError::CountOverflow)?;
----

For security values, also prefer `.get(i)` over direct indexing `buf[i]`.
The former returns `Option` on out-of-bounds; the latter panics.

For credential comparisons, use `subtle::ConstantTimeEq` to prevent
timing oracle attacks — see xref:patterns:pattern-constant-time.adoc[Constant-Time Comparison].

*NIST SP 800-218 SSDF PW.4.4 / NIST SP 800-53 SC-28*

==== PW.4.5 — Secure Cryptography

The SSDF requires using approved cryptographic modules. For UMRS, "approved"
means FIPS 140-2 or 140-3 validated. This is a property of a specific
*implementation*, not a crate name:

* A crate can use `AES-256-GCM` but not be FIPS validated.
* RHEL 10 in FIPS mode provides validated OpenSSL — but this must be
  verified against the NIST CMVP database for each deployment.
* The `ring` crate uses strong algorithms but does not hold FIPS validation
  as of this writing.

Before writing cryptographic code, verify the FIPS validation status of the
specific implementation at csrc.nist.gov/projects/cryptographic-module-validation-program.

See also xref:cryptography:fips-cryptography-cheat-sheet.adoc[FIPS cryptography reference].

*NIST SP 800-218 SSDF PW.4.5 / FIPS 140-2 / FIPS 140-3*

== CCE Citation Format

=== What CCE Is

A Common Configuration Enumeration (CCE) identifier is a stable, unique reference to
a specific configuration guidance item defined in a Security Technical Implementation
Guide (STIG) or a Security Content Automation Protocol (SCAP) content stream.

CCE identifiers take the form `CCE-NNNNN-N`, where the body is a five-digit number
and the check digit is a single digit.
The `CCE-` prefix is always uppercase and always present.
There are no abbreviated or alternate forms.

Example: `CCE-89386-7`

A CCE identifier alone does not identify which STIG or SCAP profile it belongs to.
Every CCE citation in UMRS documentation and source code must include a provenance
qualifier — the profile name in parentheses immediately after the identifier.
For RHEL 10 content, that qualifier is `(RHEL 10 STIG)`.

=== When to Cite a CCE Identifier

Cite a CCE identifier only when the corresponding STIG requirement has been
confirmed to map to a UMRS type, function, or indicator.

Do not cite a CCE if:

* The mapping is approximate or aspirational — describe the relationship in prose instead.
* The CCE belongs to a profile other than RHEL 10 — do not use RHEL 9 or RHEL 8 identifiers
  in RHEL 10 STIG context.
* The source is the `cce-nist-crossref.md` index but the control mapping has not been
  independently verified against the STIG itself.

CCE identifiers are stable, but STIG revisions can change the underlying guidance.
Record the STIG version when a CCE is first introduced into the codebase.

=== Format in Rust Doc Comments

In Rust `///` and `//!` doc comments, CCE citations follow NIST control citations.
CCE never appears alone and never precedes the NIST citation.

The format is:

----
- CCE-NNNNN-N (RHEL 10 STIG) — <imperative description of the requirement>
----

The description should be the STIG signal description in normalized imperative form.
Do not copy descriptions that are truncated at a dash or that describe a check step
rather than a compliance requirement — normalize these to complete sentences that
state the required configuration state.

.Example — kernel module type with both NIST and CCE citations
[source,rust]
----
/// Controls the kernel module load state for a specific module.
///
/// # Compliance
///
/// - NIST SP 800-53 CM-7(a) — restrict functionality to the minimum required
/// - NIST SP 800-53 CM-7(b) — prohibit or restrict use of unnecessary functions
/// - CCE-87455-2 (RHEL 10 STIG) — Ensure kernel module 'bluetooth' is disabled
----

.Example — SELinux enforcement state type
[source,rust]
----
/// Models the current SELinux enforcement mode as reported by the kernel.
///
/// # Compliance
///
/// - NIST SP 800-53 AC-3 — access enforcement; depends on SELinux being in enforcing mode
/// - NIST SP 800-53 SC-7(21) — boundary protection; SELinux enforces domain isolation
/// - CCE-89386-7 (RHEL 10 STIG) — Ensure SELinux State is Enforcing
----

The `# Compliance` section heading is preferred when citing multiple controls.
For a single short citation, an inline citation after the type description is acceptable.

IMPORTANT: Do not add CCE citations to accessors, Display impls, or any item that does
not directly implement a compliance requirement. The tiering rules in
xref:compliance-annotations.adoc#_the_tiered_annotation_requirement[The Tiered Annotation Requirement]
still apply.

=== Format in Antora Documentation Pages

==== Inline citations in body text

Use parenthetical citation, preserving the exact `CCE-NNNNN-N` format.
No special markup is required around a CCE cited inline.

----
The SELinux enforcement check (CCE-89386-7) maps to NIST SP 800-53 AC-3 and SC-7(21).
----

Do not use code font around an inline CCE citation unless it appears in a code listing.
CCE identifiers are reference numbers, not code tokens.

==== Compliance cross-reference tables

When presenting multiple CCE mappings in a table (for example, in a compliance registry
or a posture signal catalog), use this column structure:

[source,asciidoc]
----
[cols="1,1,2,2,3", options="header"]
|===
|CCE |STIG Profile |Signal Name |NIST Controls |Description

|CCE-89386-7
|RHEL 10 STIG
|`selinux_state`
|AC-3, SC-7(21)
|Ensure SELinux State is Enforcing

|CCE-88366-0
|RHEL 10 STIG
|`selinux_policytype`
|AC-3, SC-7(21)
|Ensure SELinux policy type is set to the required profile
|===
----

Rules for compliance tables:

* CCE goes in its own column — do not embed it in the description column.
* Signal names go in monospace (backtick-wrapped in the source).
* NIST control enhancements use parenthesized form without a space: `CM-7(a)`, not `CM-7 (a)`.
* Descriptions must be complete sentences. Do not copy truncated STIG descriptions.
  Normalize check-step artifacts to requirement statements.
* Always include the STIG Profile column. More than one STIG exists; the profile name
  disambiguates which body of guidance the CCE belongs to.

=== Format in CLI and TUI Output

==== `--json` output

In structured JSON output, the CCE identifier is a machine-readable field alongside the
NIST controls and the STIG severity level.

[source,json]
----
{
  "indicator": "kernel_module_bluetooth_disabled",
  "cce": "CCE-87455-2",
  "stig_profile": "RHEL 10",
  "nist_controls": ["CM-7(a)", "CM-7(b)"],
  "severity": "medium"
}
----

The `cce` field preserves the exact `CCE-NNNNN-N` format.
The `stig_profile` field is always present when `cce` is present.
The `severity` field uses lowercase: `"high"`, `"medium"`, `"low"`.

==== Human-readable compact view

In human-readable compact output (for example, a posture summary table), suppress the CCE
identifier. Show it only in expanded detail view.

----
[MEDIUM] kernel bluetooth module disabled
----

In expanded or verbose view, present CCE as a secondary detail line:

----
[MEDIUM] kernel bluetooth module disabled
         CCE-87455-2 · CM-7(a) CM-7(b)
----

Do not abbreviate `CCE-NNNNN-N` in any output.
Do not place the CCE on the primary indicator line — it is reference metadata, not an
operator-facing status.

NOTE: These CLI and TUI display conventions are forward specifications.
As of this writing, not all tools have implemented `--json` output or expanded detail views.
New tool implementations should follow this format.
