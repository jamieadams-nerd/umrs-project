= Developer Guide
:description: How to write high-assurance code for UMRS: design patterns, compliance annotations, build tooling, and crate architecture.

// NIST 800-218 SSDF PW.1 — Define security requirements for the software
// NSA RTB VNSSA — Non-bypassable security requirements baked into process

The UMRS platform provides a set of Rust crates designed to support the development of
high-assurance system software.
These libraries implement both high-assurance engineering techniques and recognized
high-assurance security patterns in order to increase confidence in the correctness,
provenance, and trustworthiness of system-derived data.

== The Core Assumption

Many programming environments provide convenient abstractions for interacting with
operating system state.
In most cases, these abstractions assume that system information can be trusted without
verification.
Files are read, parsed, and consumed with the assumption that the data retrieved from
the operating system accurately reflects reality.

The UMRS platform operates under a different assumption: *system state must be validated
before it is trusted.*

This emphasis on correctness is not merely about returning accurate values or displaying
correct information.
In many systems, information obtained from the operating system becomes input to other
subsystems that make decisions based on that data.
Security policies, configuration logic, compatibility handling, and automated system
behavior frequently depend on correctly understanding the system environment.

For example, accurately determining the operating system and platform characteristics
is often critical.
Different operating systems expose different kernel interfaces, security models,
filesystem semantics, and configuration expectations.
If foundational platform identification is incorrect or unverified, subsequent
components may apply incorrect assumptions about how the system behaves.
Those incorrect assumptions can lead to faulty configuration decisions, incorrect
security posture evaluation, or improper system behavior.

For this reason, the correctness and assurance of the data itself becomes foundational
to the correctness of all decisions that follow.

== Foundational Crate Architecture

The UMRS platform is organized as a layered set of foundational crates that provide
progressively higher levels of abstraction while preserving strict validation guarantees.

[cols="1,3", options="header"]
|===
|Crate |Purpose

|`umrs-hw`
|Hardware level libraries like timers and cpu extension information

|`umrs-platform`
|Low-level platform interrogation, hardened readers, kernel interface access,
and trusted system ingestion

|`umrs-selinux`
|Strongly typed abstractions and primitives for interacting with SELinux policy and 
system security state. In the future, there will be a mutually exclusive `umrs-apparmor`.

|`umrs-core`
|Higher-level orchestration utilities, shared abstractions, and reusable platform logic

|`umrs-ui`
|Provides high-level user interface templates for both Textual User Interface (TUI) and future
graphical user interface (GUI). Besides, templates it provides several helper functions to make
humans happy.
|===

This layered design allows systems to use only the level of functionality required

== High-Assurance System Interrogation

Typical software libraries retrieve operating system information by directly reading
files such as `/etc/os-release` or by issuing system calls without validating the
integrity of the data source.

The UMRS platform instead performs progressive trust validation before accepting
system data as trustworthy.
This process follows a trust ladder beginning at T0 and progressing upward through
increasingly higher levels of assurance.
Each step gathers additional corroborating evidence about the provenance and integrity
of the system information being retrieved.

During this process the platform may:

* obtain authoritative information from kernel pseudo-filesystems such as sysfs,
  procfs, or securityfs
* validate filesystem provenance and mount topology
* confirm package manager ownership of system files
* verify file integrity where cryptographic mechanisms are available
* apply strict parsing and validation rules before returning structured results

Only after these verification steps succeed does the platform read and parse the
target data source.

For a detailed walkthrough of this approach, see
xref:os-detection-deep-dive.adoc[OS Detection Pipeline — Deep Dive].
That page traces the full interrogation sequence for OS detection, including
the trust ladder, the sealed evidence cache, and the compliance mapping.

== Evidence Chains and Verifiable Assurance

A key design principle of UMRS is that trust decisions must be inspectable and auditable.
When information is retrieved through the UMRS libraries, the result includes an
evidence chain describing the validation steps that were performed to establish trust
in the data.

The evidence chain records:

* which trusted sources were consulted
* what verification checks were performed
* the trust level reached
* the conditions under which the data was accepted

Because the validation process follows recognized high-assurance patterns, these evidence
chains can also be used to demonstrate compliance and traceability, cross-referencing
validation steps against NIST and NSA RTB guidance.

== Strongly Typed System Primitives

In addition to secure ingestion techniques, the UMRS crates expose strongly typed
primitives representing common system concepts.
Many traditional system APIs return loosely structured strings or unvalidated data.
The UMRS platform instead exposes strongly typed structures whose values are validated
at construction time.

Examples include strongly typed representations of platform identities, security
contexts, labels and attributes, system identifiers, and other security-sensitive
system metadata.
By enforcing validation during object construction, the platform ensures that downstream
logic interacts only with verified and well-formed data structures.
