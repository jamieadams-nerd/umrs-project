== Philosophy

[quote]
____
Make the right thing easy to do and the wrong thing hard to do.
____

This is the design goal behind every pattern in this guide.
Security in UMRS is not a layer added after the code is written.
It is encoded in the type system, the module boundaries, and the
construction rules, so that the compiler enforces it before a
human reviewer ever sees the code.

Each section below is organized around *what you are about to do* —
open a file, parse some input, construct a type, do arithmetic, report
an error.
That is how code gets written.
The NIST and NSA RTB citations are there for auditors who read the same
document; they are at the end of each pattern, not the beginning.

=== The three-layer model

----
This guide          ← why each pattern exists; what threat it defeats
rustdoc API docs    ← the contract for each public function
Source code         ← the implementation, annotated with control citations
----

The doc comments in the source are the authoritative bottom layer.
`cargo doc` delivers that layer as the API reference.
This guide is the layer above it — it explains the reasoning so that a
developer can apply the same thinking to the next problem, not just
copy the pattern mechanically.

=== On gatekeeping and leaving early

Two habits that map directly onto the patterns below:

. *Gatekeeping at function entry* — screen all inputs at the top of a
  function before doing any work.
  The code never reaches the sensitive operation if the input is wrong.
  This is validate-at-construction applied to function scope.

. *Leave early and often* — return `Err` the moment a security condition
  is violated.
  Do not continue executing with degraded or uncertain state.
  This is fail-closed applied to control flow.

Both habits are visible throughout the UMRS codebase.
The patterns below show where.

== 1. Opening and Reading Files

Every file read in UMRS is a potential security event.
The filesystem is shared, mutable, and partially controlled by other
processes.
These four patterns govern every I/O operation in the codebase.

=== 1.1 TOCTOU Safety — Anchor to a Single File Descriptor

*The danger.*
A file opened by path can be replaced, renamed, or symlinked to a
different target between the `open` call and any subsequent operation.
This is a Time-of-Check/Time-of-Use (TOCTOU) race condition.
An attacker who can write to the directory can swap the target during
that window.

*The pattern.*
Open the file once.
Perform all subsequent operations through the resulting file descriptor.
Never re-open by path for a second operation on the same logical resource.
Use `rustix` fd-based syscalls (`fgetxattr`, `ioctl_getflags`) rather
than path-based syscalls (`getxattr`).

*In the UMRS codebase.*

`SecureXattrReader::read_raw()` in `xattrs.rs` reads the raw xattr bytes
through the fd the caller already holds — it never opens a path:

[source,rust]
----
// xattrs.rs
pub fn read_raw(file: &File, attr: &str) -> io::Result<Vec<u8>> {
    let size = fgetxattr(file, attr, &mut [])
        .map_err(|e| io::Error::from(e))?;

    if size == 0 {
        return Err(io::Error::new(io::ErrorKind::InvalidData, "Empty xattr"));
    }

    let mut buffer = vec![0u8; size];
    let bytes_read = fgetxattr(file, attr, &mut buffer)
        .map_err(|e| io::Error::from(e))?;

    buffer.truncate(bytes_read);
    if buffer.last() == Some(&0) {
        buffer.pop();
    }
    Ok(buffer)
}
----

`SecureDirent` documents this invariant explicitly in its module comment:

[source,rust]
----
// secure_dirent.rs — design invariants
// - TOCTOU safety: a single `File` handle is opened once; all
//   subsequent metadata reads (ioctl_getflags, fgetxattr) operate
//   through that same fd. The path is never re-resolved after open.
----

*Citation.* NIST 800-53 SI-7 (Software and Information Integrity)

*Tool example.*
xref:umrs-tools:umrs-stat.adoc[umrs-stat] exposes this pattern at the operator level.
Every attribute read in the Security tab — SELinux label, inode flags, encryption source —
goes through `SecureDirent`'s single-open fd path.
The IMA/EVM flag row shows whether the kernel has an integrity hash for the file.
If a `TpiDisagreement` is present, the label cannot be trusted for policy decisions.

=== 1.2 Provenance Verification — Check Filesystem Magic Before Trusting Kernel Data

*The danger.*
`/sys/fs/selinux/` and `/proc/` are kernel pseudo-filesystems.
An attacker or misconfigured container can bind-mount an
attacker-controlled directory over either path.
Any byte read from that path would come from the attacker, not the
kernel.

*The pattern.*
Before reading any byte from a kernel pseudo-filesystem, verify the
filesystem magic number.
`/sys/fs/selinux/` has `SELINUX_MAGIC`; `/proc/` has `PROC_SUPER_MAGIC`.
A bind-mounted impostor will have a different magic value and must be
rejected immediately.

Critically, the magic check must be performed on the *open file descriptor*,
not on the path.
Calling `statfs(path)` and then `File::open(path)` separately leaves a
window between the two calls during which an attacker can substitute a
bind-mounted directory — a TOCTOU race on the check itself.
The correct sequence is: open the file first, then call `fstatfs` on the
resulting descriptor.
See also section 1.1.

*In the UMRS codebase.*

`SecureReader::execute_read()` in `umrs-platform: kattrs.rs` opens the file first,
then performs the fd-anchored magic check on the open descriptor:

[source,rust]
----
// umrs-platform: kattrs.rs
fn execute_read(path: &Path, expected_magic: FsType) -> io::Result<T::Output> {
    // Open the file first — the magic check is performed on the open fd,
    // not on the path. This closes the TOCTOU window that exists between
    // a path-based statfs() and the subsequent File::open().
    let mut file = File::open(path)?;

    let stats = fstatfs(&file).map_err(io::Error::other)?;
    if stats.filesystem_type() != expected_magic {
        // NIST 800-53 SI-12: the path is sent to the audit log, not to the
        // caller-visible error string. Kernel paths are not sensitive per se,
        // but embedding variable data in user-visible errors creates a pattern
        // that is unsafe when extended to security labels or credentials.
        log::error!(
            "Integrity Failure: {} is not on the authorized filesystem",
            path.display()
        );
        return Err(io::Error::new(
            io::ErrorKind::PermissionDenied,
            "Integrity Failure: path is not on the authorized filesystem",
        ));
    }

    let mut buffer = [0u8; 16];
    let bytes_read = file.read(&mut buffer)?;
    T::parse(&buffer[..bytes_read])
}
----

If the magic check fails, `file` is dropped immediately and nothing is read.
The path is emitted to the audit log via `log::error!` — it does not
appear in the error string returned to the caller (NIST 800-53 SI-12).

*Citation.* NIST 800-53 SI-7

=== 1.3 Compile-Time Path Binding — StaticSource Trait

*The danger.*
A path string passed at runtime to a kernel attribute reader can be
wrong — mistyped, injected, or intentionally substituted.
Even a correct path can have the wrong expected filesystem magic if
the caller forgets to supply it.

*The pattern.*
Bind the path and the expected filesystem magic to the *type*, not the
call site.
The `StaticSource` trait in `umrs-platform: kattrs.rs` makes the path and magic
associated constants of each kernel attribute type.
The compiler verifies that every call uses the correct values.
There is no runtime parameter to get wrong.

*In the UMRS codebase.*

[source,rust]
----
// umrs-platform: kattrs.rs
pub trait StaticSource: KernelFileSource {
    const PATH: &'static str;
    const EXPECTED_MAGIC: FsType = SELINUX_MAGIC;

    // Default implementation routes through SecureReader::execute_read,
    // which performs the fd-anchored fstatfs magic check before reading.
    // Provenance verification is non-bypassable via this method.
    fn read() -> io::Result<Self::Output> {
        SecureReader::<Self>::new().read()
    }
}
----

Each concrete type (`SelinuxEnforce`, `SelinuxMls`, `SelinuxPolicyVers`,
`ProcFips`) declares its own `PATH` and `EXPECTED_MAGIC` as associated
constants, and each carries a doc comment with the applicable NIST 800-53
control citations.

The default `read()` implementation on `StaticSource` delegates to
`SecureReader::<Self>::new().read()`, which routes through `execute_read`.
This means provenance verification is non-bypassable via the trait method —
there is no code path through `StaticSource::read()` that skips the
fd-anchored magic check.

A caller invokes `SelinuxEnforce::read()` and cannot accidentally supply
the wrong path, forget the magic check, or bypass the fd-anchored
verification — the type and the trait default together enforce all three.

*Citation.* NSA RTB RAIN (Non-Bypassability)

'''

=== 1.4 Trust Gates — Only Read Configuration When the Kernel Confirms

*The danger.*
`/etc/selinux/config` reflects boot-time configuration, not the running
kernel state.
If SELinux is disabled or the kernel is running a different policy,
the config file may describe a state that does not exist.
Treating it as authoritative when the kernel disagrees is a trust
violation.

*The pattern.*
Gate configuration file reads behind a kernel status check.
Only read the config when the kernel has confirmed the subsystem is
active.
If the kernel says SELinux is disabled, return `None` for the policy —
do not guess from the config.

*In the UMRS codebase.*

[source,rust]
----
// status.rs
impl SelinuxStatus {
    pub fn current() -> Self {
        let enabled = is_selinux_enabled();

        let enforcing = if enabled {
            SelinuxEnforce::read()
                .map(|state| state == EnforceState::Enforcing)
                .unwrap_or(false)
        } else {
            false
        };

        // Only trust the config file when the kernel confirms SELinux is active.
        let policy = if enabled {
            read_policy_from_config()
        } else {
            None
        };

        Self { enabled, enforcing, policy }
    }
}
----

*Citation.* NIST 800-53 CM-6 (Configuration Settings)

'''

== 2. Parsing Security-Relevant Input

Any input that drives an access control or classification decision is
security-relevant.
A single parsing flaw in that path is a potential privilege escalation.
These patterns govern every parser in the codebase.

=== 2.1 Two-Path Independence (TPI)

*The danger.*
A single parser has a single point of failure.
A subtle logic flaw — an off-by-one in a boundary check, an ambiguous
grammar rule — can pass malformed input to an access control decision.
The flaw may not be visible in testing because both the input and the
expected output are wrong in the same direction.

*The pattern.*
Parse with two completely independent implementations.
If both succeed and agree, the result is trustworthy.
If they disagree, or if either fails, fail closed — return an error,
never a default or a degraded result.

In UMRS, SELinux security contexts are parsed with `nom` (a declarative
parser combinator) and `FromStr` (an imperative character-by-character
implementation).
These are genuinely independent: different algorithms, different logic
paths, different failure modes.

*In the UMRS codebase.*

[source,rust]
----
// xattrs.rs
pub fn read_context(file: &File) -> io::Result<SecurityContext> {
    let raw_bytes = Self::read_raw(file, XATTR_NAME_SELINUX)?;
    let context_str = std::str::from_utf8(&raw_bytes).map_err(|_| {
        io::Error::new(io::ErrorKind::InvalidData, "Non-UTF8 Context")
    })?;

    // PATH A: nom parser (declarative)
    let (_, context_a) = parse_context_nom(context_str).map_err(|e| {
        log::error!("TPI Path A (nom) failed: {e}");
        io::Error::new(io::ErrorKind::InvalidData, "TPI Path A Failed")
    })?;

    // PATH B: FromStr (imperative)
    let context_b: SecurityContext = context_str.parse().map_err(|e| {
        log::error!("TPI Path B (FromStr) failed: {e}");
        io::Error::new(io::ErrorKind::InvalidData, e.to_string())
    })?;

    // TPI GATE: if the results disagree, neither is trusted
    if context_a != context_b {
        log::error!("CRITICAL: RTB Redundancy Failure!");
        log::error!("PATH A: {context_a:?}");
        log::error!("PATH B: {context_b:?}");
        return Err(io::Error::new(
            io::ErrorKind::PermissionDenied,
            "RTB Redundancy Failure: Security Ctx logic mismatch",
        ));
    }

    Ok(context_a)
}
----

*Citation.* NSA RTB (Redundancy) / NIST 800-218 SSDF PW.4.1

'''

=== 2.2 Fail-Closed on Disagreement

*The danger.*
When two independent paths disagree, the tempting response is to pick
one, log a warning, and continue.
This is fail-open.
In a security context, a disagreement means something unexpected
happened — and the safe response to the unexpected is to stop.

*The pattern.*
Any disagreement between independent parse or validation paths is
treated as a `PermissionDenied` error.
Log at `error!` level — loudly and immediately — before returning.
Never return `Ok` with a degraded or defaulted result when a check
has fired.

This pattern appears wherever TPI is applied.
It also governs the type-validation gate in `umrs-platform: kattrs.rs`:

[source,rust]
----
// umrs-platform: kattrs.rs
pub fn validate_type_redundant(context: &str) -> io::Result<&str> {
    let (_, type_a) = parse_type_path_a(context).map_err(|_| {
        io::Error::new(io::ErrorKind::InvalidData, "Path A Logic Failure")
    })?;

    let type_b = parse_type_path_b(context)?;

    if type_a != type_b {
        return Err(io::Error::new(
            io::ErrorKind::PermissionDenied,
            "RTB Redundancy Failure: Logic Mismatch Detected",
        ));
    }

    Ok(type_a)
}
----

*Citation.* NSA RTB RAIN (Non-Bypassability)

'''

=== 2.3 Path Validation — Single Gate at Construction

*The danger.*
A null byte in a path string terminates the string at the C layer —
everything after the null is silently discarded.
A newline character in a path can break log records or inject false
entries into structured logs.
A path or component that exceeds `PATH_MAX` or `NAME_MAX` can cause
silent truncation in kernel calls.

*The pattern.*
Validate paths exactly once, at the point where the raw string becomes
a typed path value.
The validation gate checks: empty, null byte, newline, `PATH_MAX` total
length, and `NAME_MAX` per component.
After construction, the type carries proof of validity — no downstream
re-validation is needed or expected.

*In the UMRS codebase.*

[source,rust]
----
// secure_dirent.rs
// NSA RTB: Single validation gate. No re-validation downstream.
// NIST SP 800-53 SI-10: rejects oversized, null-bearing, and
// newline-bearing paths before they reach any system call.
fn validate_path_common(raw: &str) -> Result<(), PathError> {
    if raw.is_empty() {
        return Err(PathError::Empty);
    }
    if raw.contains('\0') {
        return Err(PathError::ContainsNull);
    }
    if raw.contains('\n') || raw.contains('\r') {
        return Err(PathError::ContainsNewline);
    }
    if raw.len() > PATH_MAX {
        return Err(PathError::TooLong { max: PATH_MAX, got: raw.len() });
    }
    for component in raw.split('/').filter(|c| !c.is_empty()) {
        if component.len() > NAME_MAX {
            return Err(PathError::ComponentTooLong {
                component: component.to_owned(),
                max: NAME_MAX,
                got: component.len(),
            });
        }
    }
    Ok(())
}
----

*Citation.* NIST 800-53 SI-10 (Information Input Validation)

'''

=== 2.4 POSIX Name Validation

*The danger.*
Usernames and group names that contain invalid characters can cause
unexpected behavior in system calls, PAM modules, and audit records.
A username beginning with a digit or containing a colon is technically
parseable but violates shadow-utils conventions and can break `/etc/passwd`
parsing.

*The pattern.*
Validate at the single construction gate.
Enforce: not empty, no null byte, length ≤ `POSIX_NAME_MAX_LEN`,
first character in `[a-zA-Z_]`, remaining characters in `[a-zA-Z0-9_-]`,
with a trailing `$` permitted only for Samba machine accounts.
The `'.'` character is explicitly excluded despite appearing in some
documentation — Linux shadow-utils does not permit it in usernames
or group names.

*In the UMRS codebase.*

[source,rust]
----
// posix/identity.rs
fn validate_posix_name(raw: &str, allow_trailing_dollar: bool)
    -> Result<(), PosixNameError>
{
    // Null byte: explicit defense against C interop confusion.
    if raw.contains('\0') {
        return Err(PosixNameError::ContainsNull);
    }
    if raw.is_empty() {
        return Err(PosixNameError::Empty);
    }
    if raw.len() > POSIX_NAME_MAX_LEN {
        return Err(PosixNameError::TooLong { max: POSIX_NAME_MAX_LEN, got: raw.len() });
    }

    let mut chars = raw.chars().enumerate();

    // First character: must be [a-zA-Z_]
    if let Some((_, first)) = chars.next()
        && !matches!(first, 'a'..='z' | 'A'..='Z' | '_')
    {
        return Err(PosixNameError::InvalidFirstChar(first));
    }

    let last_idx = raw.len().saturating_sub(1);
    for (i, ch) in chars {
        let valid = match ch {
            'a'..='z' | 'A'..='Z' | '0'..='9' | '_' | '-' => true,
            '$' if allow_trailing_dollar && i == last_idx => true,
            _ => false,
        };
        if !valid {
            return Err(PosixNameError::InvalidChar(ch));
        }
    }
    Ok(())
}
----

*Citation.* IEEE Std 1003.1 (POSIX) / Linux shadow-utils

'''

=== 2.5 UTF-8 Boundary Enforcement

*The danger.*
Non-UTF8 byte sequences in filenames can cause incorrect display,
silent truncation, and log injection.
If a filename containing `\xff\n` reaches a log record, the newline
splits the record and can cause the remainder to be interpreted as a
separate log entry — a log injection attack.

*The pattern.*
Reject any `OsStr` that is not valid UTF-8 at the boundary where the
kernel name enters the type system.
Never pass raw bytes through to display, logging, or comparison.

*In the UMRS codebase.*
`SecureDirent` documents this invariant in its module-level comment:

[source,rust]
----
// secure_dirent.rs — design invariants
// - No raw strings: paths are AbsolutePath/ValidatedFileName,
//   filenames are UTF-8 validated with non-UTF8 explicitly rejected
//   at the OsStr boundary.
----

*Citation.* NIST 800-53 SI-10

== 3. Constructing Security-Relevant Types

In UMRS, a successfully constructed type is a proof.
`Category(42)` proves the value is a valid category id.
`SensitivityLevel(3)` proves the value is a valid sensitivity level.
Downstream code that receives one of these types never needs to
re-validate — the type carries the guarantee.

=== 3.1 Validate at Construction

*The danger.*
A value that is constructed without validation — a bare `u16` for a
category id, a raw `String` for a security type — has no guarantee
of validity at the point of use.
Every caller must re-validate, or trust that the previous caller did.
Either way, the check can be skipped.

*The pattern.*
Constructor returns `Result`.
If construction succeeds, the value is within bounds and meets all
structural requirements.
Downstream code receives a value, not a `Result` — the error was
handled at the boundary.

*In the UMRS codebase.*

[source,rust]
----
// category.rs
// Valid range: c0 through c1023 (MAX_CATEGORY = 1023).
pub const fn new(id: u16) -> Result<Self, CategoryError> {
    if id > MAX_CATEGORY {
        return Err(CategoryError::OutOfRange(id));
    }
    Ok(Self(id))
}
----

[source,rust]
----
// sensitivity.rs
// Valid range: s0 through s15 (MAX_SENSITIVITY = 15).
pub const fn new(level: u16) -> Result<Self, SensitivityError> {
    if level > MAX_SENSITIVITY {
        return Err(SensitivityError::OutOfRange(level));
    }
    Ok(Self(level))
}
----

Once `Category::new(42)` succeeds, any code holding a `Category` can
index into a `CategorySet` without bounds-checking.
The proof travels with the value.

*Citation.* NIST 800-218 SSDF PW.4.1

'''

=== 3.2 Newtype Wrappers

*The danger.*
`Inode` and `DevId` are both `u64`.
Without distinct types, the compiler permits passing one where the other
is expected.
The mistake is silent and produces incorrect audit records.

*The pattern.*
Wrap each primitive in a named struct with a private inner field.
The types are zero-cost — no runtime overhead — but the compiler
enforces semantic correctness.

*In the UMRS codebase.*

[source,rust]
----
// posix/primitives.rs
// NIST SP 800-53 AU-3: inode number is a required audit record field.
pub struct Inode(u64);

// Not globally unique — the same number can appear on different devices.
// Always pair with DevId for global uniqueness.
pub struct DevId(u64);

pub struct HardLinkCount(u32);
----

`HardLinkCount` also carries domain-specific security logic:

[source,rust]
----
// posix/primitives.rs
// NIST SP 800-53 AC-3: hard-linked files may bypass path-based controls.
pub const fn is_multiply_linked(self) -> bool {
    self.0 > 1
}
----

Hard links on CUI systems are a finding — a file accessible through
two directory entries may be readable outside its expected access
control tree.
The type encodes that knowledge; callers do not need to re-derive it.

*Citation.* NIST 800-53 AU-3 / AC-3

'''

=== 3.3 Private Fields and Controlled Constructors

*The danger.*
A public struct field lets any caller construct an arbitrary internal
state, bypassing all validation.
A `CategorySet { bits: [u64::MAX; 16] }` constructed directly would
represent every category simultaneously, regardless of any access
control policy.

*The pattern.*
Make fields private.
Expose only named constructor functions that enforce structural
invariants.
There should be no way to construct an invalid value that the type
system allows to compile.

*In the UMRS codebase.*

[source,rust]
----
// category.rs
// The 'bits' field is private. The only construction paths are new() and full().
pub struct CategorySet {
    bits: [u64; 16],   // 1024 bits — private
}

impl CategorySet {
    #[must_use]
    pub const fn new() -> Self {
        Self { bits: [0; 16] }   // empty: no compartments
    }

    #[must_use]
    pub const fn full() -> Self {
        Self { bits: [u64::MAX; 16] }   // full: all compartments
    }
}
----

A caller cannot construct a `CategorySet` with an arbitrary bit pattern.
They start from `new()` or `full()` and modify through the controlled
`insert()` and `remove()` methods.

*Citation.* NSA RTB RAIN (Non-Bypassability)

'''

=== 3.4 `#[must_use]` Discipline

*The danger.*
A constructor or query method returns a `Result` or a computed value.
The caller ignores the return value.
The compiler says nothing.
The security check was performed but the result was discarded.

*The pattern.*
Annotate every constructor, every security-relevant query, and every
method whose return value reflects a security condition with
`#[must_use]`.
The compiler will warn — and because the project runs with `-D warnings`,
the build will fail — if any call site ignores the return value.

*In the UMRS codebase.*

[source,rust]
----
// Constructors — must_use because ignoring Ok(value) discards the type proof
#[must_use]
pub const fn new() -> Self { ... }

// Queries — must_use because ignoring true/false is a silent security miss
#[must_use]
pub const fn dominates(&self, other: &Self) -> bool { ... }

#[must_use]
pub const fn is_setuid(self) -> bool { ... }

#[must_use]
pub const fn is_multiply_linked(self) -> bool { ... }
----

`#[must_use]` is free.
It costs nothing at runtime and adds a compile-time guarantee that every
caller is forced to make a decision about the result.

'''

== 4. Working with Numbers

Integer arithmetic on security-relevant values is not ordinary arithmetic.
An overflow in a sensitivity level comparison is not a logic bug —
it is a potential privilege escalation.
These patterns govern every numerical operation on classification,
permission, or index values.

=== 4.1 Checked and Saturating Arithmetic

*The danger.*
Rust's release builds disable overflow checks by default.
An integer that wraps silently can produce a value on the wrong side of
an access control boundary.
`s15 + 1` wrapping to `s0` is a privilege demotion;
a category index overflowing to `0` is a permissions collapse.

*The pattern.*
Use `checked_*`, `saturating_*`, or an explicit `.min()` guard for any
arithmetic on security-relevant values.
When an explicit `as` cast is necessary, suppress the Clippy warning
with `#[allow(clippy::cast_possible_truncation)]` and add a comment
explaining why the cast is provably safe.

*In the UMRS codebase.*

[source,rust]
----
// posix/primitives.rs
// nlink is u64 in MetadataExt but fits in u32 in practice.
// The .min() guard guarantees the value fits before the cast.
#[must_use]
pub fn from_u64(raw: u64) -> Self {
    #[allow(clippy::cast_possible_truncation)]
    Self(raw.min(u64::from(u32::MAX)) as u32)
}
----

The comment is required — it tells a reviewer why the `#[allow]` is safe
rather than lazy.
An unexplained `#[allow(cast_possible_truncation)]` in a security codebase
is a red flag; an explained one is an audit artifact.

*Citation.* NIST 800-218 SSDF PW.4.4

'''

=== 4.2 Bounds-Safe Indexing

*The danger.*
A `[i]` index operation panics in debug mode and produces undefined
behavior in release mode if `i` is out of bounds.
For security-relevant arrays, an out-of-bounds index is a security
event — not a logic error to be caught by the runtime.

*The pattern.*
Where the index comes from an external or unverified source, use
`.get(i)` and handle the `None` case explicitly.
Where the index comes from a type that was already validated at
construction, document why the access is provably safe.

*In the UMRS codebase.*

`CategorySet::index()` is provably safe because `Category::new()` already
validates that the id is ≤ 1023:

[source,rust]
----
// category.rs
// Safe: Category validates id <= 1023 at construction, so:
//   word = id / 64  is always in 0..=15  (fits [u64; 16])
//   bit  = id % 64  is always in 0..=63  (fits u64 shift)
const fn index(cat: Category) -> (usize, u64) {
    let id = cat.id() as usize;
    let word = id / 64;
    let bit  = id % 64;
    (word, 1u64 << bit)
}
----

The comment is the proof.
An auditor reading this code can verify: `Category` guarantees ≤ 1023;
therefore `word` ≤ 15; therefore `self.bits[word]` never panics.

*Citation.* NIST 800-218 SSDF PW.4.4

'''

=== 4.3 Fixed-Size Deterministic Layout

*The danger.*
A heap-allocated, variable-length structure for a security bitmask has
unpredictable timing, unpredictable memory layout, and complicates
serialization stability.
A `Vec<u64>` that grows on demand could silently change capacity and
layout between runs.

*The pattern.*
Use a fixed-size stack-allocated array for any security-relevant bitmask.
Zero heap allocation means zero allocator influence on timing or layout.
The serialized form is stable across Rust versions and architectures
when combined with documented endianness.

*In the UMRS codebase.*

[source,rust]
----
// category.rs
// 1024 bits = 16 words × 64 bits.
// Layout: [u64; 16], little-endian bit ordering.
// Stable: no heap, no allocator, auditable by inspection.
pub struct CategorySet {
    bits: [u64; 16],
}
----

*Citation.* NSA RTB (Deterministic Execution)

'''

=== 4.4 Lattice Dominance Math

*The danger.*
The kernel implements MLS dominance with `ebitmap_and()` plus a
comparison loop.
A custom implementation that diverges from the kernel formula — even
slightly — produces access control decisions that differ from what the
kernel would make.
That divergence is a security defect.

*The pattern.*
Implement the exact kernel formula.
Document the kernel equivalent by name.
Do not abstract or optimize in ways that obscure the correspondence.

*In the UMRS codebase.*

[source,rust]
----
// category.rs
// Kernel equivalent: ebitmap_and() + comparison loop.
// NIST 800-53 AC-4: Dominance Check (Lattice Mathematics).
//
// Subject dominates Object when Subject has at least all of Object's categories.
// Mathematically: (Subject & Object) == Object
pub fn dominates(&self, other: &Self) -> bool {
    for i in 0..16 {
        if (self.bits[i] & other.bits[i]) != other.bits[i] {
            return false;
        }
    }
    true
}
----

The formula is identical to the kernel's.
The comment names the kernel function so a reviewer can cross-check
against the kernel source.

*Citation.* NIST 800-53 AC-4 (Information Flow Enforcement)

'''

== 5. Reporting Errors and Security Findings

Errors and findings in UMRS are data, not text.
This section explains what that means and why it matters.

=== 5.1 Error Information Discipline

*The danger.*
An error message that includes a security label, a file path, a
username, or key material is visible to whoever receives the error.
In a CUI system, error strings can be logged in places with broader
access than the data they describe.

*The pattern.*
Error types are structured enums.
Variants carry the information needed for programmatic handling —
original OS errors, input lengths, violating characters — but never
security labels, key material, or classified data.
Sensitive context goes to audit logs, which have controlled access.
User-visible error strings describe the error category, not the
sensitive value that triggered it.

*In the UMRS codebase.*

[source,rust]
----
// secure_dirent.rs
// NIST SP 800-53 AU-3: error variants carry detail for audit
// record generation without requiring string parsing by the caller.
pub enum SecDirError {
    Metadata(std::io::Error),
    AccessDenied(std::io::Error),   // itself a security observation
    InvalidPath(PathError),
    InvalidFileName,
    SelinuxReadError(std::io::Error),
    SelinuxParseError,
}

impl std::fmt::Display for SecDirError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            Self::Metadata(e)         => write!(f, "metadata error: {e}"),
            Self::AccessDenied(e)     => write!(f, "access denied: {e}"),
            Self::InvalidPath(e)      => write!(f, "invalid path: {e}"),
            Self::InvalidFileName     => write!(f, "invalid filename component"),
            Self::SelinuxReadError(e) => write!(f, "SELinux xattr read error: {e}"),
            Self::SelinuxParseError   => write!(f, "SELinux context parse failed (both paths)"),
        }
    }
}
----

Note that `SelinuxParseError` carries no raw input — the input that
failed to parse is never included in the user-visible message.

*Citation.* NIST 800-53 SI-12 (Information Management and Retention)

'''

=== 5.2 Security Findings as Data

*The danger.*
Security findings buried in log strings — "WARNING: file is setuid" —
are visible in the log but invisible to code.
A calling program cannot query them, count them, sort them, or act on
specific ones without parsing text.
Text parsing of security findings is fragile and auditor-hostile.

*The pattern.*
Security findings are enum variants.
A `SecurityObservation::Setuid` is a value that can be matched,
counted, filtered, sorted, and serialized.
The enum is the finding record.
The log line, if any, is derived from the enum — not the other way
around.

*In the UMRS codebase.*
`SecureDirent` records security findings as `SecurityObservation`
enum values.
A caller can ask "does this entry have any high-severity findings"
without parsing any string:

[source,rust]
----
// Usage pattern — no string parsing, no text matching
let findings: Vec<&SecurityObservation> = dirent
    .observations()
    .iter()
    .filter(|o| o.is_high_severity())
    .collect();
----

*Citation.* NIST 800-53 AU-3 (Content of Audit Records)

'''

=== 5.3 Loud Failure

*The danger.*
A security check fires but the error is swallowed, logged at `debug!`,
or converted to an `Ok` with a default value.
The system continues operating as if nothing happened.
The adversary who triggered the condition learns that it is safe to
proceed; the operator never knows it occurred.

*The pattern.*
Every `Err` return on a security path is preceded by `log::error!`.
The error is loud, immediate, and auditable.
No security-relevant failure returns `Ok` with degraded content.

*In the UMRS codebase.*

[source,rust]
----
// xattrs.rs — TPI gate
if context_a != context_b {
    log::error!("CRITICAL: RTB Redundancy Failure!");
    log::error!("PATH A OBJECT: {context_a:?}");
    log::error!("PATH B OBJECT: {context_b:?}");
    return Err(io::Error::new(
        io::ErrorKind::PermissionDenied,
        "RTB Redundancy Failure: Security Ctx logic mismatch",
    ));
}
----

The `CRITICAL:` prefix is intentional — it makes the event
grep-able in audit logs and easy to detect in monitoring rules.

*Citation.* NSA RTB RAIN

'''

=== 5.4 Audit Cards

*The danger.*
A kernel attribute read occurs — enforce state, MLS status, FIPS mode —
and the only record is a raw value somewhere in a log.
An auditor reviewing the system cannot tell what attribute was read,
from which path, or what it means without additional context.

*The pattern.*
Every kernel attribute type can render itself as a structured audit
card showing: the kobject, the attribute name, the path, the value,
a description, and a kernel note.
The card is machine-formatted and grep-friendly.

*In the UMRS codebase.*

`AttributeCard<T>` owns its value and carries a `read_at: SystemTime`
timestamp that records when the read was performed.
The `Display` implementation prints the timestamp as a Unix-second value,
making the card self-contained: an auditor can correlate it with system
logs without consulting any external state.

The recommended production API is `SecureReader::read_with_card()`, which
produces a card that is proof the value was obtained through the full
provenance-verified read path.

[source,rust]
----
// umrs-platform: kattrs.rs
impl<T: KernelFileSource + StaticSource> fmt::Display for AttributeCard<T> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let ts = self.read_at
            .duration_since(std::time::UNIX_EPOCH)
            .unwrap_or_default()
            .as_secs();
        write!(
            f,
            "--- [ UMRS KERNEL ATTRIBUTE CARD ] ---\n\
             KernelObj : {}\n\
             Attribute : {}\n\
             Path      : {}\n\
             Value     : {:?}\n\
             ReadAt    : {ts}\n\n\
             Description: \n{}\n\n\
             Note: \n{}\n\
             ---------------------------------------",
            T::KOBJECT, T::ATTRIBUTE_NAME, T::PATH,
            self.value, T::DESCRIPTION, T::KERNEL_NOTE
        )
    }
}
----

*Citation.* NIST 800-53 AU-3 (Content of Audit Records)

'''

== 6. Mechanically Enforcing Safe Code

=== 6.1 `#![forbid(unsafe_code)]`

*The pattern.*
Every crate root in this workspace declares:

[source,rust]
----
// lib.rs
// NIST 800-218 SSDF PW.4.3 / NSA RTB: Provable safe-code guarantee.
// #![forbid] cannot be overridden by any inner #[allow].
// This is a compile-time proof, not a policy.
// An auditor can verify it by reading a single line.
#![forbid(unsafe_code)]
----

`#![forbid]` is stronger than `#![deny]`.
`#![deny]` can be locally overridden with `#[allow(unsafe_code)]`.
`#![forbid]` cannot — any inner `#[allow]` is itself a compile error.
The guarantee is mechanical and auditor-verifiable without reading
any other line of code.

*Why this matters.*
Unsafe Rust permits raw pointer arithmetic, direct memory access,
and FFI without any compiler-checked invariants.
One unsound `unsafe` block can invalidate the memory safety guarantees
that every other pattern in this guide depends on.
Forbidding it entirely eliminates that class of risk from the audit
surface.

*Citation.* NIST 800-218 SSDF PW.4.3 / NSA RTB

'''

=== 6.2 Clippy Policy

The workspace runs Clippy with `pedantic` and `nursery` lint groups
enabled and `-D warnings` in effect.
Any Clippy finding is a build failure.

Suppressing a lint requires `#[allow(lint_name)]` on the specific
function — not a global `#![allow]` — with a comment explaining why
the suggested rewrite would reduce clarity or correctness.
An unexplained `#[allow]` in a security codebase is a code review
finding.

The full list of suppressed lints and their rationale is maintained in
`CLAUDE.md` under the Clippy Policy section.

*Citation.* NIST 800-218 SSDF PW.4

== 7. Comparing Sensitive Values

[WARNING]
====
This section describes a pattern that is *required* but *not yet
implemented* in the codebase.
It must be in place before `vaultmgr` handles any key material,
tokens, or credentials.
====

=== 7.1 Constant-Time Comparison

*The danger.*
The machine compares two values by subtracting them and inspecting the
sign bit of the result.
When comparing byte sequences, the CPU stops at the first byte that
differs — there is no reason to continue.
This means a comparison of two 32-byte tokens takes measurably less
time when the first byte differs than when the first 31 bytes match.

An attacker who can make many comparison attempts and measure the
response time can learn, byte by byte, how close their guess is to
the correct value.
This is a timing oracle — a side-channel attack that requires no
memory access, no privilege escalation, and no crash.
It works over a network.

Standard `==` on `&[u8]`, `String`, or any slice type uses early-exit
comparison.
It is not safe for tokens, MACs, or credentials.

*The pattern.*
Use the `subtle` crate for any comparison of security-relevant byte
sequences.
`ConstantTimeEq` performs the comparison in a fixed number of steps
regardless of where the inputs first differ.

[source,rust]
----
// Required pattern — not yet in codebase
use subtle::ConstantTimeEq;

fn verify_token(provided: &[u8], expected: &[u8]) -> bool {
    // Fixed-time comparison — does not exit early on first difference.
    // Prevents timing oracle attacks on token and credential comparisons.
    provided.ct_eq(expected).into()
}
----

*Supply chain note.*
The `subtle` crate is maintained by the `dalek-cryptography` group and
is widely used in the Rust cryptographic ecosystem.
Before adding it, run `cargo audit` and verify it has no known
vulnerabilities.
It has no transitive dependencies.

*Scope.*
This pattern applies to any comparison involving:
tokens, session identifiers, MACs, HMACs, password hashes, or any
other value where knowledge of the correct answer provides an advantage
to an attacker.

*Citation.* NIST 800-218 SSDF PW.4.4 / NIST 800-53 SC-28

=== Note on PW.4.5 and FIPS Cryptography

FIPS 140-2/3 validation is a property of an *algorithm implementation*,
not of a crate name.
A crate that uses `AES-256-GCM` is not FIPS validated unless the
specific implementation has been through the CMVP process.

Common misconceptions:

* The `ring` crate uses strong algorithms but is *not* FIPS 140-2/3
  validated as of this writing.
* `openssl` linked against the system OpenSSL on RHEL 10 *may* be
  FIPS validated if the system is in FIPS mode and using the
  validated module — but this must be verified for each deployment.
* The choice of algorithm is necessary but not sufficient.
  The *module* that implements it must carry a CMVP certificate number.

Before writing any cryptographic code in this project, verify the
FIPS validation status of the specific implementation against the
NIST CMVP database (csrc.nist.gov/projects/cryptographic-module-validation-program).

See also the xref:cryptography:openssl-no-vendoring.adoc[OpenSSL no-vendoring]
decision in the Architecture module.

=== Note on Formal Verification

The patterns in this guide achieve security through the Rust type
system and design discipline.
AdaCore's SPARK/Ada language achieves equivalent guarantees through
formal mathematical proof — every function's pre- and post-conditions
are mechanically verified.
SPARK represents the tier above Rust in the high-assurance spectrum.
For systems where even Rust's structural guarantees are insufficient,
SPARK is the reference point.

== Extended Pattern Catalog

The patterns in the core library (xref:patterns:index.adoc[Pattern Library]) cover the
mechanisms most directly implemented in the UMRS codebase today. The catalog below covers
additional patterns that are either architectural disciplines, design-level obligations, or
patterns whose time to implement is approaching. They are organized in five groups.

Authoritative sources for this catalog: NIST SP 800-53 Rev 5, NIST SP 800-53B,
NIST SP 800-218 SSDF v1.1, ANSSI Secure Rust Guide, Rust Secure Code Working Group,
and the Rust Foundation Safety-Critical Rust Coding Guidelines effort.

=== Group A: API and Trust Boundary Design

These patterns govern how module boundaries are drawn and how authority is passed between them.

==== Least-Privilege API

*What it is.* Design modules so that callers receive only the operations they need.
Use narrow trait bounds, small wrapper types, and explicit constructor gating.
No "god object" with unrestricted authority.

*Threat.* Broad APIs grant callers capabilities they should not have.
An attacker who reaches the API surface can invoke operations outside the intended scope.

*In Rust.* Prefer narrow trait objects over concrete `&mut self` admin handles.
Expose distinct types for distinct operation classes rather than one configurable struct.
Gate construction behind a function that encodes the privilege level.

*Controls.* NIST SP 800-53 AC-6 (Least Privilege).

==== Separate Processing Domains

*What it is.* Split trusted and untrusted work into separate processes, crates, or modules
with explicit typed interfaces. Avoid shared mutable state across trust levels.

*Threat.* Collocating trusted and untrusted computation allows privilege escalation through
shared memory, shared references, or implicit side-channel coupling.

*In Rust.* Use message passing and typed request/response boundaries rather than shared
mutable references. In SELinux environments, distinct processes get distinct SELinux types,
which enforces the boundary at the kernel level.

*Controls.* NIST SP 800-53 AC-6(5) (Least Privilege — Privileged Accounts).

==== Explicit Trust Boundary

*What it is.* Mark every boundary where trust changes: file input, network input,
environment variables, command output, kernel pseudo-filesystems, external configuration,
xattrs, and IPC. Validate and normalize at the boundary; never deeper.

*Threat.* Untrusted data that passes a boundary without validation can corrupt internal
state, cause incorrect access decisions, or reach audit output.

*In Rust.* The boundary is the constructor call. All `TryFrom`/`FromStr`/`parse()`
implementations are trust boundaries. Anything beyond the constructor is already validated.
Never pass raw bytes or strings through more than one layer.

*Controls.* NIST SP 800-53 SI-10 (Information Input Validation).

==== Capability-Style Interface

*What it is.* Instead of granting global authority, pass objects that carry a single
specific capability: append to a log but not delete it; read an xattr but not write it.

*Threat.* Global authority handles (file handles, admin objects) can be misused if
they reach the wrong code path. An auditor cannot verify access is bounded without
inspecting every use site.

*In Rust.* Define wrapper types that expose only the operations they are meant to permit.
A type that holds a `File` opened read-only is a proof that the holder cannot write.

*Controls.* NIST SP 800-53 AC-6 / AC-3.

==== Complete Mediation

*What it is.* Every access to a protected resource goes through the same enforcement path.
No side doors, no bypass handles, no "already checked" shortcuts that skip policy.

*Threat.* Partial mediation is a classic vulnerability class. Code that validates once and
then passes a raw handle around allows callers to bypass policy by using the raw handle directly.

*In Rust.* Centralize privileged operations behind dedicated APIs.
The provenance verification pattern (section 1.2) applies this to kernel pseudo-filesystem reads.
Private constructors in security types apply it to label parsing.

*Controls.* NIST SP 800-53 AC-3 / NSA RTB RAIN (Non-Bypassability).

==== Trusted Path / Trusted Channel

*What it is.* For sensitive operations, use a deliberately constrained interface and
identify it as the trusted path. No side channels for critical actions. Stronger
authentication and clearer audit events on that path.

*Threat.* Untrusted paths used for privileged operations allow an attacker to bypass
the controls that are only applied to the designated trusted path.

*In Rust.* Encode the trusted path as a specific type or function. Gate access to it
behind the strictest available authentication and privilege checks. Ensure every call
on the trusted path emits an audit event.

*Controls.* NIST SP 800-53 SC-11 (Trusted Path).

'''

=== Group B: Type System and Construction Discipline

These patterns use Rust's type system to make invalid states unrepresentable.

==== Strongly Typed Security Domain Model

*What it is.* Represent sensitive concepts as distinct types: `SelinuxType`, `CapabilitySet`,
`TrustedPath`, `ValidatedLabel`, `AuditEventId`, `NonEmptyPath`. Move invalid state out of
the reachable program state space.

*Threat.* Stringly-typed or weakly-typed security data is easily confused, mixed, or misrouted.
A function that accepts `String` cannot distinguish a security label from a filename.
Type confusion leads to incorrect access decisions and audit gaps.

*In Rust.* Newtype wrappers with private fields and validated constructors.
`SecurityContext` is the primary example: it is not a `String`, and it cannot be constructed
from a `String` without going through the dual-parser TPI path.

*Controls.* NIST SP 800-218 SSDF PW.4.1 / NIST SP 800-53 SI-10.

==== Trusted-Constructor Pattern

*What it is.* Make critical types constructible only through checked constructors
(`TryFrom`, `FromStr`, dedicated builders) that enforce invariants.
Never expose unchecked public fields on security-sensitive types.

*Threat.* Public fields allow callers to construct values that bypass validation.
A type constructed without the validated constructor can hold invalid or attacker-controlled state.

*In Rust.* All fields on security-relevant types are private.
The only path to a valid instance is through a constructor that validates and returns `Result`.
This is "validate at construction" — the UMRS compile-time contract rule.

*Controls.* NIST SP 800-218 SSDF PW.4.1 / NSA RTB RAIN.

==== Immutable-by-Default Policy

*What it is.* Make configurations, policy objects, and validated security decisions
immutable after construction. Mutability is narrow and explicitly visible.

*Threat.* Mutable configuration objects can be modified after validation.
Post-construction mutation creates windows where the object holds an inconsistent state.
It also obscures the provenance of the current value.

*In Rust.* Validated types with private fields are immutable by default — there are no
setters that bypass the constructor's invariants. If a policy object must be updated,
construct a new one from scratch through the same validated constructor.

*Controls.* NIST SP 800-53 CM-5 (Least Privilege — Access Restrictions for Change).

==== Explicit Lifecycle / State-Machine

*What it is.* Represent security-sensitive workflows as enums plus explicit transitions
rather than ad hoc flags. Make illegal state transitions unrepresentable.

*Threat.* Flag-based state management allows code to reach impossible state combinations
(e.g., "validated but not parsed", "sealed but no key"). These states produce undefined behavior
that is hard to reason about and easy to exploit.

*In Rust.* Use enums to model lifecycle states. Transitions are functions that consume the
old state and produce the new one. Callers cannot hold a reference to both states simultaneously.

[source,rust]
----
// The compiler enforces that you cannot use a sealed result after breaking the seal.
enum CacheState { Empty, Pending(PendingResult), Sealed(SealedResult) }
----

*Controls.* NIST SP 800-218 SSDF PW.4 / NIST SP 800-53 SA-8.

'''

=== Group C: Reliability and Correctness Discipline

These patterns prevent the failure modes that create security vulnerabilities in critical paths.

==== No-Panic Production Path

*What it is.* In production code paths, return structured errors rather than panicking.
Treat panics as exceptional — reserved for truly unrecoverable programming errors.

*Threat.* A panic in a high-assurance component terminates the process. In a security
mediator, this is a denial-of-service vulnerability and a fail-open risk if callers
assume success on an interrupted path.

*In Rust.* Use `Result` and `?` throughout. Avoid `.unwrap()` — it is denied in this
codebase. Use `.expect()` only where a panic genuinely indicates a programming error that
should never occur at runtime, and add a comment explaining why the invariant holds.
The `#[deny(clippy::unwrap_used)]` lint enforces this.

*Controls.* NIST SP 800-53 SI-17 (Fail-Safe Procedures) / NSA RTB RAIN.

==== Deterministic Error Taxonomy

*What it is.* Use explicit error enums with stable, well-named variants.
Each variant represents a distinct failure category with enough context for audit and diagnosis.
Avoid stringly-typed or ad hoc errors in security-critical paths.

*Threat.* Generic or string-based errors make it impossible to programmatically
distinguish failure causes. Callers cannot filter by category, security tools cannot
classify events, and auditors cannot trace failures to their root cause.

*In Rust.* Define a crate-level or module-level `Error` enum. Each variant covers
one failure class. Implement `Display` for human-readable messages but keep the enum
variants as the canonical representation. Do not nest `String` where a variant suffices.

*Controls.* NIST SP 800-53 AU-3 (Audit Record Content) / NIST SP 800-218 SSDF RV.1.

==== Secure Arithmetic (boundary discipline)

The core pattern is covered in xref:patterns:pattern-secure-arithmetic.adoc[Secure Arithmetic].
The additional discipline here: apply checked arithmetic to any integer derived from
security-relevant input even when overflow seems unlikely. The value could come from an
attacker-controlled file or kernel attribute. "Unlikely" is not a guarantee.

==== Verification by Test Vector

*What it is.* For parsers, policy logic, and normalization code, define deterministic
"known-good / known-bad" test vectors and keep them versioned with the code.

*Threat.* Parser or policy logic that passes unit tests on well-formed input may fail
silently on edge cases, malformed input, or adversarial input. Without explicit test
vectors, regressions are not caught until production.

*In Rust.* Create integration tests (under `tests/`) with explicitly named vectors covering:
valid canonical input, valid non-canonical input (whitespace, alternate formats), boundary
values, and known-invalid input that must be rejected. Version the vectors alongside the code.

*Controls.* NIST SP 800-218 SSDF PW.7 (Test Executable Code) / NIST SP 800-53 SA-11.

'''

=== Group D: Operational Discipline and Governance

These patterns connect the codebase to auditable, assessable evidence of compliance.

==== Audit-First Security Event Design

*What it is.* For security-relevant actions, define the audit event before implementing the code.
Specify: what happened, who requested it, what object was affected, what decision was made,
what evidence supports the decision.

*Threat.* Code written first, logging added as an afterthought produces incomplete or inconsistent
audit records. An assessor reviewing logs cannot reconstruct the access decision from partial data.

*In Rust.* Define the audit event as a struct or enum variant. The code path emits the event
as part of its normal execution, not as an afterthought appended after the fact. The log
record should be self-contained: a reader should be able to reconstruct the access decision
from the record alone.

*Controls.* NIST SP 800-53 AU-3 / AU-12 / CMMC CA.L2-3.3.1.

==== Provenance-Carrying Data

*What it is.* Attach provenance metadata to critical data as it moves through the system:
source, validation status, timestamp, authority, digest, and transformation history.
The Sealed Evidence Cache (SEC) pattern is a specific implementation of this principle.

*Threat.* Data that crosses module boundaries loses its provenance. A downstream module
that receives a validated label cannot verify that validation actually occurred, or when.
An attacker who injects data after the validation step can feed unvalidated values into enforcement logic.

*In Rust.* Wrapper structs that carry payload plus metadata.
The metadata is set once at construction and is inaccessible for modification.
Downstream callers receive both the value and its provenance proof.

See xref:patterns:pattern-sec.adoc[Sealed Evidence Cache] for the implemented example.

*Controls.* NIST SP 800-53 SI-7 / NIST SP 800-218 SSDF PW.4.

==== Known-Good Baseline

*What it is.* Establish canonical defaults, canonical configuration serialization,
and known-good startup baselines. Validate runtime configuration against those baselines
and detect drift.

*Threat.* Systems that start from unknown or partially-modified configuration may be operating
outside the validated posture. Drift from the known-good baseline is a common precursor to
both security failures and compliance findings.

*In Rust.* Represent the known-good baseline as a typed constant or configuration struct.
Load runtime configuration through the same validated constructor. Compare loaded configuration
against the baseline at startup and emit an audit event on drift.

*Controls.* NIST SP 800-53 CM-6 (Configuration Settings) / CMMC CM.L2-3.4.2.

==== Security Control Traceability

*What it is.* Every security-critical module, type, and function carries explicit citations
to the NIST or NSA control it satisfies. Source code, test cases, and documentation all
point back to the same requirement.

*Threat.* Undocumented security decisions cannot be verified by an auditor.
The auditor must assume the implementation is correct or re-derive the intent from first principles.
Unannotated security code fails a compliance review.

*In Rust.* Module-level `//!` doc blocks cite the governing controls.
Security-critical types and functions carry `///` citations.
See xref:compliance-annotations.adoc[Compliance annotations guide] for the full requirement.

*Controls.* NIST SP 800-218 SSDF PW.1 / NIST SP 800-53 SA-8 / NIST SP 800-53A.

==== Evidence-Producing Design

*What it is.* Design the system so it naturally emits the evidence an assessor would request:
structured logs, configuration snapshots, version data, hashes, policy identifiers,
and validation results. Evidence is a first-class output of the system, not an afterthought.

*Threat.* Systems designed without assessability in mind require retroactive instrumentation
during audits. Retroactive instrumentation is expensive and often incomplete.

*In Rust.* Structure `Display` and `Debug` impls to emit parseable key=value output.
Emit structured log events at every significant decision point.
Include version and configuration information in startup log output.
Design for `cargo doc` to produce auditable API documentation with control citations.

*Controls.* NIST SP 800-53A (Assessment Procedures) / NIST SP 800-218 SSDF PO.4.

==== Subset-of-Privileged-Users

*What it is.* Administrative functions — audit management, configuration mutation,
key operations — are callable only through explicitly separated roles or interfaces.
Operational and administrative APIs are distinct.

*Threat.* Conflating operational and administrative APIs allows operational code paths
to trigger privileged operations without explicit privilege elevation.

*In Rust.* Separate admin types from operational types. An operational caller cannot
obtain an admin handle without explicit privilege grant. The type system enforces this
separation structurally.

*Controls.* NIST SP 800-53 AC-6(5) / CMMC AC.L2-3.1.6.

=== Group E: Process and Workspace Governance

These patterns apply at the workspace, toolchain, or development process level.

==== Small Unsafe Island

*What it is.* If `unsafe` is ever necessary (e.g., FFI, performance-critical operations
with external libraries), isolate it in a tiny, dedicated module. Document the invariants
that the unsafe code relies on. Wrap it in a safe public API. Keep the unsafe surface
auditable.

*Threat.* `unsafe` scattered throughout a codebase is an audit liability. An auditor
must locate and review every `unsafe` block. Undocumented `unsafe` blocks have no stated
invariants and cannot be verified correct.

*Note for UMRS.* The codebase currently has `#![forbid(unsafe_code)]` in every crate root.
This means unsafe is not currently used anywhere. If a future requirement makes unsafe
necessary, this pattern defines how to contain it.

*Controls.* NIST SP 800-218 SSDF PW.4.3 / NIST SP 800-53 SA-11.

==== Unsafe-Surface Measurement

*What it is.* Use `cargo-geiger` to measure the total unsafe code surface in the crate
graph — both direct and transitive. Track this metric over time. Any increase triggers review.

*Threat.* A dependency added for convenience may introduce unsafe code that is invisible
in a code review of first-party code. The unsafe surface grows silently.

*In practice.* Run `cargo geiger` before each release. Treat an increase in unsafe count
as a finding requiring justification. This is a governance metric, not a blocker — but
the trend must be understood.

See also xref:patterns:pattern-supply-chain.adoc[Supply Chain Hygiene].

*Controls.* NIST SP 800-218 SSDF PW.3 / NIST SP 800-53 SR-3.

==== Safe Replacement of Unsafe Dependencies

*What it is.* Prefer safe Rust crates over crates with large unsafe surfaces.
When a safe equivalent exists, prefer it. Track opportunities to replace unsafe-heavy
dependencies in the workspace.

*Threat.* Unsafe code in dependencies is outside the control of this project but contributes
to the attack surface and complicates formal review.

*In practice.* The Rust Secure Code WG maintains the "safety-dance" effort cataloging
safe alternatives. Use `cargo-geiger` to identify the highest-unsafe-surface dependencies
and evaluate whether safer alternatives exist.

*Controls.* NIST SP 800-218 SSDF PW.3 / NIST SP 800-53 SR-3 / SR-5.

==== Configuration Is Code-Like

*What it is.* Treat security-relevant configuration as typed, validated, versioned input
with migration rules and integrity checks. Configuration is not free-form text.

*Threat.* Unvalidated configuration can produce incorrect security decisions silently.
An attacker or misconfiguration that changes a security-relevant setting without validation
or audit may go undetected.

*In Rust.* Parse configuration files through the same validated-constructor pattern used
for runtime input. Log configuration at startup (values permitting — see error information
discipline). Detect and log drift from the known-good baseline on startup.

*Controls.* NIST SP 800-53 CM-6 / NSA RTB Trust Gate.

==== Reviewability Over Cleverness

*What it is.* In security-critical paths, prefer simple, explicit, auditable code over
clever abstractions. The code must be understandable by a reviewer who is not the author
and did not write the code.

*Threat.* Clever abstractions obscure intent. An auditor reviewing a dense combinator chain
cannot verify the security property it encodes. Cleverness introduces review gaps.

*In Rust.* Break complex method chains into intermediate `let` bindings with descriptive names.
Use `if let` / `match` instead of `.map_or_else()` when the intent is clearer. Clippy aesthetic
lints that reduce readability are explicitly suppressed in this codebase — see `CLAUDE.md`.

*Controls.* NIST SP 800-218 SSDF PW.2 (Review Software Designs).

==== Defense-in-Depth Composition

*What it is.* No single mechanism is sufficient. Rust type safety, explicit validation,
DAC, capabilities, seccomp, SELinux policy, audit logging, and configuration management
must reinforce each other. Each layer assumes the others may be absent or compromised.

*Threat.* Reliance on a single control creates a single point of failure. An attacker who
bypasses one layer reaches an unprotected system.

*In practice.* When adding a new feature, enumerate the layers it touches. Identify which
layers are currently active and which are absent or unverified. Flag gaps. The UMRS design
explicitly combines SELinux policy, Rust type safety, provenance verification, TOCTOU safety,
and audit logging because no single mechanism covers all adversary capabilities.

*Controls.* NIST SP 800-53 SC-39 (Process Isolation) / NIST SP 800-53 AC-4 / NSA RTB VNSSA.

==== Performance-Secondary-but-Measured

*What it is.* Correctness and security come first. Performance is optimized only with
explicit measurement that demonstrates the invariants are preserved by the optimization.

*Threat.* Premature optimization introduces complexity, reduces reviewability, and creates
subtle bugs. Performance optimizations in security-critical code have historically introduced
timing side-channels and integer errors.

*In practice.* Build the correct, strongly typed version first.
Measure with explicit benchmarks before optimizing.
Any optimization that changes behavior (caching, batching, short-circuiting) must be
reviewed for security impact. See the xref:patterns:pattern-execution-measurement.adoc[Execution Measurement] pattern for timing instrumentation.

*Controls.* NIST SP 800-218 SSDF PW.4 (general engineering discipline).
