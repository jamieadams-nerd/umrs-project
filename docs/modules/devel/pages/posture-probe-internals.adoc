== Overview

The kernel security posture probe lives in `umrs-platform/src/posture/`.
It reads kernel security state from procfs, sysfs, securityfs, and modprobe.d
and produces a `PostureSnapshot` — an immutable, point-in-time assessment of
whether the system's kernel-level security controls meet the hardened baseline.

The probe is not a scanner.
It does not search for vulnerabilities or test exploit paths.
It answers a narrower question: for each of the indicators in the static catalog,
is the kernel currently configured as the hardened baseline requires?

== Indicator Taxonomy

Every indicator is described by an `IndicatorDescriptor` in the `INDICATORS` static array.
The descriptor is `const` — all values are compiler-verified, not loaded at runtime.
This satisfies NIST SP 800-218 SSDF PW.4 (compile-time binding of security-relevant
constants) and eliminates the attack surface of a runtime-parsed configuration file.

=== IndicatorClass

`IndicatorClass` determines where the live value comes from and whether contradiction
detection applies.

[cols="2,3,2", options="header"]
|===
| Variant | Live source | Contradiction detection

| `Sysctl`
| `/proc/sys/` (verified against `PROC_SUPER_MAGIC`)
| Yes — compared against sysctl.d merge tree

| `KernelCmdline`
| `/proc/cmdline` (verified against `PROC_SUPER_MAGIC`)
| Yes — compared against BLS bootloader entry

| `SecurityFs`
| `/sys/kernel/security/` (verified against `SECURITYFS_MAGIC`)
| No — LSM state is not user-configurable via sysctl.d

| `DistroManaged`
| `/proc/sys/crypto/fips_enabled`
| Partial — FIPS path only

| `ModprobeConfig`
| `/sys/module/` (verified against `SYSFS_MAGIC`)
| Yes — compared against modprobe.d merge tree

|===

All live reads route through the `kattrs::traits::StaticSource::read()` default
method, which verifies filesystem magic via `fstatfs` before reading the node.
The provenance check cannot be skipped — it is the default method path, not an
opt-in call.
This satisfies NSA RTB RAIN (Non-Bypassability).

=== DesiredValue

`DesiredValue` expresses the hardened baseline as a type-safe comparison rather
than a string.
The `meets_*` methods implement the check for each variant.

[cols="2,3", options="header"]
|===
| Variant | Meaning

| `Exact(u32)`
| Live value must equal this integer exactly

| `AtLeast(u32)`
| Live value must be >= this threshold

| `AtMost(u32)`
| Live value must be \<= this threshold

| `CmdlinePresent(&'static str)`
| This exact token must appear in `/proc/cmdline` (whitespace-delimited)

| `CmdlineAbsent(&'static str)`
| This exact token must NOT appear in `/proc/cmdline` (whitespace-delimited)

| `Custom`
| Indicator-specific logic; `meets_integer` and `meets_cmdline` return `None`

|===

Token matching for `CmdlinePresent` and `CmdlineAbsent` is whole-word.
`mitigations=off` does not match `mitigations=off,nosmt` — they are different
whitespace-delimited tokens.
This prevents false negatives where a combined flag bypasses detection.

== CPU Mitigation Sub-Indicators (Phase 2b)

=== The Problem

Phase 1 includes the `Mitigations` indicator, which checks that `mitigations=off` is
absent from `/proc/cmdline`.
This umbrella flag disables all CPU vulnerability mitigations in one switch.

However, the kernel also accepts per-CVE weakening overrides — individual flags that
disable a specific mitigation while leaving others active.
A kernel command line such as:

----
spectre_v2=off mds=off
----

does not trigger the `Mitigations` indicator, because `mitigations=off` is not present.
Phase 2b adds eight sub-indicators, one per weakening override, to catch selective
disabling.

=== The Pattern

All eight CPU mitigation sub-indicators share the same structure:

[cols="1,3", options="header"]
|===
| Field | Value

| `class`
| `IndicatorClass::KernelCmdline`

| `live_path`
| `/proc/cmdline`

| `sysctl_key`
| `None` — no sysctl key; cmdline only

| `desired`
| `DesiredValue::CmdlineAbsent("<flag>=off")`

| `impact`
| `High` or `Medium` depending on CVE severity

|===

An indicator passes when its `CmdlineAbsent` token is not present.
Absent means the mitigation is either active at the kernel default or not applicable
to this CPU — both are safe outcomes.
The indicator detects only the explicit weakening override; it does not distinguish
between "mitigation active" and "mitigation not applicable to this hardware."

The umbrella indicator and the sub-indicators are complementary, not redundant.
A clean `Mitigations` result (no `mitigations=off`) does not guarantee clean
sub-indicator results.
Both must pass for a complete boot integrity assessment.

=== Indicator Table

[cols="2,1,3", options="header"]
|===
| `IndicatorId` | Impact | Token checked (must be absent)

| `SpectreV2Off`
| High
| `spectre_v2=off`

| `SpectreV2UserOff`
| Medium
| `spectre_v2_user=off`

| `MdsOff`
| High
| `mds=off`

| `TsxAsyncAbortOff`
| Medium
| `tsx_async_abort=off`

| `L1tfOff`
| High
| `l1tf=off`

| `RetbleedOff`
| High
| `retbleed=off`

| `SrbdsOff`
| Medium
| `srbds=off`

| `NoSmtOff`
| Medium
| `nosmt=off`

|===

NOTE: `NoSmtOff` checks for `nosmt=off` — the flag that re-enables Simultaneous
Multi-Threading when the kernel was booted with `nosmt`.
If a site disabled SMT to improve MDS and L1TF isolation, `nosmt=off` silently
undoes that decision.
This indicator detects that reversal.

=== Reader Path

`KernelCmdline` indicators share a single `/proc/cmdline` read.
`PostureSnapshot::collect()` reads the cmdline once at collection time and passes
the string to each indicator's `DesiredValue::meets_cmdline()` call.
No per-indicator procfs open is performed.

No reader change is required when adding a new `KernelCmdline` sub-indicator.
The only code changes are an `IndicatorId` variant in `indicator.rs`, a `label()`
match arm, and an `IndicatorDescriptor` entry in `catalog.rs`.

=== Adding a New Sub-Indicator

When a new per-CVE mitigation weakening flag appears in a kernel release, add it
in the section of `indicator.rs` marked:

[source,rust]
----
// ── CPU mitigation sub-indicators (Phase 2b) ─────────────────────────
----

Use `IndicatorClass::KernelCmdline` and `DesiredValue::CmdlineAbsent("flag=off")`.
Set `impact` to `High` if the CVE enables cross-process or cross-VM memory reads;
set it to `Medium` for narrower threat models (Intel-specific hardware, or attacks
requiring prior access).

IMPORTANT: After adding the variant, update the exhaustive match in
`catalog_covers_all_indicator_ids` in `posture_tests.rs`.
The test will fail to compile — not fail at runtime — if a variant is missing.
This is intentional: the compiler enforces catalog completeness.

For the complete six-step process, see
xref:posture-probe.adoc#_how_to_add_a_new_indicator[How to Add a New Indicator].

== CorePattern — TPI Classification (Phase 2b)

=== Background

`kernel.core_pattern` is a sysctl that controls how the kernel handles core dumps
when a process crashes.
Two formats are valid:

* A file path template — the kernel writes the dump directly to the filesystem.
* A pipe handler — a value beginning with `|` causes the dump to be piped to a
  program such as `|/usr/lib/systemd/systemd-coredump`.

On hardened systems, the piped handler format is required.
A managed handler can apply access control, compression, and encryption before
storing the dump.
A raw filesystem path writes process memory — which may include key material,
credentials, and CUI data — directly to disk with no handler accountability.

WARNING: Core dumps from privileged processes are a ready-made data extraction path.
`kernel.core_pattern` configured as a raw filesystem path means any user who can
read the dump file gains access to the crashed process's complete memory contents,
including credentials and private keys.

The indicator uses `IndicatorClass::Sysctl` with `DesiredValue::Custom` because neither
an integer comparison nor a cmdline token check can express the hardened state.
Classification is handled by a dedicated TPI function.

=== Why TPI Is Applied Here

The check "does the value start with `|`?" is not sufficient on its own.

A bare `|` (pipe character only, no path) passes a single byte check — its first
byte is `|`.
But it is not a valid handler; it names no program to pipe the dump to.

Likewise, `|usr/bin/handler` starts with `|` but the path is not absolute.
A handler with a relative path depends on the working directory at crash time,
which is not under operator control.

Two independent validation paths applied to the same input detect these edge cases.
This is the TPI pattern: two parsers, different algorithms, fail closed on any
disagreement.
See xref:patterns:pattern-tpi.adoc[Two-Path Independence (TPI)].

=== The Two Paths

The outer function `classify_core_pattern` calls `classify_core_pattern_inner`,
which implements both paths and combines the results at a gate.

*Path 1 — Structural*

Checks whether the first byte of the raw string is `|` (ASCII 0x7C).
This is a byte-level check with no interpretation of the remaining content.

[source,rust]
----
let path1_is_handler = value.as_bytes().first() == Some(&b'|');
----

*Path 2 — Semantic*

Trims whitespace, takes the first whitespace-delimited token, checks that the token
starts with `|`, and verifies that the character immediately after `|` is `/`
(an absolute path).

[source,rust]
----
let path2_is_handler = {
    let first_token = value.split_whitespace().next().unwrap_or("");
    if let Some(after_pipe) = first_token.strip_prefix('|') {
        after_pipe.starts_with('/')
    } else {
        false
    }
};
----

Path 2 rejects a bare `|` (no character after the pipe), `|usr/bin/handler`
(relative path), and any value where the first token does not begin with `|`.

*TPI Gate*

[source,rust]
----
match (path1_is_handler, path2_is_handler) {
    (true, true)   => CorePatternKind::ManagedHandler,
    (false, false) => CorePatternKind::RawPath,
    _ => {
        log::warn!(
            "posture: core_pattern TPI disagreement: \
             path1_handler={path1_is_handler} \
             path2_handler={path2_is_handler} — classified as Invalid (fail-closed)"
        );
        CorePatternKind::Invalid
    }
}
----

Path disagreement is logged at `warn!` level with both path results for audit
visibility.
The result is `Invalid`, which the snapshot treats as unhardened.

=== Classification Outcomes

[cols="2,1,3", options="header"]
|===
| `CorePatternKind` | Hardened? | Condition

| `ManagedHandler`
| Yes
| Both paths agree: first byte is `\|` and remainder is an absolute path

| `RawPath`
| No
| Both paths agree: no leading `\|`; dump writes directly to filesystem

| `Invalid`
| No (fail-closed)
| Paths disagree, or value is empty

|===

=== Example Values

[cols="3,2,3", options="header"]
|===
| Value | Classification | Reason

| `\|/usr/lib/systemd/systemd-coredump`
| `ManagedHandler`
| Both paths agree: `\|` present, handler path is absolute

| `\|/bin/true`
| `ManagedHandler`
| Valid minimal handler path

| `/var/crash/core.%e.%p`
| `RawPath`
| No `\|` prefix; raw filesystem path template

| `core`
| `RawPath`
| No `\|` prefix; relative path template

| `` (empty string)
| `Invalid`
| Empty input; both paths fail

| `\|`
| `Invalid`
| Path 1 passes (byte is `\|`); Path 2 fails (no absolute path after `\|`)

| `\|usr/bin/handler`
| `Invalid`
| Path 1 passes; Path 2 fails (path after `\|` is not absolute)

|===

The `Invalid` cases are the reason TPI is applied.
A structural-only check would classify `|` and `|usr/bin/handler` as hardened —
both have `|` as the first byte.
The semantic path rejects both because neither identifies a valid absolute handler.

=== How `snapshot.rs` Uses the Classification

`PostureSnapshot::collect()` handles `CorePattern` through a dedicated dispatch arm.
The arm calls `read_live_core_pattern`, which reads the sysctl via `CorePatternReader`
(provenance-verified through `StaticSource::read`) and applies `classify_core_pattern`
to the result.

The `meets` outcome is:

* `Some(true)` if kind is `ManagedHandler`
* `Some(false)` if kind is `RawPath` or `Invalid`
* `None` if the kernel node could not be read

`None` is reported as unknown, not as a failure.
Unavailability is distinct from a policy violation.

=== Pattern Execution Measurement

Both `CorePatternReader::parse` and `classify_core_pattern` log wall-clock
duration in debug builds:

[source,rust]
----
#[cfg(debug_assertions)]
log::debug!(
    "posture: TPI classify_core_pattern completed in {} µs, result={result:?}",
    start.elapsed().as_micros()
);
----

Timing is logged only in `#[cfg(debug_assertions)]` builds and is not emitted in
release builds.
This satisfies the Pattern Execution Measurement Rule.

== Fail-Closed Behavior Across the Probe

All indicators fail in the same direction: if a value cannot be read, the indicator is
reported as unknown (`?`), not as hardened.
A missing kernel node is not treated as a hardening confirmation.

`CorePatternKind::Invalid` applies the same principle at the classification level.
A value that cannot be unambiguously classified as a managed handler is not
reported as hardened.

The underlying principle is documented in
xref:patterns:pattern-fail-closed.adoc[Fail-Closed].
