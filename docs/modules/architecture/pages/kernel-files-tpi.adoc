= Two-Path Independence in Kernel File Parsing
:toc: left
:description: How the TPI pattern is applied to kernel attribute file parsing in UMRS to satisfy NSA RTB RAIN requirements.

To satisfy the Redundant invariant of RAIN, UMRS never trusts a single parsing
algorithm. A bug in a library or a subtle logic error in a state machine could lead to
an incorrect "Allow" decision.

For CUI labels, UMRS implements:

`Path A (nom)`::
A declarative, combinator-based parser. Highly readable and great for structural
validation.

`Path B (Manual State Machine)`::
A low-level, iterative approach that manually steps through bytes. "Dumb" logic —
easy to audit for off-by-one errors.

== Why Two Correct Parsers Are Better Than One

To an architect, this is the most common critique of the Redundant requirement in
high-assurance engineering. But in the NSA RTB world, the goal is not just to have
a "working" parser — it is to protect against a logic flaw in the parser itself that
an attacker could exploit.

If you have one "perfect" parser and it has a bug you did not see, the attacker wins.
If you have two different algorithms and one has a bug, the system detects the
disagreement and stops the attack.

=== The "Type Confusion" Attack

Imagine checking if a user can access a file labeled `umrs_secret_t`.

Path A (complexity): uses a high-level library like `nom`. Fast and handles complex
strings, but suppose it has a bug where it stops parsing if it sees a null byte (`\0`)
or a specific escape character.

Path B (simplicity): a "dumb" byte-iterator that counts colons. It does not care about
escape characters.

The attack: an attacker passes a malicious string:
`user:role:umrs_secret_t\0:dummy_level`.

* Path A hits the `\0`, stops, and returns `umrs_secret_t`. (Access granted.)
* Path B counts the bytes, sees the whole string, and returns `umrs_secret_t\0`.
* The TPI gate: `umrs_secret_t != umrs_secret_t\0`. *Denied.*

=== Why This Matters for Category Ranges

SELinux category ranges can be tricky: `c0.c3` means `c0`, `c1`, `c2`, `c3`.

A "clever" parser (Path A) might have a bug where it interprets `c0.c3` as just `c0`
and `c3` (missing the middle values). A "dumb" manual parser (Path B) sees the `.` and
correctly iterates across the full range. A mismatch is detected. The system halts.

== Implementing TPI in UMRS

UMRS implements this pattern in `SecurityContext` using `nom` and `FromStr`:

[source,rust]
----
/// NIST 800-53 AC-4 / NSA RTB (Redundant)
/// Ensures two independent logic paths agree on a security label.
pub fn validate_label_redundant(raw_input: &str) -> io::Result<MlsLabel> {
    // Path A: The "Modern" Parser (nom)
    let label_a = parse_with_nom(raw_input)?;

    // Path B: The "Conservative" State Machine (FromStr)
    let label_b = parse_with_state_machine(raw_input)?;

    // The TPI Gate: If they don't agree exactly, FAIL CLOSED.
    if label_a != label_b {
        return Err(io::Error::new(
            io::ErrorKind::InvalidData,
            "RTB Integrity Failure: Redundant parsing mismatch"
        ));
    }

    Ok(label_a)
}
----

Both paths are correct implementations of the specification. The value appears when
invalid or malicious data enters the system — the two paths will disagree, and access
is denied.

== Real-World RTB Implementation Note

In actual Raise the Bar systems, requirements often include:

* *Algorithm Diversity* — one parser written in one paradigm, one in another (e.g., Rust
  combinator vs. C-style iteration).
* *Vendor Diversity* — two different teams writing to the same specification.

In UMRS, the equivalent is `nom` (declarative) and `FromStr` (imperative). They are
both correct ways to find the fields in a SELinux context, but they use different parts
of the CPU and different memory logic to do it.

== Applying This to 1024-Bit Category Bitmasks

The TPI pattern applies not just to string parsing but to any security-relevant
computation. For `CategorySet` bitmask dominance checks:

* One path computes dominance using the bitmask operations (`&` and equality).
* A second path verifies the result using a different traversal (e.g., iteration over
  set bits).

If both approaches agree, the dominance check is trusted. If they disagree, the system
fails closed.

== Compliance Mapping

[cols="1,3"]
|===
| Standard | Satisfaction

| `NSA RTB RAIN — Redundant`
| Two independent parsing paths must agree; disagreement triggers fail-closed denial.

| `NIST 800-53 SI-7`
| Software integrity is maintained by preventing a single logic flaw from resulting in
  an incorrect security decision.

| `NIST 800-53 AC-4`
| Information flow enforcement depends on correct label parsing; TPI ensures the
  parse is trustworthy.
|===
