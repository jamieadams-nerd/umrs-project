= Strong-Typing Rationale
:description: Why UMRS models SELinux security contexts as validated, domain-specific types rather than raw strings.
:toc: left

This page collects the rationale for UMRS's strong-typing approach to
SELinux security context modeling.

For the full treatment, see
xref:rationale-strongly-typed.adoc[Rationale for a strongly-typed SELinux modeling library].

== Summary

Traditional SELinux tooling represents security contexts as unstructured
strings. In a high-assurance system, this is insufficient. A raw string
can be malformed, truncated, or constructed from untrusted input and
passed directly to a kernel interface — with no validation boundary between
userland and kernel policy enforcement.

UMRS models each field of a security context as a validated Rust type.
A `SecurityContext` cannot be constructed with an invalid sensitivity
level, an out-of-range category ID, or a missing required field. The type
system enforces structural validity at the point of construction, before
any value reaches the kernel.

This approach satisfies NSA RTB RAIN (security checks always invoked),
NIST 800-53 AC-4 (information flow enforcement), and NIST 800-218 SSDF
PW.4 (secure coding practices).

See xref:rationale-strongly-typed.adoc[Rationale for a strongly-typed SELinux modeling library]
for the full argument, detailed examples, and threat analysis.
