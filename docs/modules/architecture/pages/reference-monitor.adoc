= Reference Monitor Design in UMRS
:toc: left
:description: How UMRS implements the reference monitor concept using NSA RTB RAIN principles and TOCTOU-safe inode anchoring.

In high-assurance security engineering, a *reference monitor* is the conceptual heart of
a secure system. The original Anderson Report (1972) defined the core properties of a
reference monitor. NSA Raise the Bar (RTB) evolves these into the RAIN principle for
modern Cross-Domain Solutions (CDS).

== The RAIN Principle (NSA RTB Core)

NSA RTB mandates that a security mechanism follow the RAIN concept to ensure it is
effective against persistent, advanced threats.

`R — Redundant`::
Security mechanisms must have overlapping layers. In UMRS code, this is the TPI
(Two-Path Integrity) gate where two independent parsers must agree on the label before
access is granted.

`A — Always Invoked`::
The monitor must be called for every single access attempt. By anchoring the tool to
the file descriptor, enforcement is invoked at the syscall level for every read and write.

`I — Independent Implementations`::
The redundant paths must be implemented using different logic or libraries to prevent a
single flaw from compromising both. UMRS uses a declarative (`nom`) versus imperative
(`FromStr`) parser to satisfy this.

`N — Non-Bypassable`::
There must be no "backdoor" or alternative route to the data. Using `rustix` for direct
inode-anchored calls prevents path-spoofing that would bypass standard userland checks.

== The Anderson Report Properties (The Foundation)

The reference monitor must meet these three classic design requirements to be considered "trusted":

`Tamperproof`::
The mechanism itself must be protected from modification by untrusted users or processes.
UMRS's use of memory-safe Rust and strictly typed primitives (e.g., bitmasks) ensures
the monitor's internal logic cannot be corrupted.

`Always Invoked`::
It must mediate every reference to every object. (Matches the 'A' in RAIN.)

`Small and Verifiable`::
It must be small enough to be subject to complete analysis and testing. This is why
UMRS prioritizes a minimized TCB and avoids large, untrusted libraries.

== Opening Files: Inode Anchoring and TOCTOU Safety

In high-assurance security, inodes are the ground truth, while paths (filenames) are
just pointers. "Anchoring to the inode" is the difference between a tool that can be
tricked and a tool that acts as a reference monitor.

=== The Vulnerability: Path-Based Logic

If you use a path-based call (like the standard `getxattr("/etc/passwd")`), the
following can happen in the microseconds between your security check and your data read:

1. Your code asks for the label of `/etc/passwd`.
2. An attacker renames `/etc/passwd` to `/etc/passwd.old` and places a symbolic link
   at `/etc/passwd` pointing to a malicious file they own.
3. Your code now reads the security label of the attacker's file, but thinks it is
   looking at the system password file.

This is a TOCTOU (Time-of-Check to Time-of-Use) race condition.

=== The Solution: Inode-Based Anchoring (Provenance)

When you use `std::fs::File::open` followed by `rustix::fs::fgetxattr(&file, ...)`:

*The Handle:* You obtain a file descriptor (fd). This fd is a direct pointer in the
kernel to a specific inode — the actual data blocks on the disk.

*The Invariant:* Once that fd is open, it does not matter if the file is renamed,
deleted, or replaced with a symlink. The fd is still "anchored" to that specific
physical inode.

*Provenance:* When you call `fgetxattr` on that fd, the kernel returns the security
label associated with that specific physical object. The label, the permissions, and
the content all belong to the same physical object — not to a name that could have
been swapped.

=== Why This Satisfies NIST and RTB

[cols="1,3"]
|===
| Requirement | Satisfaction

| `NIST 800-53 AC-3` (Access Enforcement)
| Mediation cannot be bypassed by filesystem manipulation.

| `NSA RTB Non-Bypassability`
| The "Security Decision Point" is moved away from the untrusted filesystem hierarchy
  (names) and into the kernel's object management (inodes).
|===

=== Application to umrs-ls

Because `umrs-ls`:

* Opens the file first.
* Passes that `File` object to `read_context`.
* Uses that same `File` object for `ioctl_getflags` (immutable bit check).

...the tool creates a high-assurance audit. It reports on the physical integrity of the
data, not just what the filename says it is.
