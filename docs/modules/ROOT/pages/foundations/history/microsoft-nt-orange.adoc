= Microsoft NT and the Orange Book
:description: Windows NT's C2 evaluation under TCSEC — what the claim meant, what it did not mean, and what the episode taught the security community.
:toc: left
:toclevels: 2

In the mid-1990s, Microsoft pursued a C2-level evaluation for Windows NT under the Trusted
Computer System Evaluation Criteria (TCSEC). The claim became one of the more instructive
episodes in security evaluation history — and a recurring example of the gap between evaluation
scope and operational reality.

== What Microsoft Actually Claimed

Windows NT 3.5 and later NT 4.0 received TCSEC evaluations targeting the C2 level (Controlled
Access Protection). Microsoft never claimed B1 or B2 MLS certification. The goal was C2 —
the highest discretionary protection level.

C2 requires:

* Discretionary Access Control (DAC)
* Individual user identification and authentication
* Auditing of security-relevant events
* Object reuse protection

C2 does not require mandatory access control, MLS labeling, trusted path, or protection
against malicious code. A system that achieves C2 is not high assurance and is not MLS-capable.

== The Evaluated Configuration Problem

TCSEC evaluations apply to a specific configuration, not to the product in general. For
Windows NT, that configuration typically involved:

* A standalone system or tightly controlled environment
* NTFS as the filesystem (FAT excluded the evaluation scope)
* Specific account policies and auditing settings
* No untrusted services
* Limited or no network services

This is where the well-known observation about NT arose. Network services were either disabled,
heavily constrained, or excluded from the evaluated configuration. As a standalone, carefully
locked-down system, NT could satisfy C2 requirements. As a networked enterprise system — which
is how nearly every NT deployment worked in practice — the evaluated configuration bore little
resemblance to reality.

The quip that "NT was C2 as long as it wasn't connected to a network" captures this gap,
though it is not entirely fair. The more precise statement is that networking was not included
in the evaluated configuration, and enabling it substantially changed the security properties
of the system.

== Why Networking Complicated the Evaluation

Under the TCSEC, networking massively expands the Trusted Computing Base. Every network service
becomes part of the attack surface. Auditing, identification, and access control become harder
to reason about. The Orange Book predated modern network threat models, and many systems —
not just NT — achieved their evaluation with networking disabled or severely restricted.

Windows NT became the most prominent example of this pattern because Microsoft marketed the C2
achievement aggressively while the gap between the evaluated configuration and real-world
deployments was unusually wide.

== Object Reuse and NT

One reason NT's C2 evaluation was configuration-specific is that object reuse guarantees
depended on NTFS. On FAT volumes, the guarantees did not hold. Paging behavior also affected
compliance. Certain subsystems had to be disabled or constrained to maintain the evaluated
posture.

Networking increased the number of objects and buffers dramatically, making object reuse
guarantees harder to reason about formally — which is part of why networking was excluded
from the evaluation scope.

== What the NT Evaluation Taught the Security Community

The NT episode reinforced lessons that MLS designers already understood:

* Evaluation scope is not the same as real-world security posture.
* Configuration matters more than marketing.
* Networked systems require a fundamentally different threat model.
* Paper compliance without operational enforcement is fragile.

These lessons contributed to:

* Later distrust of TCSEC for networked environments.
* The shift to Common Criteria, which uses Protection Profiles to define the intended
  operational environment explicitly.
* Greater emphasis on mandatory enforcement mechanisms such as SELinux.
* Increased scrutiny of "checkbox security" claims.

NT was not unique. Many vendors achieved C2 in lab conditions with locked-down systems that
looked nothing like production deployments. NT became the punchline because the contrast between
the evaluation claim and actual deployment practice was especially visible.

== Relevance to UMRS

UMRS takes the opposite stance. Every capability is evaluated against the operational
environment it will actually be used in — networked, administered by humans, subject to
misconfiguration. The emphasis is on kernel-enforced boundaries that hold regardless of
application correctness, not evaluation configurations that hold only in a lab.

The NT evaluation story is a useful reminder of why enforcement matters more than
certification. A system can be technically compliant with an evaluation standard while
providing no meaningful protection in the environment where it is actually deployed.
That is the failure mode UMRS is designed to avoid.
