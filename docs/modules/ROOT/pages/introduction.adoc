The genesis of the project was to incorporate NARA-registered controlled unclassified (CUI) labeling
to an SELinux system using the multi-category system (MCS). The first phase was to simply apply the
labeling and ensure awareness within the boundaries of the system and resources labeled. How they
should be treated and handled. The second phase will broaden to include enforcement policies.

However, during this process several other areas of high-availability turned out to be very
interesting and worth exploring. Given the person’s background in the classified world and
strict, isolated systems, this seemed a worthy challenge.

The UMRS Project is a reference system to explore and learn: 

* Using SELinux MCS labeling mechanism 
* Operations to Enhance High-Assurance 
* High-assurance software development 
* Use of AI to improve quality of tooling


== What UMRS Demonstrates

Hardening is widely documented. High-assurance engineering is not.

The difference:

* *Hardening* — applying configuration to reduce attack surface.
* *High-assurance engineering* — providing evidence that the system behaves as
  intended, continues to behave as intended, and can prove that fact to third parties.

UMRS is a blueprint for the second kind of work. Every capability — key inventory
tools, audit-log signing and chaining, MLS configuration interfaces, FIPS-state
inspection utilities — is accompanied by explicit mappings to NIST controls, STIG
requirements, and applicable security guidance. The intent is to show how
configuration and code create evidence, not just settings.

Many organizations deploy an operating system as shipped and never explore how much
value careful customization adds. This reference system exists to show that systems
can be adapted and strengthened while maintaining verifiability, accreditation
readiness, and rigorous operational integrity.

== The Problem UMRS Addresses

Standard Linux systems lack first-class support for mandatory label-based
access control on unclassified data. SELinux provides the enforcement
infrastructure, but the userland tooling for constructing, validating,
and auditing MLS security labels is largely string-based and
weakly-typed. Labeling errors — misspelled context fields, out-of-range
sensitivity levels, malformed category sets — often pass silently through
tooling until they surface as enforcement anomalies.

UMRS addresses this gap with a strongly-typed Rust modeling layer over
SELinux constructs. Every security label component is validated at
construction time. Invalid labels cannot be instantiated. The type
system enforces what policy requires.

Beyond the modeling layer, UMRS applies a set of high-assurance coding
patterns — Two-Path Independence parsing, TOCTOU-safe I/O, fail-closed
error handling, provenance verification of kernel attributes — that
make the enforcement logic auditable and resistant to common
implementation-level attacks. These patterns derive from NSA RTB
requirements, NIST 800-218 SSDF, and operational experience in
high-assurance environments.

== CUI Handling

Controlled Unclassified Information (CUI) is a handling framework — not an
MLS label, not a clearance, and not a classification level.

CUI was created to standardize how the U.S. government handles sensitive but
unclassified information that previously carried inconsistent labels: FOUO,
SBU, LES, and others. The authoritative source is Executive Order 13556 (2010)
and the NARA CUI Registry.footnote:[NARA CUI Registry: https://www.archives.gov/cui]

UMRS attaches CUI category labels (for example, `c0`) to objects to provide
granularity below the sensitivity level. This combination — sensitivity level
plus category set — gives auditors a precise, machine-enforced record of what
was handled, when, and under what controls.

[IMPORTANT]
====
CUI applies to information — not to people, systems, or clearance levels.
A process handling CUI data operates at `U-CONTROLLED` with the appropriate
category bits set in its MLS range.
====

For the full label model, see xref:architecture:mls-label-model.adoc[MLS Label Model and CUI Mapping].

== Control Mappings Accompany Every Feature

UMRS does not describe a configuration and leave auditors to map it to
requirements. Each feature includes explicit control citations:

* Inventorying cryptographic signing keys maps to controls around trusted code
  paths, cryptographic integrity, privileged operations, and auditability.
* Audit-log signing chains satisfy tamper-evidence, forensic readiness, and
  incident response controls.
* MLS administrative safeguards tie directly to mandatory access control and
  separation-of-duty requirements.

Every high-assurance capability in UMRS describes which security controls it
helps address and why. The intention is to show how capabilities create
evidence, not just configuration.

== Project Origin

UMRS was started by an engineer with years of operational experience on
classified MLS systems — environments where Bell-LaPadula confidentiality
enforcement, trusted paths, and mandatory access control were not academic
concepts but daily operational requirements. The goal of UMRS is to bring the
same engineering discipline and assurance techniques into unclassified CUI
environments, which increasingly face the same information-flow control
requirements without the same tooling history.

The project is a clean-room implementation. No classified source code,
no proprietary MLS system code, and no line-by-line reimplementation of
existing SELinux userland libraries appears in the codebase.

The project is released under the MIT license. Adaptation and reuse are
explicitly encouraged.

== Audiences

This documentation is written for three audiences, who will use it differently.

*New engineers* will find concept introductions, architecture background, and
developer guides that explain both the what and the why — including why
high-assurance patterns exist and what threat they defeat.

*Security auditors* will find precise control citations (NIST 800-53, CMMC,
NSA RTB), auditable design rationale, and compliance framework documentation
that traces enforcement decisions to their governing standards.

*Potential adopters* will find honest descriptions of what the system does,
what is stable, what is prototype, and how to evaluate whether UMRS fits their
environment. Organizations using UMRS should adapt the MLS label hierarchy to
their own sensitivity categories, review and extend the SELinux policy for their
specific application set, and verify the STIG baseline against their deployment
environment.
