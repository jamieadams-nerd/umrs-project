== What the Boot ID Is

The Linux Boot ID is a 128-bit UUID that uniquely identifies a single boot instance
of the kernel. It is generated once during kernel initialization and remains constant
for the entire lifetime of that boot session.

----
$ cat /proc/sys/kernel/random/boot_id
3c0e9b4c-9c2e-4e2b-9b6e-5b9c61c49c0f
----

Key properties:

* UUID format (128 bits)
* Generated once per boot from the kernel random subsystem (`kernel/random.c`)
* Constant for the entire current boot
* Changes on every reboot, every kexec, and every virtual machine restart
* Not stored on disk — lives in kernel memory only; disappears on reboot
* Exposed as a read-only procfs pseudo-file

[cols="1,2", options="header"]
|===
| Property | Value

| Kernel interface | `/proc/sys/kernel/random/boot_id`
| Filesystem | procfs
| Generated by | Linux kernel random subsystem
| Lifetime | Current boot only
| Mutability | Read-only
| Format | UUID (RFC 4122)
|===

== Boot ID vs. Machine ID

The boot ID is often confused with the machine ID. They serve distinct purposes.

[cols="1,1,2", options="header"]
|===
| Identifier | Lifetime | Purpose

| Machine ID (`/etc/machine-id`)
| Persistent across reboots
| Identifies the physical or virtual machine instance

| Boot ID (`/proc/sys/kernel/random/boot_id`)
| Current boot only
| Identifies the runtime session — one specific boot of the kernel

|===

A journald log entry typically carries both. Together they answer two independent questions:

* *Which machine* produced this event? (machine ID)
* *During which boot session?* (boot ID)

== How journald Uses the Boot ID

`systemd-journald` records the boot ID in the structured journal field `_BOOT_ID` on
every log entry it writes.

----
_BOOT_ID=3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f
----

Note that the hyphenated UUID format used in `/proc/sys/kernel/random/boot_id` is
stored without hyphens in the journal field.

=== Querying by Boot

List all boots in the journal:

----
$ journalctl --list-boots
-2  3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f  Tue 10:00
-1  7a9b8c4a3d2e4c6a9d5b7f1a2c4e5f6d  Tue 10:45
 0  1a2b3c4d5e6f7081920a0b0c0d0e0f10  Tue 11:12
----

The offset `0` is the current boot. Negative offsets count backwards.

Query a specific previous boot:

----
$ journalctl -b -1
----

Query by explicit boot ID:

----
$ journalctl _BOOT_ID=3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f
----

== Security Properties

The boot ID is not a cryptographic value and is not suitable for use as a secret
or authentication token.

[cols="2,1", options="header"]
|===
| Property | Value

| Unique per boot | Yes
| Random (non-deterministic) | Yes
| Cryptographically secure | Not guaranteed
| Suitable as a secret | No
| Suitable as a correlation identifier | Yes

|===

The boot ID is a runtime correlation identifier. Its security value is forensic:
it partitions log records into discrete boot sessions without ambiguity.

== Edge Cases

*Containers.* Containers that share the host `/proc` namespace (default Docker and
most container runtimes) see the *host's* boot ID. The boot ID does not identify
the container — it identifies the host kernel's boot session.

*kexec.* A system rebooted via `kexec` generates a new boot ID, which correctly
identifies the new kernel instance.

*Virtual machines.* A VM reboot produces a new boot ID, the same as physical hardware.

== UMRS: Boot ID as Runtime Event Epoch

In the UMRS logging architecture, the boot ID functions as a *Runtime Event Epoch*:
the natural partitioning boundary between discrete runtime sessions.

The correlation hierarchy for UMRS log analysis is:

----
machine_id
    └── boot_id
            └── events (timestamped, ordered within the session)
----

This hierarchy is directly supported by journald's `_MACHINE_ID` and `_BOOT_ID`
fields without any additional instrumentation. UMRS tools inherit it automatically
by using the system syslog interface — `systemd-journald` attaches both fields to
every UMRS event at ingestion time.

=== Reading the Boot ID from Rust

When UMRS components need the current boot ID (for example, to construct the sealing
key in the xref:patterns:pattern-sec.adoc[Sealed Evidence Cache] pattern), it
is read from the standard procfs path:

[source,rust]
----
// Read the current boot ID for use as a cache sealing input.
// This is a procfs read — provenance verification (fstatfs against PROC_SUPER_MAGIC)
// is required before trusting the value.
let boot_id = std::fs::read_to_string("/proc/sys/kernel/random/boot_id")?
    .trim()
    .to_owned();
----

NOTE: Direct `std::fs::read_to_string` is shown for brevity. In production UMRS code,
reads from `/proc/` use the `ProcfsText` kernel attribute type from `umrs-platform`,
which performs provenance verification (fstatfs magic check) before returning data.
See xref:devel:high-assurance-patterns.adoc[High-Assurance Coding Patterns] (section 1.2, Provenance Verification).

=== Boot ID and the Sealed Evidence Cache

The Sealed Evidence Cache (SEC) pattern uses the boot ID as one input to the ephemeral
sealing key. Because the boot ID changes on every reboot, a sealed cache entry from a
previous boot cannot be replayed in a new boot session — the seal verification will
fail with a different key. This makes the boot ID a natural session boundary for
cache invalidation without requiring persistent key storage.

See xref:patterns:pattern-sec.adoc[Sealed Evidence Cache (SEC)] for the full implementation.

=== Compliance Note

Recording the boot ID in UMRS audit log events satisfies the "when" and "session context"
components of NIST SP 800-53 AU-3 (Audit Record Content), which requires that audit records
contain the information needed to establish what type of event occurred, when the event
occurred, and the identity of the system component that generated the event.
