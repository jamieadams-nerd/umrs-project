= Log Filesystem and Journald Tuning
:toc: left
:description: How to optimize /var/log mount options, configure journald for high-volume logging, and understand why /var/log/audit must be treated differently.

== Goal

Optimize a dedicated `/var/log` filesystem for logging performance and safety. Assumes
a RHEL-style system with XFS or ext4, `journald` in use, and `sudo` access only.

NOTE: These tuning steps affect `/var/log` only. `/var/log/audit` must be treated
differently — see xref:#audit-differences[Audit Log Differences].

== Step 1: Confirm /var/log Is a Dedicated Filesystem

. Confirm `/var/log` is mounted independently:
+
[source,bash]
----
sudo df -h /var/log
----
+
Expected output:
+
[source]
----
Filesystem                      Size  Used Avail Use% Mounted on
/dev/mapper/vg_logs-lv_varlog   20G   4.1G  16G  21% /var/log
----

. Confirm filesystem type and logical volume:
+
[source,bash]
----
sudo lsblk -f
----
+
Note the filesystem type (`xfs` or `ext4`) and logical volume name for the entry
where `MOUNTPOINT` is `/var/log`.

== Step 2: Trace /var/log Through LVM

This step is awareness, not tuning. It confirms that changes to `/var/log` mount
options affect only that logical volume and not other volumes on the same disk.

. View the logical volume:
+
[source,bash]
----
sudo lvdisplay /dev/mapper/vg_logs-lv_varlog
----

. See which physical volumes back the volume group:
+
[source,bash]
----
sudo pvs
sudo vgs
----

. Confirm the mapping:
+
[source,bash]
----
sudo lsblk -o NAME,TYPE,SIZE,MOUNTPOINT,PKNAME
----

== Step 3: Back Up fstab

Before modifying mount options:

[source,bash]
----
sudo cp /etc/fstab /etc/fstab.backup
----

CAUTION: This step is irreversible if you overwrite `fstab` and the backup is lost.
Keep the backup in a safe location.

== Step 4: Optimize /var/log Mount Options

Goal: reduce unnecessary metadata writes, preserve audit integrity, remain STIG-friendly.

. Edit fstab:
+
[source,bash]
----
sudo vi /etc/fstab
----
+
Locate the `/var/log` line. Example original entry:
+
[source]
----
/dev/mapper/vg_logs-lv_varlog   /var/log   xfs   defaults   0 0
----

. For *XFS*, use:
+
[source]
----
noatime,nodiratime,nodev,nosuid,noexec,logbufs=8
----
+
Example updated line:
+
[source]
----
/dev/mapper/vg_logs-lv_varlog   /var/log   xfs   noatime,nodiratime,nodev,nosuid,noexec,logbufs=8   0 0
----
+
Option descriptions:
+
* `noatime,nodiratime` — removes access-time updates, reducing write traffic.
* `nodev,nosuid,noexec` — enforce log-only behavior and STIG expectations.
* `logbufs=8` — increases XFS metadata log buffering under concurrent writes.

. For *ext4*, use:
+
[source]
----
noatime,nodiratime,nodev,nosuid,noexec,data=ordered
----

. Apply the changes:
+
[source,bash]
----
sudo mount -o remount /var/log
----

. Verify active options:
+
[source,bash]
----
sudo mount | grep ' /var/log '
----

== Step 5: Configure journald

`journald` is a major writer into `/var/log`.

. Ensure persistent journal storage exists:
+
[source,bash]
----
sudo mkdir -p /var/log/journal
sudo chown root:systemd-journal /var/log/journal
sudo chmod 2755 /var/log/journal
----

. Configure journald:
+
[source,bash]
----
sudo vi /etc/systemd/journald.conf
----
+
In the `[Journal]` section, set at minimum:
+
[source]
----
Storage=persistent
SystemMaxUse=5G
SystemKeepFree=1G
----
+
Adjust capacity values for your disk size.

. Optional rate limiting (increase if log drops occur during bursts):
+
[source]
----
RateLimitIntervalSec=30s
RateLimitBurst=10000
----

. Restart journald:
+
[source,bash]
----
sudo systemctl restart systemd-journald
----

== Step 6: Verify Logging Health

. Generate a test log message:
+
[source,bash]
----
sudo logger "var log tuning verification message"
----

. Confirm journald writes:
+
[source,bash]
----
sudo journalctl -n 10
----
+
The test message should appear in the last 10 entries.

. Confirm files in `/var/log`:
+
[source,bash]
----
ls -l /var/log
----

[[audit-differences]]
== Audit Log Differences: /var/log/audit

`/var/log/audit` must be treated differently from `/var/log`. The same performance
tuning options that are acceptable for general logs are not appropriate for audit logs.

=== Why They Differ

`/var/log` (general logging)::
Tuned for throughput and burst tolerance. `journald` and `rsyslog` are designed to
buffer aggressively. Dropping logs under extreme stress is a documented behavior.

`/var/log/audit` (auditd)::
Tuned for integrity, ordering, and guaranteed write behavior. `auditd` writes
synchronously by design. System behavior may degrade or halt if logs cannot be written.
This is intentional.

=== Mount Options for /var/log/audit

Use conservative defaults. Do *not* apply `noatime` or increase write buffering:

[source]
----
nodev,nosuid,noexec,relatime,seclabel
----

=== auditd Space Management

`auditd` has its own space threshold controls that serve as policy levers, not
performance knobs:

[source]
----
space_left         = <warning threshold>
admin_space_left   = <critical threshold>
disk_full_action   = SUSPEND or HALT
disk_error_action  = SYSLOG
----

Configure these values to match your site policy. If `auditd` performance is a
concern, the correct approach is reducing audit rule scope and eliminating
duplicate rules — not changing disk I/O settings.

Red Hat's guidance can be summarized as: "General logging is a throughput problem.
Audit logging is a correctness problem."

== NIST and STIG Compliance Summary

[cols="1,3"]
|===
| Control | Satisfied by

| `AU-4` (Audit Storage Capacity)
| Dedicated `/var/log` filesystem; `journald` size limits; `logrotate` policies.

| `AU-5` (Response to Audit Processing Failures)
| Rate limiting, size caps, and `auditd` space actions prevent failures from
  overwhelming the system.

| `AU-9` (Protection of Audit Information)
| `nodev,nosuid,noexec` on both `/var/log` and `/var/log/audit`; integrity options
  preserved.

| `AU-11` (Audit Record Retention)
| Persistent storage and size management are prerequisites for any defined retention period.

| `SC-32` (System Partitioning)
| Dedicated logical volumes for `/var/log` and `/var/log/audit`.
|===

Applicable STIG IDs: `RHEL-09-231025` (separate `/var/log`), `RHEL-09-231030` (separate
`/var/log/audit`). The RHEL 10 STIG carries forward the same requirements.
