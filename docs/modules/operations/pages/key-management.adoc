= Cryptographic Key Management
:toc: left
:description: Key lifecycle management practices, naming conventions, storage layout, and compliance guidance.

In the context of information security and compliance frameworks like NIST Special
Publication 800-53, *cryptographic key management* refers to the comprehensive set of
practices, processes, and systems that oversee the full lifecycle of cryptographic keys.
This lifecycle spans all stages from key generation, distribution, storage, and rotation,
to archival and eventual destruction (cryptoperiod management).

Effective key management ensures that cryptographic mechanisms — which protect the
confidentiality, integrity, and authenticity of sensitive data — remain robust and
trustworthy throughout their operational lives. It is the foundational security control
that underpins all other uses of encryption within an organization.

The importance of robust key management cannot be overstated, as the security of
encrypted data is entirely dependent on the security of the keys used to protect it.
A weak or compromised key management framework can render even the strongest encryption
algorithms useless. Poorly managed keys expose an organization to significant risks,
including data breaches, loss of data integrity, and compliance failures with regulatory
standards such as FISMA, HIPAA, or GDPR.

By implementing strong controls outlined in guidance like NIST 800-53 controls (e.g.,
CM-5, SC-12, SC-13), organizations establish a trusted system for safeguarding these
critical assets, thereby maintaining secure operations and building verifiable trust in
their information systems.

Many people get tripped up in audits — not the crypto itself, but how well the key
lifecycle is documented and justified. This document addresses three layers:

* What standards actually require (authoritative controls).
* What auditors typically expect to see.
* A practical, accreditation-safe documentation checklist you can actually use.

== Key Classes and Naming Conventions

=== What Keys Are Needed

For a high-assurance RHEL system, the following signing keys are needed:

* System audit logs (auditd output files).
* System and application logs (rsyslog, journald, and application logs under `/var/log`).
* Rotated log archives (e.g., compressed tarballs from logrotate).
* Configuration baselines (e.g., files under `/etc`).
* Software or script releases shipped internally.
* Backups (optional but recommended).

=== Naming Convention

Good key names express scope and purpose, algorithm family, and rotation version or date.

Pattern:
----
<scope>_<use>_<algo-size>_<vNN>.key    (private key)
<scope>_<use>_<algo-size>_<vNN>.pub    (public key)
<scope>_<use>_<algo-size>_<vNN>.pem    (PEM-bundled public material)
----

Where:

* `scope` = host or environment domain, e.g., `host1`, `cdslogger`, `ha`
* `use` = `audit`, `syslog`, `log`, `archive`, `config`, `release`, `backup`
* `algo/size` = `rsa-3072`, `rsa-4096`, `ecdsa-p256`
* `vNN` = version, e.g., `v01`, `v02`

.Example key names for a host named `ha-logger01`
----
ha-logger01_audit_sign_rsa-3072_v01.key
ha-logger01_audit_sign_rsa-3072_v01.pub

ha-logger01_log_sign_rsa-3072_v01.key
ha-logger01_log_sign_rsa-3072_v01.pub

ha-logger01_archive_sign_rsa-3072_v01.key
ha-logger01_archive_sign_rsa-3072_v01.pub

ha-logger01_config_sign_rsa-3072_v01.key
ha-logger01_config_sign_rsa-3072_v01.pub
----

If you prefer dates instead of version numbers:
----
ha-logger01_audit_sign_rsa-3072_2025-12-08.key
ha-logger01_audit_sign_rsa-3072_2025-12-08.pub
----

=== What Each Key Signs

`Audit signing key`::
Signs files produced by auditd: `/var/log/audit/audit.log`, `/var/log/audit/audit.log.*`,
and any exported audit archives.

`Logging (non-audit) signing key`::
Signs rotated log files for `/var/log/messages`, `/var/log/secure`, `/var/log/maillog`,
application logs under `/var/log/<app>/`. Keep audit and non-audit logs under separate keys.

`Archive signing key`::
Signs higher-level archives: tarballs created from log snapshots
(`/var/archives/logs/*.tar.gz`), forensic bundles.

`Configuration baseline signing key`::
Signs configuration snapshots and baselines: checksummed config manifests
(`/var/lib/config-baseline/etc-baseline.json`), tarballs of `/etc` state.

`Software and script release key`::
Signs internal tools, scripts, and packages shipped internally.

`Backup signing key (optional)`::
Signs backup manifests or backup archives before off-system transfer.

== Storage Layout and Permissions

=== Directory Structure

Use `/etc/pki/ha-keys/` as the central root, organized by key family:

----
/etc/pki/ha-keys/
    audit/
        private/
        public/
        archive/
    logging/
        private/
        public/
        archive/
    config/
        private/
        public/
        archive/
    release/
        private/
        public/
        archive/
----

.Example paths for the audit signing key pair
----
/etc/pki/ha-keys/audit/private/ha-logger01_audit_sign_rsa-3072_v01.key
/etc/pki/ha-keys/audit/public/ha-logger01_audit_sign_rsa-3072_v01.pub
----

=== Permissions and Ownership

For *private keys*:

* Directory: `0700`, owned by a dedicated service account (e.g., `logsignd:logsignd`).
* File: `0600`, same owner.

[source,bash]
----
chown -R logsignd:logsignd /etc/pki/ha-keys
chmod 700 /etc/pki/ha-keys
chmod 600 /etc/pki/ha-keys/<family>/private/*.key
----

For *public keys*:

* Directory: `0755` (or `0750` for restricted environments).
* File: `0644`.

=== Passphrases

For automated, non-interactive signing (log rotation, journald export, cron jobs),
private keys are typically stored without passphrases. Compensating controls are:

* Strict file permissions (`0700`/`0600`).
* A dedicated restricted service user account.
* Optionally, hardware binding via TPM or HSM.

In high-assurance environments, the ideal is to use a hardware module (HSM or TPM)
such that the private key never leaves hardware. On commodity RHEL without HSMs,
unencrypted private key files with tight permissions and good process isolation
is the standard approach.

=== No System Auto-Creation

There is no general RHEL startup service that auto-generates the kind of specialized
signing keys described here. auditd, rsyslog, journald, and logrotate do not create
signing keys automatically. Your signing utility is responsible for generating and
managing all key pairs.

== Key Lifecycle Management

=== Metadata File per Key Family

Maintain a metadata file at `<family>/audit_signing_keys.json`:

[source,json]
----
[
  {
    "key_name": "ha-logger01_audit_sign_rsa-3072_v01.key",
    "public_name": "ha-logger01_audit_sign_rsa-3072_v01.pub",
    "created": "2025-12-08T00:00:00Z",
    "activated": "2025-12-08T00:05:00Z",
    "deactivated": null,
    "retired": null,
    "status": "active",
    "algorithm": "RSA-3072",
    "scope": "audit"
  }
]
----

=== Rotation Procedure

When rotating a key:

1. Mark the current key in the JSON metadata as `"retired"` with a `"deactivated"` timestamp.
2. Move the current private key to `<family>/archive/private/`.
3. Move the current public key to `<family>/archive/public/`.
4. Generate a new `vNN+1` key pair in the main `private/` and `public/` directories.
5. Update the JSON metadata to mark the new key as `"active"`.
6. Rebuild the per-family and global public key bundles.

=== Public Key Export

Maintain both per-family bundles and a global bundle:

----
/etc/pki/ha-keys/audit/public/audit_sign_pubkeys.pem
/etc/pki/ha-keys/all_public_keys.pem
----

These bundles contain all current active public keys plus archived public keys that
may still be needed to verify older logs and archives.

To export for transport to a verification node, SIEM, or offline forensic workstation:

[source,bash]
----
ha_signing_ctl export-public --dest /var/tmp/ha-signing-pub-bundle
----

=== Never Delete Old Public Keys

Do not delete old public keys until:

* All data signed with them has reached the end of its retention period.
* A documented key-destruction procedure authorizes destruction.

Verification tools must be able to find the correct public key by fingerprint
from the signature metadata and archive bundles.

== Cryptographic Defaults

For a RHEL 10 FIPS system, default signing key parameters are:

* Algorithm: RSA
* Key length: 3072 bits minimum

NIST SP 800-131A Rev.2 and SP 800-57 Part 1 treat RSA-2048 as providing only
112-bit security. RSA-3072 provides approximately 128-bit security and is the
forward-looking default. NSA and DoD guidance is moving toward RSA-3072 as a
minimum, with RSA-4096 preferred for new keys in high-sensitivity environments.

Allowed algorithm and key-size combinations:

* RSA: 2048, 3072, 4096
* ECDSA: P-256, P-384 (FIPS-approved curves)

== Key Inventory Reporting

The signing utility produces a key inventory report containing for each key:

* Key role (e.g., `audit_sign`, `log_sign`).
* Key name and path.
* Algorithm and size.
* Creation, activation, deactivation, and retirement dates.
* Intended lifetime (from policy, e.g., 1 year).
* Current status: active, retired, compromised, or destroyed.
* Associated artifacts (paths the key is used to sign).

This inventory is:

* A direct input to compliance documentation.
* A control artifact for auditors and internal key-management review.
* Tied to NIST SP 800-57-style requirements for documented key lifetimes, rotations,
  and cryptoperiods.

== Compliance Alignment

[cols="1,3"]
|===
| Control | Applicability

| `SC-12` | Cryptographic key establishment and management — lifecycle, rotation, archival
| `SC-13` | Cryptographic protection — FIPS-approved algorithms and key sizes
| `AU-9` | Protection of audit information — signing audit logs
| `AU-10` | Non-repudiation — signed artifacts link to authenticated key holders
| `SI-7` | Software and information integrity — signing and verification of artifacts
| `CM-6` | Configuration settings — key management configuration controlled and reviewed
|===
