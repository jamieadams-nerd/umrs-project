= Glossary
:description: Terminology definitions used across UMRS documentation.
:toc: left
:toclevels: 2

This glossary defines security, assurance, cryptographic, MLS, and platform terms
used throughout UMRS documentation.

Terms that appear in multiple modules have a single authoritative definition here.
Where a term maps to a NIST control, standard, or UMRS implementation, that mapping is noted.

NOTE: This glossary is the reader-facing definition of UMRS terms.
The project also maintains an internal approved terminology list at `.claude/approved_terminology.md`
(preferred English forms and do-not-abbreviate rules) and a translator vocabulary at
`.claude/agent-memory/umrs-translator/vocabulary.md` (English-to-French mappings).
All three lists must remain consistent.

== Assurance and Integrity

=== Assurance

Confidence justified by evidence, not by belief.

In high-assurance engineering, assurance is earned through structured evidence: formal proofs,
auditing, controlled processes, and independent evaluation.
Assurance is bounded — it covers specific properties of specific components under specific
conditions, and it degrades if controls are bypassed or become unverifiable.

See also: NIST 800-53 SA-4; xref:ROOT:what-is-high-assurance.adoc[What Is High Assurance?]

=== Attestation

A cryptographic or authoritative assertion by an entity that a specific claim is true
at a specific time.

Examples:

* "This file was received intact at time T."
* "This system was in configuration state Y when measured."
* "This data was produced by process Z."

Attestation answers: "I assert this is true, and I am accountable for that assertion."

Attestation is distinct from provenance.
Provenance describes history.
Attestation asserts a specific fact about a specific artifact at a specific moment.

In UMRS, the IMA measurement log is an implicit attestation: it records what was seen,
when, and in what state, in a tamper-visible ledger.

See also: NIST 800-53 AU-10 (Non-repudiation).

=== Auditability

The ability to reconstruct events and decisions after the fact with sufficient fidelity
to support accountability.

Auditability requires complete records, meaningful semantics (who, what, when, why),
stable identifiers, and integrity protection of the audit records themselves.
Audit logs that cannot answer "who, what, when, why" are compliance theater, not auditability.

See also: NIST 800-53 AU-3 (Content of audit records).

=== Authenticity

The assurance that an artifact is genuinely what it claims to be — that it comes from
the identity it purports to come from and has not been substituted.

Authenticity answers: "Is this really from who or what it claims?"

Authenticity requires identity binding, provenance or attestation, and integrity.
Courts frequently reject digital evidence for failure to authenticate, even when integrity
is present — the distinction matters in forensic and legal contexts.

=== Chain of Custody

The documented and continuous control history of an artifact from creation or acquisition
to final disposition.

Chain of custody answers: "Who had control of this, when, and under what authority?"

In digital systems, chain of custody requires:

* Integrity protection of the artifact itself
* Integrity protection of the custody records
* Explicit transfer events — no silent handoffs

A gap in the chain is a break in trust.
Most unclassified systems fail here: logs exist, but custody records are not themselves
cryptographically protected.
IMA/EVM prevents this by making the measurement log tamper-visible.

=== Custody Trust vs. Content Trust

Two distinct properties that must not be conflated.

*Custody trust* — "This artifact has not changed since we received it."

*Content trust* — "This artifact is accurate, valid, or correct."

A system can trust custody while explicitly withholding content trust.
UMRS claims custody trust through IMA/EVM enforcement.
It does not automatically claim content trust — a CUI document whose IMA hash verifies
is demonstrably unaltered, but whether its content is accurate is a separate determination.

Conflating these two properties is a common cause of audit failures.

See also: xref:security-concepts:integrity-and-provenance.adoc[Integrity and Provenance in UMRS].

=== Integrity

The assurance that data has not been altered in an unauthorized or undetected manner.

Integrity answers: "Has this changed since it was last trusted?"

Integrity does not mean the data is correct, appropriate, or authorized for use.
It means the bits are the same as when last trusted, and any change is detectable.

Enforced in UMRS via:

* *IMA (Integrity Measurement Architecture)* — file content hashes, kernel measurement log
* *EVM (Extended Verification Module)* — HMAC over security-relevant xattrs

See also: NIST 800-53 SI-7 (Software and information integrity).

=== Integrity Assurance

The degree of confidence that integrity protections are correctly implemented, enforced,
and verifiable.

The difference between "we hash files" and integrity assurance:

* The hash must be meaningful (correct algorithm, correct scope)
* The algorithm must be FIPS-validated
* Enforcement must be at the kernel level — not in a daemon that can be stopped
* The measurement log must be integrity-protected (append-only, signed)
* The mechanism must resist bypass

Integrity assurance is a system property, not a property of a single mechanism.

=== Ground Truth

The authoritative, kernel-verified source of a security-relevant value.

Ground truth contrasts with derived or reported values that could be forged or manipulated
by userspace.
In UMRS, kernel pseudo-filesystems (`/proc/`, `/sys/fs/selinux/`) are considered ground truth
for their respective subsystems, but only after provenance verification confirms the source
is the kernel (not a bind mount or overlay that a privileged user constructed).

The term signals that the value is the definitive one — the system state as the kernel
sees it, not as any application or configuration file reports it.

See also: xref:#_provenance[Provenance]; xref:patterns:pattern-provenance.adoc[Provenance Verification pattern].

=== Non-repudiation

The ability to prevent an entity from credibly denying a prior action.

Non-repudiation is a legal and procedural property, enabled by cryptography but not
guaranteed by it alone.
Requirements: digital signatures with key custody controls, trusted timestamps, and
audit records that cannot be silently altered.

UMRS satisfies this through signed audit logs and IMA/EVM measurements that tie actions
to kernel-verified identities.

See also: NIST 800-53 AU-10.

=== Trusted Computing Base (TCB)

The set of hardware, firmware, and software components that must be trusted for a system's
security policy to be enforced correctly.

The TCB is the boundary of trust.
If any component inside the TCB is compromised, the security guarantees of the system fail.
If a component outside the TCB is compromised, the security guarantees still hold.

Minimizing the TCB is a core high-assurance engineering principle.
A smaller TCB means fewer components an auditor must evaluate, fewer places a flaw can hide,
and a smaller attack surface for a targeted compromise.

In UMRS, the TCB includes:

* The hardware (CPU, TPM if present)
* The bootloader (GRUB with measured boot)
* The Linux kernel (with SELinux, IMA, EVM, dm-crypt)
* The SELinux policy

Application code, utilities, and services are outside the TCB — the TCB enforces policy
on them without trusting them.

See also: NIST SP 800-53 SA-3; xref:ROOT:what-is-high-assurance.adoc[What Is High Assurance?].

=== Provenance

The documented history of an artifact: where it originated, how it was produced, and
what transformations have been applied.

Provenance answers: "Where did this come from, and how did it get here?"

Provenance alone does not guarantee correctness or legitimacy.
It establishes traceable history.
In high-assurance systems, provenance must be tamper-evident, attributable to a specific
identity (human, system, or device), and preserved across transformations.

In UMRS, provenance verification applies before trusting data from kernel pseudo-filesystems
(`/proc/`, `/sys/fs/selinux/`): the filesystem magic number is checked via `statfs`
to confirm the source is the kernel, not a bind-mount or overlay (NIST 800-53 SI-7).

See also: xref:patterns:pattern-provenance.adoc[Provenance Verification pattern].

== SELinux and MLS

=== Access Vector Cache (AVC)

The in-kernel cache that stores the results of recent SELinux policy decisions.

When a subject requests access to an object, the kernel checks the AVC first.
If the decision is cached, it is returned immediately without consulting the full policy.
On a cache miss, the security server evaluates the policy and caches the result.

The AVC is a performance mechanism — it does not change the enforcement model.
Cached decisions are still policy-correct.
When policy is reloaded, the AVC is flushed so new decisions are re-evaluated.

=== Audit Event

A discrete, timestamped record of a security-relevant action or decision.

Audit events are generated at the kernel level (SELinux AVC denials, IMA measurements)
and at the application level (structured log output).
Each event must capture who acted, on what, when, and with what outcome — the minimum
content specified by NIST 800-53 AU-3 (Content of audit records).

An audit event is not the same as a log line.
A log line is a serialization format.
An audit event is the structured, queryable record of a specific decision.
UMRS represents audit events as typed enum variants so callers can match, filter, and
count them programmatically.

See also: xref:security-concepts:integrity-and-provenance.adoc[Integrity and Provenance in UMRS].

=== Category Set

A bitmask that represents the set of compartments or categories assigned to a security context.

// TODO: Expand this entry when SELinux reference pages are more thoroughly documented.
// The lattice dominance model and bit-ordering conventions will be covered in detail there.
In UMRS, `CategorySet` is a 1024-bit `[u64; 16]` fixed-size stack-allocated array.
Category membership determines whether a process's category set _dominates_ an object's
category set — a required condition for MLS access decisions alongside the sensitivity level.

See also: xref:reference:selinux/context.adoc[Security context reference].

=== Controlled Unclassified Information (CUI)

A handling framework for sensitive but unclassified information — not a classification
level, clearance, or MLS label.

CUI was established by Executive Order 13556 (2010) to standardize handling of information
previously labeled inconsistently (FOUO, SBU, LES, and others).
The authoritative registry is maintained by NARA.footnote:[https://www.archives.gov/cui]

In UMRS, CUI objects are labeled at the `U-CONTROLLED` sensitivity level with
appropriate category bits.

See also: xref:architecture:mls-label-model.adoc[MLS Label Model and CUI Mapping].

=== Discretionary Access Control (DAC)

An access control model in which the owner of a resource decides who may access it.

Under DAC, file owners can grant or revoke permissions for other users.
POSIX permissions (read, write, execute bits; `chmod`, `chown`) and POSIX ACLs are
the DAC mechanisms on Linux.

DAC is discretionary because the owner has discretion — and can make mistakes.
A misconfigured `chmod 777` exposes a file to everyone on the system.
No kernel mechanism prevents the owner from making that error.

In UMRS, DAC (POSIX permissions) and MAC (SELinux) are both active.
MAC decisions run after DAC decisions.
If either denies access, the access is rejected.
MAC cannot be overridden by changing file permissions.

Contrast with xref:#_mandatory_access_control_mac[Mandatory Access Control].

=== Extended Verification Module (EVM)

A Linux kernel integrity subsystem that protects the security-relevant extended attributes
(`xattrs`) of files using an HMAC.

IMA measures file content.
EVM protects the metadata that controls how IMA interprets that content — specifically,
the IMA hash stored in `security.ima` and SELinux labels stored in `security.selinux`.
Without EVM, an attacker who can write xattrs could change a file's IMA hash or its
SELinux security context without triggering detection.

EVM computes an HMAC over the protected xattrs using a kernel-held key.
Any unauthorized modification to the protected xattrs causes EVM verification to fail.

In UMRS, EVM is used alongside IMA to provide layered integrity enforcement.

See also: xref:#_integrity_measurement_architecture_ima[Integrity Measurement Architecture]; NIST SP 800-53 SI-7.

=== Integrity Measurement Architecture (IMA)

A Linux kernel integrity subsystem that measures file content and records the measurements
in a signed, append-only log.

When a file is accessed, IMA computes a hash of its content and compares it against a
reference value (stored in the `security.ima` xattr or a reference manifest).
If the hash does not match, IMA can deny access (in `enforce` mode) or log the mismatch
(in `fix` or `log` mode).

The IMA measurement log records every distinct file access with its hash value.
The log is append-only and protected by the TPM (when a TPM is present).
This log is the primary tamper-evident record of what the system has executed.

In UMRS, IMA is active in enforce mode.
Files that do not carry a valid IMA signature or have not been measured are denied
execution and access, depending on the IMA policy.

See also: xref:#_extended_verification_module_evm[Extended Verification Module];
NIST SP 800-53 SI-7 (Software and information integrity).

=== Linux Security Module (LSM)

A Linux kernel framework that allows security subsystems to hook into kernel operations
and enforce access control policy.

LSM hooks are inserted at security-relevant kernel call sites — file opens, network
connections, process operations, IPC — and give registered security modules the ability
to allow or deny the operation.

SELinux is the primary LSM used in UMRS.
IMA and EVM also use the LSM hook infrastructure.
Multiple LSMs can be stacked (RHEL 10 supports LSM stacking by default).

The LSM framework is the architectural mechanism that makes the reference monitor
property possible in Linux: all access decisions pass through LSM hooks,
so a registered security module cannot be bypassed by user-space code.

=== Mandatory Access Control (MAC)

An access control model in which policy decisions are made by the system, not by the
data owner.

Users and applications cannot grant access that policy does not permit.
SELinux implements MAC in the Linux kernel via the Linux Security Module (LSM) framework.

Contrast with discretionary access control (DAC), where owners can grant access to anyone.
In UMRS, both DAC (POSIX permissions) and MAC (SELinux) are active.
MAC decisions override DAC decisions.

=== MLS Range

The pair of MLS levels (low and high) that defines the range within which a subject may
operate or an object may be accessed.

Format: `sensitivity_low:categories_low-sensitivity_high:categories_high`.

Example: `s0:c0-s3:c0,c2` means the subject operates at or between those two levels.

=== Multi-Level Security (MLS)

An access control model that enforces information flow restrictions based on a hierarchical
sensitivity label structure.

MLS systems assign every subject (process) and object (file, socket, IPC object) a
sensitivity level and a category set.
Access decisions enforce two core properties:

* *No-read-up* — a subject may not read objects at a higher sensitivity level.
* *No-write-down* — a subject may not write to objects at a lower sensitivity level.

These rules derive from the Bell–LaPadula model for confidentiality.
The Biba model applies analogous rules for integrity (no-read-down for integrity,
no-write-up for integrity).

In UMRS, SELinux MLS policy enforces the sensitivity label hierarchy.
The hierarchy runs from `GENERAL` (publicly shareable, unclassified) through
`U-CONTROLLED` (highest unclassified sensitivity requiring explicit handling controls).

See also: xref:architecture:mls-label-model.adoc[MLS Label Model and CUI Mapping];
xref:#_sensitivity_level[Sensitivity Level].

=== Policy Enforcement

The act of the reference monitor evaluating a subject's request against the active
security policy and producing a permit or deny decision.

Policy enforcement is non-discretionary: neither the subject nor the data owner
can override the decision.
Enforcement happens in the kernel; it is not a userland check that can be skipped.

The term distinguishes active, kernel-mediated enforcement from passive policy statements,
logging, or advisory warnings.
A system that logs policy violations but does not block them is not enforcing policy.

See also: xref:security-concepts:reference-monitor.adoc[Reference Monitor Design in UMRS];
NIST 800-53 AC-3 (Access Enforcement).

=== Reference Monitor

A security mechanism that mediates every access between subjects and objects, ensuring
the access complies with the active security policy.

The reference monitor must be non-bypassable, always invoked, and tamper-resistant.
These three properties are required — any exception breaks the security guarantee.

In UMRS, SELinux is the kernel-level reference monitor.
Every file open, socket connection, and IPC operation passes through the SELinux access
vector cache (AVC) before being permitted or denied.

See also: xref:security-concepts:reference-monitor.adoc[Reference Monitor Design in UMRS].

=== Role-Based Access Control (RBAC)

An access control model that grants permissions based on the role a user or process holds,
not on the identity of that user or process directly.

Roles group permissions.
Subjects are assigned to roles.
A subject's effective permissions are determined by its current role.

In SELinux, roles constrain which types a user can transition to.
A user in the `sysadm_r` role can run processes of types allowed for system administrators.
A user in the `user_r` role is restricted to types appropriate for unprivileged sessions.

RBAC in SELinux is a middle layer between user identity and type enforcement (TE).
The full stack is: user → role → type → sensitivity level.

=== Security Context

The full, structured SELinux label assigned to a subject or object.
A security context has five components:

```
user : role : type : sensitivity_level : category_set
```

Every process and file on a SELinux system has a security context.
Access decisions compare the subject's security context against the object's security context
using the active policy.

Example: `system_u:system_r:httpd_t:s0:c0,c1`

In UMRS, each component is modeled as a distinct validated Rust type.
A `SecurityContext` cannot be constructed with a malformed sensitivity level, an
out-of-range category, or a missing required field.

NOTE: "Security label" is a colloquial shorthand that people use to mean the full security
context.
In UMRS documentation and code, always use _security context_ for precision.

See also: xref:reference:selinux/context.adoc[Security context field reference].

=== Security Label

_Colloquial synonym for security context._ Not a preferred term in UMRS documentation.

"Security label" appears widely in informal writing and vendor documentation as a shorthand
for the full SELinux security context.
In UMRS documentation, use _security context_ — the precise term that names all five components.

=== Sensitivity Level

The hierarchical component of a security context (for example, `s0` through `s15`).

NOTE: "Sensitivity label" is a colloquial, non-preferred variant of this term.
Use _sensitivity level_ consistently.

Bell–LaPadula rules govern read and write access by sensitivity level:

* *No-read-up*: A subject at `s4` cannot read objects labeled `s5` or higher.
* *No-write-down*: A subject at `s4` cannot write to objects labeled `s3` or lower.

These rules prevent both unauthorized disclosure (upward reads) and covert downgrade
(downward writes).

== Cryptography

=== Asymmetric Cryptography

Cryptography that uses a mathematically related key pair: a public key (freely distributed)
and a private key (kept secret).

Also called _public-key cryptography_.
Used for key agreement (ECDH), digital signatures (ECDSA, ML-DSA), and key encapsulation (ML-KEM).
Asymmetric algorithms are the ones vulnerable to Shor's algorithm — this is why post-quantum
cryptography exists.

Contrast with symmetric cryptography, which uses a single shared key.

=== AES (Advanced Encryption Standard)

A symmetric block cipher standardized by NIST (FIPS 197).
Used in UMRS for data confidentiality: dm-crypt full-disk encryption, IMA HMAC keying.
Preferred key size: 256 bits (AES-256).

=== Authenticated Encryption

Encryption that provides both confidentiality and integrity protection in a single operation.

Example: AES-GCM (Galois/Counter Mode) computes a ciphertext and an authentication tag.
Any tampering with the ciphertext causes tag verification to fail.

Preferred over confidentiality-only encryption modes in new designs.

=== Cipher Mode

The operational method used to apply a block cipher to a data stream.

[cols="1,1,2", options="header"]
|===
| Mode | Property | Notes

| GCM | Authenticated encryption | Preferred for network and file encryption
| XTS | Disk encryption | Standard for full-disk encryption (dm-crypt)
| CBC | Confidentiality only | Legacy; no integrity protection
| ECB | None | Disallowed — reveals block patterns
|===

=== Crypto Policy (RHEL System-Wide Cryptographic Policy)

A RHEL mechanism that enforces a consistent set of cryptographic algorithm constraints
across all system libraries and services from a single configuration point.

The active policy is set with `update-crypto-policies --set <POLICY>` and affects
OpenSSL, GnuTLS, NSS, Kerberos, OpenSSH, and other system libraries.
The policy file is stored in `/etc/crypto-policies/`.

Standard policies in RHEL 10:

[cols="1,3", options="header"]
|===
| Policy | Description

| `DEFAULT`
| Secure, broadly compatible. Includes PQC support on RHEL 10.1+.

| `FIPS`
| Restricts to FIPS 140-3 validated algorithms. Disables PQC. Required for CUI handling
  under NIST 800-171 and CMMC.

| `FUTURE`
| Stricter algorithm requirements intended for future deployments.

| `LEGACY`
| Enables older algorithms for compatibility. Not appropriate for high-assurance deployments.
|===

On RHEL 10.1+, the `DEFAULT` policy includes PQC support (ML-KEM, ML-DSA) through
OpenSSL, GnuTLS, and Go.
Enabling `FIPS` mode removes PQC availability because NIST has not yet completed FIPS 140-3
validation for post-quantum modules.

UMRS deployments handling CUI must use the `FIPS` policy.
See xref:cryptography:crypto-post-quantum.adoc#_rhel_10_pqc_availability[RHEL 10 PQC Availability]
for the current FIPS/PQC compatibility status.

See also: NIST 800-53 SC-13 (Cryptographic Protection); NIST 800-53 CM-6 (Configuration Settings).

=== CRQC (Cryptographically-Relevant Quantum Computer)

A quantum computer powerful enough to break currently deployed public-key cryptography
using Shor's algorithm.

No CRQC exists today.
Expert consensus places the likely emergence at 5 to 15 years.
The "harvest now, decrypt later" threat model means data encrypted today with RSA or ECDH
is at risk even before a CRQC is built.

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography].

=== Digital Signature

A cryptographic mechanism that provides authentication, integrity, and non-repudiation
for digital data.

The signer uses a private key to produce the signature; any party with the corresponding
public key can verify it.
A valid signature proves: the data has not been altered, and the holder of the private key
produced it.

Classical algorithms: RSA, ECDSA, DSA (all vulnerable to Shor's algorithm).
Post-quantum algorithms: ML-DSA (FIPS 204), SLH-DSA (FIPS 205).

=== DRBG (Deterministic Random Bit Generator)

A cryptographic algorithm that generates pseudorandom bits from a seed value.

NIST-approved DRBGs (Hash_DRBG, HMAC_DRBG, CTR_DRBG) are the only acceptable RNG
implementations in FIPS-compliant systems.

In practice, UMRS code uses the OS CSPRNG (`/dev/urandom` or `getrandom(2)`) as the
preferred interface; the kernel's DRBG backs it.

=== ECDH (Elliptic Curve Diffie-Hellman)

A key agreement algorithm that establishes a shared secret between two parties without
transmitting the secret.

Preferred curve: P-384 (~192-bit security).

See also: xref:cryptography:fips-cryptography-cheat-sheet.adoc[FIPS Cryptography Quick Reference].

=== ECDSA (Elliptic Curve Digital Signature Algorithm)

A digital signature algorithm based on elliptic curve cryptography.

Preferred curve: P-384.
Used in UMRS for signing audit log chains and software artifacts.

=== FIPS Mode

An operating system configuration that restricts the kernel and userspace cryptographic
libraries to use only FIPS-validated algorithms and implementations.

On RHEL 10, FIPS mode is enabled at install time or via `fips-mode-setup`.
When active, the kernel sets `/proc/sys/crypto/fips_enabled` to `1`, and the OpenSSL
FIPS provider becomes the only available cryptographic backend.

In UMRS, `ProcFips` is a first-class kernel attribute.
Any cryptographic operation must use a FIPS-validated primitive when FIPS mode is active.

=== FIPS (Federal Information Processing Standard)

A class of standards published by NIST defining approved cryptographic algorithms and
implementation requirements for U.S. government systems.

Key standards relevant to UMRS:

* FIPS 140-2/3 — cryptographic module validation
* FIPS 197 — AES
* FIPS 180-4 — SHA-2 hash functions
* FIPS 186-5 — ECDSA and RSA signatures
* FIPS 203 — ML-KEM (post-quantum key encapsulation)
* FIPS 204 — ML-DSA (post-quantum digital signatures)
* FIPS 205 — SLH-DSA (post-quantum hash-based signatures)

=== Hash Function

A one-way mathematical function that maps arbitrary input data to a fixed-length digest.

Properties: deterministic, collision-resistant, pre-image resistant.

Used in UMRS for integrity checking, digital signatures, key derivation, and audit
log chaining.

[cols="1,1,2", options="header"]
|===
| Algorithm | Digest size | Status

| SHA-512 | 512 bits | Preferred
| SHA-384 | 384 bits | Preferred
| SHA-256 | 256 bits | Approved
| SHA-1 | 160 bits | Disallowed for signatures
| MD5 | 128 bits | Disallowed
|===

=== Grover's Algorithm

A quantum algorithm that provides a quadratic speedup for searching unstructured spaces.

Applied to symmetric cryptography, it halves the effective security level: a 128-bit
key provides approximately 64 bits of quantum security.
Consequence: AES-128 is insufficient in a post-quantum threat model; AES-256 remains adequate.

Unlike Shor's algorithm, Grover's does not break symmetric cryptography — it weakens it.
The mitigation is to use larger key sizes, not to replace algorithms.

=== HMAC (Hash-based Message Authentication Code)

A keyed hash construction that provides both integrity and authenticity.

HMAC combines a secret key with a hash function (e.g., HMAC-SHA-256) to produce a tag
that can only be verified by a party holding the same key.

Used in UMRS for: EVM extended attribute protection, Sealed Evidence Cache (SEC) integrity
seals, and key derivation (HKDF).

See also: NIST SP 800-107.

=== Hybrid Cryptography

A migration strategy that combines classical and post-quantum algorithms in a single
protocol exchange.

Example: hybrid key agreement uses ECDH P-384 combined with ML-KEM-768 — the resulting
shared secret is secure as long as _either_ algorithm remains unbroken.

Hybrid deployment provides defense in depth during the transition period before
post-quantum algorithms have accumulated sufficient operational confidence.

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography — Migration Approach].

=== HKDF (HMAC-based Key Derivation Function)

A KDF that derives cryptographic keys from shared secrets using HMAC.
Standardized in NIST SP 800-56C.
Preferred variant: HKDF-SHA-384.

=== KEM (Key Encapsulation Mechanism)

A cryptographic primitive that allows one party to securely transmit a shared secret
to another using the recipient's public key.

A KEM is _not_ a key agreement protocol like ECDH (a Non-Interactive Key Exchange / NIKE).
In a KEM, the sender generates the shared secret and encapsulates it; the recipient
decapsulates it with their private key.
Code that uses an ECDH API cannot be trivially swapped for a KEM without changes to the
key establishment flow.

ML-KEM (FIPS 203) is the NIST-standardized post-quantum KEM.

=== KDF (Key Derivation Function)

A function that derives one or more cryptographic keys from a source of key material
(a shared secret, a password, or another key).

KDFs ensure derived keys are cryptographically independent and properly distributed.
NIST-approved KDFs include HKDF (SP 800-56C) and PBKDF2 (SP 800-132).

=== ML-DSA (Module-Lattice Digital Signature Algorithm)

A NIST-standardized post-quantum digital signature algorithm.
Standardized in FIPS 204 (2024).
Formerly known as CRYSTALS-Dilithium.

Preferred for new high-assurance signature use cases where post-quantum resistance is required.
Preferred parameter set: ML-DSA-87 (highest security margin).

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography].

=== ML-KEM (Module-Lattice Key Encapsulation Mechanism)

A NIST-standardized post-quantum key encapsulation mechanism.
Standardized in FIPS 203 (2024).
Formerly known as CRYSTALS-Kyber.

Preferred for new key establishment protocols where post-quantum resistance is required.
Preferred parameter set: ML-KEM-1024 (highest security margin).

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography].

=== RSA (Rivest-Shamir-Adleman)

A classical public-key cryptosystem used for key exchange and digital signatures.

Security basis: the difficulty of factoring large integers — a problem Shor's algorithm
solves efficiently on a quantum computer.

RSA remains widely deployed (RPM signing, TLS, SSH) but is being migrated to post-quantum
replacements: ML-KEM (FIPS 203) for key encapsulation, ML-DSA (FIPS 204) for signatures.
NIST IR 8547 establishes a 2035 deprecation deadline for RSA in federal systems.

=== Shor's Algorithm

A quantum algorithm that efficiently solves integer factorization and discrete logarithm
problems — the mathematical foundations of RSA, ECDH, ECDSA, and DSA.

A sufficiently powerful quantum computer running Shor's algorithm would break all
currently deployed public-key cryptography.
This is the primary motivation for post-quantum cryptography.

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography — The Quantum Threat].

=== Symmetric Cryptography

Cryptography that uses a single shared key for both encryption and decryption (or for
both tag generation and verification in the case of MACs).

Examples: AES (encryption), HMAC (authentication), SHA-2 (hashing).
Symmetric algorithms are not broken by Shor's algorithm but are weakened by Grover's
algorithm — use AES-256 (not AES-128) in post-quantum threat models.

Contrast with asymmetric cryptography, which uses key pairs.

=== Post-Quantum Cryptography (PQC)

Cryptographic algorithms designed to resist attacks by sufficiently powerful quantum computers.

Classical algorithms (RSA, ECDSA, ECDH) rely on mathematical problems believed to be
efficiently solvable by quantum computers via Shor's algorithm.
NIST standardized three PQC algorithms in 2024:

* FIPS 203 — ML-KEM (key encapsulation)
* FIPS 204 — ML-DSA (digital signatures)
* FIPS 205 — SLH-DSA (hash-based signatures)

See also: xref:cryptography:crypto-post-quantum.adoc[Post-Quantum Cryptography].

=== SLH-DSA (Stateless Hash-Based Digital Signature Algorithm)

A NIST-standardized post-quantum digital signature algorithm.
Standardized in FIPS 205 (2024).
Formerly known as SPHINCS+.

Uses hash-function security assumptions rather than lattice-based assumptions —
a conservative choice for deployments that prefer not to rely on the security of
lattice-based mathematics.

Signatures are larger than ML-DSA; appropriate where conservative assumptions
outweigh size constraints.

== UMRS Patterns

=== Two-Path Independence (TPI)

A parsing and validation pattern that requires two completely independent implementations
to parse the same input and agree before the result is trusted.

The threat TPI addresses is parser confusion: if a single parser misinterprets a
security-critical input — a security context, a label, a policy value — the system may
grant access it should deny, or deny access it should grant.
A second independent parser eliminates the class of single-parser errors.
If the two parsers disagree, the system fails closed.

In UMRS, the canonical implementation is `SecurityContext`, which uses both a `nom`
combinator parser and a `FromStr` implementation.
These two paths are architecturally independent.
A parsing result is only accepted when both paths produce identical output.
Any disagreement returns an error.

TPI is required for all parsers that consume security-critical input.
The pattern applies to: security context parsing, sensitivity level parsing, category set
parsing, and any new parser introduced for security-relevant configuration or kernel data.

See also: xref:patterns:pattern-tpi.adoc[Two-Path Independence pattern];
NIST SP 800-218 SSDF PW.4.

=== Fail-Closed

A behavioral property of a security mechanism: when the mechanism encounters an error,
unexpected input, or ambiguous state, it denies access rather than permitting it.

Fail-closed is the default safety posture for security enforcement in UMRS.
A parser that fails closed returns an error on malformed input — it does not guess,
default to permissive, or fall through to a less restrictive path.

The alternative — fail-open — permits access when the mechanism cannot make a determination.
Fail-open is appropriate for availability-critical systems where denying is worse than
permitting.
In a security-enforcement context, fail-open is almost always wrong.

In UMRS, every security-critical function that returns `Result` must carry `#[must_use]`
so callers cannot discard the error.
This is the compile-time enforcement of the fail-closed property.

See also: NSA RTB (Fail Secure); NIST SP 800-53 SI-10 (Information Input Validation).

=== TOCTOU Safety

A property that prevents time-of-check to time-of-use (TOCTOU) races when accessing
files and kernel attributes.

A TOCTOU race occurs when:

. A security check is performed on a resource (for example, checking that a file belongs
   to root and has the expected SELinux label).
. The resource is accessed using a different, re-resolvable reference (for example, a path).
. Between steps 1 and 2, an attacker substitutes a different resource at the path.

The check passed, but the access operates on the attacker's resource.

TOCTOU-safe access anchors both the check and the access to the same file descriptor.
Once an `fd` is open, it refers to exactly one inode.
An attacker cannot substitute a different file at the path to affect an already-open `fd`.

In UMRS, all file and xattr access is anchored to file descriptors.
`SecureDirent` and `SecureXattrReader` are the types that enforce this property.
Directory traversal also uses fd-anchored (`openat`) operations to prevent directory
substitution attacks.

See also: xref:patterns:pattern-toctou.adoc[TOCTOU Safety pattern];
NIST SP 800-53 SI-10.
