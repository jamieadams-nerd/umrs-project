= What AI Does and Does Not Do
:description: A precise accounting of AI agent scope in the UMRS project — what they produce, where human judgment gates every decision, and why the boundaries are where they are.
:toc: left
:toclevels: 2

[id="what-ai-does-and-does-not-do"]

The most important question about AI involvement in any project is not "did you use AI?"
It is "what did AI actually do, and where does human judgment take over?"

This page answers that question precisely.

== What AI Agents Do

=== Standards Research and Citation Identification

Agents query the project's RAG knowledge base — a collection of checksummed reference
documents including NIST SPs, FIPS standards, CMMC guides, DoD instructions, and
foundational access-control papers.
They retrieve specific chunks from those documents and identify applicable controls.

This is the highest-value AI contribution in the project.
A single engineer cannot hold the full content of NIST SP 800-53 Rev 5, NIST SP 800-171
Rev 3, CMMC Level 2 Assessment Guide, and DoDM 5200.01 in working memory simultaneously.
Agents can search across all of them in seconds and return specific citations with source
attribution.

The boundary: agents identify candidates.
The engineer confirms them.
See xref:knowledge-provenance.adoc[Knowledge Provenance and Claim Traceability] for the
full verification chain.

=== Documentation Drafting

Agents write documentation against an established architecture.
The engineer has already made the structural and design decisions.
The agent produces prose that describes those decisions accurately.

Documentation generated by agents is reviewed by the engineer before acceptance.
Control citations in documentation are verified against the RAG knowledge base.
The Antora module structure, page placement, and navigation decisions are made by
the senior technical writer agent subject to engineer approval.

=== Code Review for Pattern Compliance

Agents review Rust code for compliance with the project's high-assurance pattern rules:
missing `#[must_use]` annotations, unsafe block detection (blocked at compile time by
`#![forbid(unsafe_code)]`), compliance annotation gaps, and pattern application
opportunities.

The security-auditor agent applies RMF assessment methodology — classifying its own
review activities by method type (Examine, Interview, Test) and identifying evidence
gaps rather than just annotation gaps.

The boundary: agents flag findings.
The engineer decides whether a finding requires action and how to address it.

=== Compliance Annotation Proposals

Agents propose compliance annotations for code comments and documentation.
They present the relevant control, the applicable passage from the reference document,
and the rationale for the mapping.

The boundary: these are proposals.
The engineer reviews and accepts or rejects each one.
The commit is the acceptance.

=== Test Case Proposals

Agents propose test cases for security-relevant code paths.
They analyze the code against the high-assurance pattern rules and identify untested
invariants, especially around fail-closed behavior, error propagation, and boundary
conditions.

=== Structural Planning

Agents maintain documentation plans, track phase completion, and manage cross-module
cross-reference consistency.
The plans directory (`.claude/plans/`) is the active work queue.
Agents read from it and report progress; the engineer sets priorities.

=== Reviewer Personas

Agents act as structured reviewer personas — a new-engineer perspective that flags
confusion in documentation, and a security-auditor perspective that flags ambiguity
and missing control citations.

These review passes surface blind spots that the primary author cannot see.
Every documentation page in this project is intended to pass both reviews before
being published.

== What AI Agents Do Not Do

=== Architectural Decisions

The security model, enforcement strategy, trust boundaries, and system design are
determined by the project engineer.
Agents analyze and propose; the engineer decides.

The reason this boundary exists: architectural decisions in a high-assurance system
require context that cannot be fully captured in any reference document.
They require knowledge of the specific deployment environment, the specific threat model,
and the specific operational requirements.
An agent that makes architectural decisions without that context will optimize for
the wrong objectives.

=== SELinux Policy Authorship

SELinux policy modules are written and reviewed by the project engineer.
Agents may assist with research — looking up relevant policy constructs, citing SELinux
Notebook sections, identifying potential constraint issues — but they do not write
production policy.

The reason: SELinux policy is binary in its effects.
A policy module that grants an unintended permission creates a real vulnerability.
The review surface for policy authorship requires expert human judgment that agent
output cannot substitute for.

=== Final Compliance Claims

UMRS documentation contains compliance claims.
Agents identify candidates; engineers confirm them.
No compliance claim is treated as accurate simply because an agent produced it.

The reason: compliance claims in a DoD-adjacent environment have legal and contractual
implications.
An incorrect citation is not just a documentation error — it is a false representation
of the system's control implementation.

=== Git Commits

Agents are explicitly prohibited from committing to the repository.
This is a hard technical rule, not a policy.

Every commit in this repository was made by a human engineer who reviewed what they
were committing.
The commit record is the audit trail.
If an AI agent could commit, that trail would be broken.

=== Access to Production Systems

Agents operate in a sandboxed development environment.
They have no access to production systems, classified data, or operational deployments.

== The Review Gate

Every AI-generated artifact passes through engineer review before acceptance.
This is the most important control in the pipeline.

For code: every AI-drafted Rust function is reviewed against the high-assurance pattern
rules.
The compile-time enforcement (`#![forbid(unsafe_code)]`, clippy pedantic + nursery) is
non-negotiable regardless of AI involvement.

For documentation: every AI-drafted page is reviewed for technical accuracy, control
citation correctness, and alignment with the documentation architecture.

For plans and structure: documentation architecture decisions proposed by agents are
reviewed and confirmed before execution.

The review gate is where human judgment exercises final authority.
The commit is where that judgment is recorded.

== Summary Table

[cols="2,3,3", options="header"]
|===
| Domain | AI contributes | Human decides

| Standards research
| Retrieves and presents specific control citations from checksummed reference documents
| Confirms accuracy; accepts or rejects the citation

| Documentation
| Drafts prose, structures pages, manages cross-references
| Reviews for accuracy; commits accepted pages

| Code review
| Flags pattern compliance issues; identifies annotation gaps; applies RMF methodology
| Acts on findings; accepts or rejects each flag

| Compliance annotations
| Identifies candidate controls with rationale
| Confirms the mapping; accepts in commit

| Test cases
| Proposes tests for untested invariants
| Reviews and accepts test implementations

| Planning
| Maintains plan documents; tracks progress
| Sets priorities; approves phase transitions

| Architectural decisions
| Does not make them
| Sole decision-maker

| SELinux policy
| Does not author it
| Sole author and reviewer

| Git commits
| Cannot commit — hard technical rule
| Every commit is a human action
|===
