= Why We Document the AI Pipeline
:description: The case for treating AI methodology as a first-class audit artifact in a high-assurance project.
:toc: left
:toclevels: 2

[id="why-transparency"]

Most projects that use AI say: "We used AI tools. All output was reviewed."

That disclosure is accurate, and it is also nearly useless.
It tells an auditor or adopter nothing about scope, knowledge sources, review requirements,
or where human judgment sits in the process.
For a general-purpose web app, that ambiguity might be acceptable.
For a project handling Controlled Unclassified Information in a DoD-adjacent environment,
it is not.

UMRS is a high-assurance project.
The same evidence discipline that applies to the code applies to the documentation of
how that code was produced.

== The Traceability Argument

UMRS documentation contains hundreds of specific compliance claims.
Control citations like `NIST SP 800-53 AC-4` or `CMMC SC.L2-3.13.3` appear in Rust doc
comments, architecture pages, and pattern descriptions throughout the codebase.

An auditor reviewing those citations has a legitimate question: where did these come from?
Were they looked up in the actual standard text, or were they generated by an AI that
"knows" NIST controls the way it "knows" other things — approximately, with confident
tone, and occasional errors?

The answer matters.
A citation that traces back to a specific chunk of a checksummed NIST SP PDF is a very
different artifact than a citation that came from a model's training data with no
verification path.

The documentation in this module makes that distinction explicit and traceable.
See xref:knowledge-provenance.adoc[Knowledge Provenance and Claim Traceability] for the
full traceability chain.

== Why "We Used AI" Is Insufficient Disclosure

The phrase covers an enormous range of practices:

* An engineer who pasted a requirement into a chat window and accepted the first response
* A team that built a grounded knowledge system with checksummed reference documents,
  domain-specific RAG collections, and structured review gates
* Everything in between

For an auditor, adopter, or compliance reviewer, these scenarios have meaningfully different
risk profiles.
The UMRS approach is the second one.
Documenting the pipeline honestly lets the reader make that assessment rather than
guessing.

== The Asymmetry Between Confidence and Accuracy

Language models are fluent.
They produce confident, well-structured output even when they are wrong.
This asymmetry is the primary risk in AI-assisted technical work: the output looks
authoritative regardless of whether it is correct.

The mitigation is grounding.
When an agent answers a question by retrieving relevant chunks from a checksummed
reference document and presenting what that document actually says, the confidence-accuracy
gap narrows significantly.
The agent is not generating from training data — it is reading and summarizing a
specific source that you can inspect.

This is why the UMRS AI pipeline centers on Retrieval-Augmented Generation (RAG).
See xref:knowledge-pipeline.adoc[How Knowledge Enters the System] for how it works.

== The Human-Judgment Requirement

No AI decision in this project is final.
Every artifact that enters the codebase or documentation — whether written by an agent
or written by an engineer — goes through human review and acceptance.

The commit record is the authoritative audit trail.
AI tools cannot commit to the repository.
A human engineer made every commit.
That engineer reviewed what they committed.

This is not a policy statement.
It is a hard technical rule enforced by how the tools are configured.

== Why This Module Is Itself a Compliance Artifact

NIST SP 800-53 SA-11 (Developer Security Testing and Evaluation) requires that development
teams establish and document their testing and evaluation processes.
NIST SP 800-53 AU-2 (Event Logging) requires documentation of what is logged and why.
NIST SP 800-53 SI-7 (Software, Firmware, and Information Integrity) requires integrity
verification for software components.

This module — and the references it points to — demonstrates that UMRS applies the same
evidence discipline to its AI development pipeline that it asks of its code.
The pipeline is inspectable.
The knowledge sources are checksummed.
The review gates are documented.
The agent behaviors are governed by standing rules that are checked into the repository.

That is what it means to treat an AI pipeline as a compliance artifact.
