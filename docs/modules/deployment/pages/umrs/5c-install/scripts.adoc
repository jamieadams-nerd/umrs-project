= Scripts: umrs-sign-mgr and umrs-shred
:toc: left
:description: Purpose, placement, and integration of the umrs-sign-mgr and umrs-shred operator scripts.

This page covers two operator scripts that support the UMRS installation and
key lifecycle. Neither script is yet present in
`components/rusty-gadgets/scripts/` as of 2026-04-16; their design is documented
here so the placement and integration are explicit when they are written.

NOTE: The legacy key management tool referenced in older operations documentation
was called `ha-signing-manager.sh` with base directory `/etc/pki/ha-sign`. The
UMRS-integrated version (`umrs-sign-mgr`) targets the authoritative key material
paths at `/etc/keys/umrs/` (keys at rest) and `/var/lib/umrs/keys/` (runtime
lifecycle state). See xref:umrs/5b-directory-structure/key-material-trees.adoc[Key Material Trees].

== umrs-sign-mgr

`umrs-sign-mgr` is the signing key lifecycle management script for UMRS. It manages
the eight key classes (audit, syslog, journal, logrotate, archive, config, rpmdb,
swrel) stored under `/etc/keys/umrs/signing/`.

=== Design Principles

* *Deterministic structure* --- keys live in well-defined locations with
  predictable names (see xref:umrs/5b-directory-structure/key-material-trees.adoc[Key Material Trees]).
* *Strong defaults* --- FIPS-approved algorithms (RSA-3072 minimum, ECDSA P-256/P-384)
  enforced by default. Refuses weaker choices.
* *No passphrases for automation* --- signing keys used by unattended jobs are
  created without passphrases. Compensating controls are file permissions, account
  isolation, and rotation.
* *Full lifecycle coverage* --- init, rotate, list, report, and import subcommands.
* *Auditor-friendly reporting* --- explicit mapping between keys, purpose, locations,
  algorithms, and lifetimes.

=== Planned Subcommands

[cols="1,3",options="header"]
|===
| Subcommand | Description

| `init`
| Initializes the key material hierarchy. Creates `/etc/keys/umrs/` subdirectories,
  enforces secure permissions and ownership (`0700 root:root`), and optionally creates
  initial signing keys for each of the eight key classes.

| `rotate <class>`
| Generates a new key pair for the given class. Moves the current private key
  to `archive/`, updates the `current` symlink, and records the rotation in
  `meta/<class>_signing_keys.json`.

| `list`
| Inventories all known keys. Reports purpose, algorithm, key length, generation
  date, expiration date, and active vs. archived status.

| `report`
| Generates an auditor-ready report: system information, one section per key
  class, and a compliance mapping to NIST SP 800-53 controls.

| `import`
| Discovers and incorporates existing public keys from configurable search roots
  (`/etc/pki`, `/etc/ssh`, application paths).
|===

=== Example: Signing and Verifying an Audit Log

Sign a rotated audit log:

[source,bash]
----
openssl dgst -sha256 \
  -sign /etc/keys/umrs/signing/ha-logger01_audit_sign_rsa-3072_v01.key \
  -out /var/log/audit/audit.log.1.sig \
  /var/log/audit/audit.log.1
----

Verify the signature:

[source,bash]
----
openssl dgst -sha256 \
  -verify /etc/keys/umrs/signing/ha-logger01_audit_sign_rsa-3072_v01.pem \
  -signature /var/log/audit/audit.log.1.sig \
  /var/log/audit/audit.log.1
----

Output `Verified OK` confirms the file has not been modified since signing
and that the signature was produced by the matching private key.

=== Logrotate Integration

In `/etc/logrotate.d/audit`:

[source]
----
postrotate
    /usr/sbin/service auditd reload >/dev/null 2>&1 || true
    /opt/umrs/bin/umrs-sign-mgr sign-audit /var/log/audit/audit.log.1
endscript
----

=== Placement

The script will be installed to `/opt/umrs/bin/umrs-sign-mgr`. It runs as root.
Per the Privileged Script Rule (`.claude/rules/secure_bash_rules.md`), it must be:

* Owned by `root:root`
* Mode `0700` or `0750`
* Located in a root-owned, non-world-writable directory

=== Compliance

[cols="1,3",options="header"]
|===
| Control | Coverage

| NIST SP 800-53 SC-12 | Key generation, rotation, archival, documented lifecycle
| NIST SP 800-53 SC-13 | Enforces FIPS-approved algorithms; refuses weaker choices
| NIST SP 800-53 AU-9  | Keys dedicated to signing audit logs; verifiable via public keys
| NIST SP 800-53 AU-10 | Signed artifacts link to authenticated key holders
| NIST SP 800-53 SI-7  | Integrity verification of logs and archives
| NIST SP 800-53 CM-6  | Configuration is explicit and reviewable
| NIST SP 800-53 CA-7  | Scheduled report runs provide continuous key posture visibility
|===

== umrs-shred

`umrs-shred` is a secure file deletion wrapper for use within the UMRS custody
boundary. It provides a documented, logged, policy-aligned alternative to `shred(1)`
for files labeled with UMRS SELinux types.

NOTE: `umrs-shred` is not yet written. This entry documents its intended purpose
and placement for planning purposes.

=== Intended Design

* Wraps `shred(1)` with pre-/post-conditions appropriate for UMRS file types.
* Logs every invocation to syslog (`auth.notice`) with the file path, SELinux type,
  and user identity.
* Refuses to operate on files outside the UMRS type set unless explicitly overridden.
* Requires operator confirmation for files labeled `umrs_pki_key_t`.

=== Placement

`/opt/umrs/bin/umrs-shred`. Runs as root. Same Privileged Script Rule requirements
as `umrs-sign-mgr`.
