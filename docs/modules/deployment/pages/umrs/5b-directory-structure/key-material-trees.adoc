= Key Material Trees
:toc: left
:description: UMRS key material directory structure, DAC permissions, and SELinux MAC types. Covers /etc/pki/, /etc/keys/umrs/, and /var/lib/umrs/keys/.

This page documents the on-disk layout for all UMRS cryptographic key material.
It covers the system PKI root (`/etc/pki/`), the UMRS-specific key material at
rest (`/etc/keys/umrs/`), and the runtime key lifecycle state (`/var/lib/umrs/keys/`).

The layout follows FHS 3.0 directory semantics and NIST SP 800-57 Part 1 Rev 5 key
lifecycle states. Where FHS and NIST create tension — and they do, in four specific
places — the conflicts and their resolutions are documented in
<<_fhs_3_0_and_nist_conflicts_and_resolutions>>.

== Standards Overview

=== FHS 3.0 File Categories

FHS 3.0 was originally published June 3, 2015 by the Linux Foundation and
republished November 6, 2025 by FreeDesktop, which now maintains the
specification. It is the current and only active version. There is no FHS 4.x.

FHS 3.0 defines four categories of files relevant to key material placement:

[cols="1,2,2",options="header"]
|===
| Category | Definition | Examples

| Static + Unshareable
| Does not change without admin action; host-specific
| `/etc/` config, private keys

| Static + Shareable
| Does not change without admin action; sharable
| `/usr/` binaries, CA certs

| Variable + Unshareable
| Changes at runtime; host-specific
| `/var/lib/` runtime state

| Variable + Shareable
| Changes at runtime; shareable
| `/var/mail/`
|===

Key material is *static + unshareable* at rest. Runtime and session key state is
*variable + unshareable*.

=== Relevant FHS 3.0 Sections

[cols="1,1,2",options="header"]
|===
| Section | Path | Relevance

| §3.5
| `/etc/`
| Host-specific configuration and static secrets

| §3.15
| `/var/`
| Variable runtime data

| §4.5
| `/opt/`
| Add-on application packages

| §3.13
| `/etc/opt/`
| Configuration for `/opt/` packages
|===

FHS §3.5 explicitly states `/etc/` must not contain binaries and is reserved for
host-local configuration. Private key material stored as inert files is consistent
with this constraint.

=== NIST SP 800-57 Part 1 Rev 5 — Key Lifecycle States

SP 800-57 §5.3 defines the following states relevant to directory placement:

[cols="1,2,2",options="header"]
|===
| State | Description | Directory Implication

| Pre-activation
| Generated, not yet in use
| `/etc/keys/umrs/staging/`

| Active
| In use for protect/unprotect operations
| `/etc/keys/umrs/<type>/`

| Suspended
| Temporarily unusable, not destroyed
| `/var/lib/umrs/keys/suspended/`

| Deactivated
| No longer used to protect, may still unprotect
| `/var/lib/umrs/keys/retired/`

| Destroyed
| Cryptographically erased
| N/A — material removed
|===

== Directory Structure

The full directory tree is shown below. Each subtree is described in detail
in the sections that follow.

[source,text]
----
/etc/pki/                               # System PKI root — RHEL convention, FHS §3.5
│                                       # NIST SC-12, SC-17, SP 800-57 §5.3.4
├── tls/
│   ├── certs/                          # Public certificates (CA bundles, host certs)
│   │   ├── ca-bundle.crt               # mode 0644 root:root — world-readable trust anchors
│   │   └── umrs-host.crt               # mode 0644 root:root
│   └── private/                        # Asymmetric private keys
│       └── umrs-host.key               # mode 0600 root:root — NIST SC-12(1)
│
├── ca-trust/                           # RHEL system trust anchor management
│   ├── source/anchors/                 # Drop zone for local/airgap CA certs
│   └── extracted/                      # Output of update-ca-trust(8)
│
├── rpm-gpg/                            # Package signing public keys (informational)
└── nssdb/                              # NSS shared certificate database

/etc/keys/                              # Application-layer key material — NIST SC-12, SP 800-57 §5.3
└── umrs/                               # Project namespace — mode 0700 root:root
    │
    ├── sealing/                        # Symmetric keys for LUKS/dm-crypt volume sealing
    │   │                               # NIST SC-28, SP 800-57 §5.3.5, SP 800-38F
    │   └── vol-root.key                # mode 0600 root:root
    │
    ├── signing/                        # IMA/DIGSIG asymmetric signing keys
    │   │                               # NIST SI-7, SC-12, SP 800-89
    │   └── ima-signing.key             # mode 0600 root:root
    │
    ├── wrapping/                       # Key Encryption Keys (KEKs) for key wrapping
    │   │                               # NIST SC-12(2), SP 800-38F (AES-KW / AES-KWP)
    │   └── kek-primary.key             # mode 0600 root:root
    │
    └── staging/                        # Pre-activation key material — SP 800-57 §5.3 pre-active state
        └── mode 0700 root:root

/etc/opt/umrs/                          # FHS §3.13 — configuration for /opt packages
│                                       # NOT for key material; policy and metadata ONLY
└── key-policy.toml                     # Key lifecycle policy — SP 800-57 §5.2, NIST PL-1, SA-8

/var/lib/umrs/                          # FHS §5.x — variable, persistent, host-specific runtime data
└── keys/                               # Runtime key lifecycle state — SP 800-57 §5.3
    │                                   # NIST SC-12, AC-3
    ├── active/                         # Currently loaded/unwrapped session keys
    │   └── mode 0700 root:root         # Short-lived; cleared on session end
    │
    ├── suspended/                      # Temporarily deactivated keys (SP 800-57 suspended state)
    │   └── mode 0700 root:root
    │
    └── retired/                        # Post-active, pre-destruction staging
        └── mode 0700 root:root         # NIST MP-6, SC-12 — awaiting secure deletion
----

=== System PKI Root (/etc/pki/)

`/etc/pki/` is the RHEL system PKI root. It is not UMRS-specific — it is the
platform standard location for TLS certificates, CA trust anchors, and the NSS
shared database. UMRS places its TLS host certificate and private key here in
the standard RHEL locations.

`/etc/pki/tls/certs/` holds public certificates. These are world-readable by
convention: CA bundles and host certificates must be accessible to services
verifying TLS connections. `/etc/pki/tls/private/` holds asymmetric private keys
at mode `0600 root:root`.

`/etc/pki/ca-trust/source/anchors/` is the drop zone for local or airgap CA
certificates. After placing a certificate here, run `update-ca-trust` to
propagate it to the extracted trust store.

Controls: NIST SP 800-53 SC-12, SC-17; SP 800-57 §5.3.4.

=== UMRS Key Material at Rest (/etc/keys/umrs/)

`/etc/keys/umrs/` is the UMRS project namespace within the system application
key directory at `/etc/keys/`. This is *not* under `/etc/opt/umrs/`. The
separation is intentional: `/etc/opt/umrs/` is for policy metadata and tool
configuration; raw key material requires a dedicated, tightly scoped path outside
the add-on package tree.

WARNING: Every directory under `/etc/keys/umrs/` must be `0700 root:root`. Every
key file must be `0600 root:root`. No group bits, no world bits. A single
permission deviation exposes key material to non-root processes under DAC. In
Phase 1, DAC is the primary enforcement boundary.

The four subdirectories map to distinct key functions and distinct NIST controls:

* `sealing/` — symmetric keys for LUKS and dm-crypt volume sealing. Controls:
  NIST SP 800-53 SC-28; SP 800-57 §5.3.5; SP 800-38F.
* `signing/` — IMA/DIGSIG asymmetric signing keys for file integrity. Controls:
  NIST SP 800-53 SI-7, SC-12; SP 800-89.
* `wrapping/` — Key Encryption Keys (KEKs) used in AES-KW and AES-KWP wrapping
  operations. Controls: NIST SP 800-53 SC-12(2); SP 800-38F.
* `staging/` — pre-activation key material. Keys in this directory are generated
  but not yet in use (SP 800-57 §5.3 pre-active state). Controls: NIST SP 800-53
  SC-12, CM-3.

=== UMRS Runtime Key State (/var/lib/umrs/keys/)

[IMPORTANT]
====
The directories under `/var/lib/umrs/keys/` are *declared and typed* in policy
but are not created by `umrs-install.sh` in the current release. The SELinux
fcontext rules and type declarations are in place. Runtime creation and lifecycle
management are tracked for a future pass.
====

`/var/lib/umrs/keys/` holds runtime key lifecycle state. It sits under
`/var/lib/umrs/` following the FHS §5.x pattern for variable, persistent,
host-specific application state.

The three subdirectories correspond to SP 800-57 §5.3 lifecycle states:

* `active/` — currently loaded or unwrapped session keys. Short-lived; content
  is cleared when the session ends. Controls: NIST SP 800-53 SC-12, AC-3, SC-4;
  SP 800-57 §5.3.3.
* `suspended/` — keys in the SP 800-57 suspended state: temporarily deactivated
  but not destroyed. Controls: NIST SP 800-53 SC-12, AC-2; SP 800-57 §5.3.
* `retired/` — post-active keys pending cryptographic destruction. This is a
  staging area, not permanent archival. A destruction service must periodically
  execute secure deletion (`shred -u` or dm-crypt key slot wipe) against this
  directory. Controls: NIST SP 800-53 SC-12, MP-6, MP-7; CMMC MP.L2-3.8.3;
  SP 800-57 §5.3.6.

NOTE: The choice to place runtime key state under `/var/lib/umrs/keys/` rather
than `/var/opt/umrs/` is a deliberate deviation from the FHS add-on package
pattern. The rationale is in <<_6_3_conflict_fhs_opt_autonomy_vs_nist_configuration_control>>.

=== Policy Metadata (/etc/opt/umrs/key-policy.toml)

`/etc/opt/umrs/key-policy.toml` is the one file in the add-on package config
tree that relates to key management. It contains the key lifecycle policy:
rotation schedules, algorithm selections, and retention periods.

This file is *policy metadata*, not key material. It does not contain keys, key
handles, or keying material of any kind. Its placement under `/etc/opt/umrs/` is
correct: it is admin-editable configuration for the `/opt/umrs/` package.

Controls: NIST SP 800-53 PL-1, SA-8; SP 800-57 §5.2.

For the full key material story, all paths are documented in the
<<_control_mapping_by_path>> table and the
<<_filesystem_permissions_reference>> table below.

== Filesystem Permissions Reference

All key-bearing directories and files must be owned `root:root`. No `setuid`,
`setgid`, or world bits are permitted on any path under `/etc/keys/` or
`/var/lib/umrs/keys/`.

[cols="3,1,2",options="header"]
|===
| Path | Mode | Rationale

| `/etc/keys/umrs/`
| `0700`
| Namespace root — no listing by non-root

| `/etc/keys/umrs/*/`
| `0700`
| Subdirectory — no listing by non-root

| `/etc/keys/umrs/**/*.key`
| `0600`
| Key files — root read/write only

| `/etc/pki/tls/private/`
| `0700`
| RHEL convention; consistent with above

| `/etc/pki/tls/private/*.key`
| `0600`
| Asymmetric private keys

| `/etc/pki/tls/certs/`
| `0755`
| Public certs — world-readable is correct

| `/var/lib/umrs/keys/`
| `0700`
| Runtime state root

| `/var/lib/umrs/keys/*/`
| `0700`
| All lifecycle subdirs
|===

== SELinux MAC Types

FHS defines *where* files live. SELinux MLS defines *who* can access them
regardless of DAC permissions. Both layers are required. DAC alone (chmod/chown)
is insufficient for MLS compliance.

=== Recommended Type Assignments

[cols="3,2,1",options="header"]
|===
| Path | SELinux Type | MLS Sensitivity

| `/etc/pki/tls/private/`
| `cert_t` (extend to `umrs_privkey_t`)
| `s3` or per-volume label

| `/etc/keys/umrs/sealing/`
| `umrs_seal_key_t`
| Match protected volume sensitivity

| `/etc/keys/umrs/signing/`
| `umrs_sign_key_t`
| `s3`

| `/etc/keys/umrs/wrapping/`
| `umrs_kek_t`
| Highest sensitivity in use

| `/var/lib/umrs/keys/active/`
| `umrs_session_key_t`
| Transient; matches active session level

| `/var/lib/umrs/keys/retired/`
| `umrs_retired_key_t`
| Preserve original sensitivity label
|===

NOTE: The `/etc/keys/umrs/staging/` path uses `umrs_sign_key_t` in the current
policy. A dedicated `umrs_staging_key_t` may be introduced by the Knox policy
review if stronger pre-activation isolation is warranted. The
`/var/lib/umrs/keys/suspended/` path similarly reuses `umrs_retired_key_t`
pending a Knox decision on whether a separate `umrs_suspended_key_t` is needed.

=== Applicable NIST Controls

* *SC-3* (Security Function Isolation): Key material paths inaccessible to
  processes below required clearance.
* *AC-3(3)* (Mandatory Access Control): SELinux MLS enforces sensitivity-based
  access independent of DAC.
* *SC-28(1)* (Cryptographic Protection of Data at Rest): Applies to retired key
  material pending destruction.

=== Critical Rules

* No `unconfined_t` domain may have read access to any path under `/etc/keys/`
  or `/etc/pki/tls/private/`.
* MLS constraints must prevent read-down: a process at `s1` cannot read material
  labeled `s3`.
* Run `restorecon -RFv` after any manual directory creation to apply file contexts.
* Custom types (`umrs_*_t`) require a policy module. Do not rely on the default
  `file_t` fallback.

[NOTE]
====
The SELinux types listed here (`umrs_seal_key_t`, `umrs_sign_key_t`, `umrs_kek_t`,
`umrs_session_key_t`, `umrs_retired_key_t`) are declared in `umrs.te` and bound
to the `umrs_secret_type` attribute. Phase 2 will add neverallow rules preventing
access from domains other than a dedicated signing domain.
====

== FHS 3.0 and NIST — Conflicts and Resolutions

[IMPORTANT]
====
This section is operationally critical. Every UMRS deployment that stores key
material encounters these four tensions. Every deviation from FHS norms has a
specific NIST-control justification; omitting these justifications from the
system security plan (SSP) is a finding.
====

FHS and NIST do not directly contradict each other, but they operate at different
layers and create real tensions that require explicit design decisions.

=== 6.1 Conflict: `/etc/` Immutability vs. Key Rotation

*FHS Position:* `/etc/` is for _static_ configuration. Files here should not
change without administrator intervention.

*NIST Position:* SP 800-57 §5.3 requires regular key rotation. SC-12 requires
automated key management support. Keys in active directories will change on
rotation schedules.

*Resolution for UMRS:*

* `/etc/keys/umrs/` treats key rotation as an administrative action (consistent
  with FHS) — rotation is a deliberate, logged, privileged operation, not an
  application runtime write.
* Application processes *must not* write directly to `/etc/keys/umrs/` at runtime.
* Rotation is performed by a privileged `umrs-keymgr` process/xtask, logging to
  journald (NIST AU-2, AU-3).
* This satisfies both: FHS static semantics preserved; NIST rotation requirement
  met via a controlled admin path.

=== 6.2 Conflict: FHS `/var/lib/` Persistence vs. NIST Key Destruction

*FHS Position:* `/var/lib/` holds persistent application state that survives
reboots.

*NIST Position:* SP 800-57 §5.3.6 and MP-6 require that destroyed keys are
cryptographically unrecoverable. Retired keys in `/var/lib/umrs/keys/retired/`
must eventually be securely deleted.

*Resolution for UMRS:*

* `/var/lib/umrs/keys/retired/` is a *staging area*, not permanent archival.
* A destruction service must periodically execute secure deletion (`shred -u` or
  dm-crypt key slot wipe) against retired material.
* Destruction events must produce an audit record (NIST AU-2, AU-9, MP-6).
* Do not rely on filesystem delete (`unlink`) — on journaling filesystems this
  does not guarantee data erasure.

=== 6.3 Conflict: FHS `/opt/` Autonomy vs. NIST Configuration Control

*FHS Position:* FHS §4.5 grants `/opt/<package>/` full directory autonomy. A
package may place anything under its own subtree, including subdirectories named
`etc/` or `keys/`.

*NIST Position:* SA-8 (Security Engineering Principles) and CM-6 require that
configuration and secrets follow an auditable, policy-controlled layout. An
ad-hoc `/opt/umrs/keys/` would be outside the system's standard audit and access
control baseline.

*Resolution for UMRS:*

* UMRS *does not* use `/opt/umrs/keys/` or any key storage under `/opt/`.
* `/etc/opt/umrs/` is used only for policy metadata (`key-policy.toml`), never
  raw key material.
* This is a deliberate deviation from FHS §4.5 autonomy in favor of NIST CM-6
  and SA-8 compliance.
* Document this decision in the system security plan (SSP).

This same reasoning applies to `/var/lib/umrs/keys/`: runtime key state belongs
under `/var/lib/` (the FHS standard location for persistent runtime data)
rather than under `/var/opt/umrs/` (the add-on package variable tree). The
standard system path is the better fit for material subject to audit and system-
level access control policies.

=== 6.4 Conflict: FHS Shareable/Unshareable vs. NIST Need-to-Know

*FHS Position:* FHS encourages placing shareable files (e.g., public CA certs)
in `/usr/share/` to allow network mounting.

*NIST Position:* SC-12 and AC-3 require that even public key material distribution
be controlled. Freely network-mounting a CA cert directory may expose the trust
anchor update mechanism to tampering.

*Resolution for UMRS:*

* Public CA certs remain in `/etc/pki/ca-trust/` (RHEL convention, non-shareable).
* UMRS does not place any PKI material under `/usr/share/`.
* In airgap deployments, CA trust anchors are distributed via controlled media,
  not NFS/autofs.

=== 6.5 Summary: Conflicts and Resolutions

[cols="2,2,2,2",options="header"]
|===
| Conflict | FHS Requirement | NIST Requirement | Resolution

| Key rotation in `/etc/`
| Static files only
| Rotation required (SP 800-57 §5.3)
| Rotation is a privileged admin operation, not a runtime write

| Secure deletion in `/var/lib/`
| Persistent state
| Cryptographic destruction (MP-6)
| Retirement staging + active destruction service

| `/opt/` autonomy for keys
| Package owns subtree
| Config control required (CM-6, SA-8)
| No key material under `/opt/`; explicit SSP deviation note

| Shareable public certs
| Place in `/usr/share/`
| Access control on trust anchors (SC-12)
| Stay in `/etc/pki/`; no NFS mount of PKI paths
|===

== Paths to Avoid

[cols="2,3",options="header"]
|===
| Path | Reason

| `/opt/umrs/keys/`
| FHS autonomy conflicts with NIST CM-6; outside standard audit baseline

| `/tmp/`
| World-writable; violates SP 800-57 §5.3 unconditionally

| `/home/<user>/`
| User-accessible; violates AC-3, SC-12

| `/usr/share/`
| Shareable mount point; violates SC-12 need-to-know for trust anchors

| `/run/`
| tmpfs; volatile across reboots; acceptable only for ephemeral session keys with explicit policy

| `/proc/`, `/sys/`
| Pseudo-filesystems; not for persistent material

| `/etc/crypto-policies/`
| RHEL system-owned; do not collide with this namespace
|===

== Implementation Checklist

* [ ] Create `/etc/keys/umrs/` with `install -d -m 0700 -o root -g root`
* [ ] Create all subdirectories with same ownership/mode
* [ ] Write SELinux policy module defining `umrs_*_t` types
* [ ] Run `restorecon -RFv /etc/keys/umrs/` after policy load
* [ ] Verify no `unconfined_t` access via `sesearch --allow -t umrs_seal_key_t`
* [ ] Create `/etc/opt/umrs/key-policy.toml` with rotation schedules and algorithm selections
* [ ] Create `/var/lib/umrs/keys/{active,suspended,retired}/` with `0700 root:root`
* [ ] Implement destruction service for `/var/lib/umrs/keys/retired/` with journald audit logging
* [ ] Document FHS §4.5 deviation (no key material under `/opt/`) in SSP
* [ ] Verify `fips=1` kernel parameter active before any key generation (CMMC SC.L2-3.13.10)

NOTE: As of the current release, steps 7 and 8 (creation of `/var/lib/umrs/keys/`
subdirectories and the destruction service) are not performed by `umrs-install.sh`.
The policy declarations are in place; runtime creation is tracked for a future pass.
