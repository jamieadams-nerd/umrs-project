In high-assurance security engineering, a *reference monitor* is the mechanism that
mediates all access to objects and enforces the system's access control policy. The
original Anderson Report (1972) defined the core properties. NSA Raise the Bar (RTB)
evolves these into the RAIN principle for modern Cross-Domain Solutions (CDS).

UMRS does not implement a reference monitor. The Linux kernel security subsystem is
the reference monitor. UMRS validates that the reference monitor is correctly
configured, active, and enforcing the intended policy. This distinction is
architecturally precise and operationally significant.

== The Reference Monitor Model

The reference monitor concept was introduced in the Anderson Report (1972), a study
commissioned by the U.S. Air Force Electronic Systems Division. The report defined a
mechanism that must validate each reference to programs or data against authorized
access rules. This concept became the foundation of MLS systems, TCSEC evaluation
criteria, security kernels, and modern LSM enforcement models.

The Orange Book (TCSEC) extended this model by defining the *security kernel* as
"the hardware and software that implements the reference monitor concept." Both the
reference monitor abstraction and the security kernel construct form the intellectual
lineage of the enforcement model that SELinux implements today.

=== The Three Properties

A reference monitor must satisfy three properties to be considered trustworthy:

[cols="1,3"]
|===
| Property | Meaning

| Complete mediation (non-bypassable)
| Every access request to every object must pass through the reference monitor.
There must be no alternate path that circumvents enforcement.

| Tamper resistance
| Subjects cannot modify the reference monitor.
The enforcement mechanism itself is protected from alteration.

| Verifiability
| The reference monitor must be small enough to analyze completely and validate.
A mechanism too large to analyze cannot be trusted.
|===

These three properties are the criteria against which any claimed reference monitor
implementation must be evaluated. They are also the criteria UMRS uses when validating
enforcement integrity — see xref:truth-concepts.adoc[Runtime Source of Truth].

NSA RTB evolves these properties into the RAIN principle — see <<rain-principle>> below.

=== The Security Kernel

The security kernel is the set of hardware and software that implements the reference
monitor concept. The security kernel:

* Enforces the system's access control policy on behalf of the reference monitor
* Is protected from modification by less-trusted subjects
* Is small enough to be analyzed and formally verified

In a Linux MLS system, the security kernel is the combination of the kernel proper,
the LSM framework, and the loaded security module (SELinux in UMRS deployments).

== The Reference Monitor in Linux and Where UMRS Fits

A reference monitor is a formal concept introduced in the Anderson Report (1972) and
later formalized in the Orange Book (TCSEC). It defines the minimal enforcement
mechanism that mediates all access to objects and must satisfy three properties:
complete mediation (non-bypassable), tamper resistance, and verifiability.

=== The Kernel Is the Reference Monitor

In a Linux system, the reference monitor lives inside the kernel:

The LSM hooks inside the kernel act as the reference monitor implementation. When a
process calls `open()`, `read()`, `write()`, `exec()`, `mount()`, or `ptrace()`, the
kernel invokes security hooks such as `security_inode_permission()`,
`security_file_open()`, and `security_task_create()`. Those hooks enforce the security
policy. The kernel security subsystem *is* the reference monitor.

The LSM framework satisfies the three reference monitor properties in Linux:

* *Complete mediation* — LSM hooks are placed at every security-relevant system call
  boundary in the kernel. Every access to a file, socket, IPC resource, or process
  passes through the hook chain.
* *Tamper resistance* — the LSM framework is part of the kernel. Modifying it requires
  kernel privilege and is detectable through integrity measurement.
* *Verifiability* — SELinux policy is loaded and verified at boot. The policy
  enforcement code is reviewed and audited as part of the kernel.

=== UMRS Validates the Reference Monitor

IMPORTANT: UMRS does not implement a reference monitor. The Linux kernel security
subsystem provides the reference monitor implementation. UMRS validates that the
reference monitor is correctly configured, active, and enforcing policy.

UMRS functions as a *security verification and observation system* — a runtime
enforcement validation layer. It sits alongside the kernel, not in the enforcement
path:

Specifically, UMRS:

* Verifies that the reference monitor is active and in enforcing mode
* Validates that security policy is correctly configured
* Detects configuration drift from the intended security posture
* Identifies conditions that could allow enforcement bypass
* Analyzes system security posture against compliance requirements

This role — external validation of the reference monitor — is what high-assurance
environments call *security posture validation* or *runtime enforcement verification*.
It is the kind of tooling that high-assurance systems often lack.

=== Why This Distinction Matters

Calling UMRS a reference monitor would imply that it mediates every access request,
enforces security policy directly, and sits in the kernel access path. That would not
be accurate. UMRS performs security posture validation, runtime inspection, enforcement
verification, and policy drift detection. The kernel enforces; UMRS verifies that the
enforcement is trustworthy.

The distinction also matters for auditors. A system that claims to implement a
reference monitor is making a strong architectural claim — one that requires evidence
of complete mediation, tamper resistance, and verifiability. UMRS makes a different
and equally precise claim: that it provides assurance evidence about the system that
does implement the reference monitor.

== The RAIN Principle (NSA RTB Core)

NSA RTB mandates that a security mechanism follow the RAIN concept to ensure it is
effective against persistent, advanced threats.

`R — Redundant`::
Security mechanisms must have overlapping layers. In UMRS code, this is the TPI
(Two-Path Integrity) gate where two independent parsers must agree on the label before
access is granted.

`A — Always Invoked`::
The monitor must be called for every single access attempt. By anchoring the tool to
the file descriptor, enforcement is invoked at the syscall level for every read and write.

`I — Independent Implementations`::
The redundant paths must be implemented using different logic or libraries to prevent a
single flaw from compromising both. UMRS uses a declarative (`nom`) versus imperative
(`FromStr`) parser to satisfy this.

`N — Non-Bypassable`::
There must be no "backdoor" or alternative route to the data. Using `rustix` for direct
inode-anchored calls prevents path-spoofing that would bypass standard userland checks.

== The Anderson Report Properties (The Foundation)

The reference monitor must meet these three classic design requirements to be considered "trusted":

`Tamperproof`::
The mechanism itself must be protected from modification by untrusted users or processes.
UMRS's use of memory-safe Rust and strictly typed primitives (e.g., bitmasks) ensures
the monitor's internal logic cannot be corrupted.

`Always Invoked`::
It must mediate every reference to every object. (Matches the 'A' in RAIN.)

`Small and Verifiable`::
It must be small enough to be subject to complete analysis and testing. This is why
UMRS prioritizes a minimized TCB and avoids large, untrusted libraries.

== Opening Files: Inode Anchoring and TOCTOU Safety

In high-assurance security, inodes are the ground truth, while paths (filenames) are
just pointers. "Anchoring to the inode" is the difference between a tool that can be
tricked and a tool that acts as a reference monitor.

=== The Vulnerability: Path-Based Logic

If you use a path-based call (like the standard `getxattr("/etc/passwd")`), the
following can happen in the microseconds between your security check and your data read:

1. Your code asks for the label of `/etc/passwd`.
2. An attacker renames `/etc/passwd` to `/etc/passwd.old` and places a symbolic link
   at `/etc/passwd` pointing to a malicious file they own.
3. Your code now reads the security label of the attacker's file, but thinks it is
   looking at the system password file.

This is a TOCTOU (Time-of-Check to Time-of-Use) race condition.

=== The Solution: Inode-Based Anchoring (Provenance)

When you use `std::fs::File::open` followed by `rustix::fs::fgetxattr(&file, ...)`:

*The Handle:* You obtain a file descriptor (fd). This fd is a direct pointer in the
kernel to a specific inode — the actual data blocks on the disk.

*The Invariant:* Once that fd is open, it does not matter if the file is renamed,
deleted, or replaced with a symlink. The fd is still "anchored" to that specific
physical inode.

*Provenance:* When you call `fgetxattr` on that fd, the kernel returns the security
label associated with that specific physical object. The label, the permissions, and
the content all belong to the same physical object — not to a name that could have
been swapped.

=== Why This Satisfies NIST and RTB

[cols="1,3"]
|===
| Requirement | Satisfaction

| `NIST 800-53 AC-3` (Access Enforcement)
| Mediation cannot be bypassed by filesystem manipulation.

| `NSA RTB Non-Bypassability`
| The "Security Decision Point" is moved away from the untrusted filesystem hierarchy
  (names) and into the kernel's object management (inodes).
|===

=== Application to umrs-ls

Because `umrs-ls`:

* Opens the file first.
* Passes that `File` object to `read_context`.
* Uses that same `File` object for `ioctl_getflags` (immutable bit check).

...the tool creates a high-assurance audit. It reports on the physical integrity of the
data, not just what the filename says it is.

== Historical Sources

The reference monitor concept and MLS system architecture trace to these foundational
documents:

* *The Anderson Report* (1972) — "Computer Security Technology Planning Study,"
  commissioned by the U.S. Air Force Electronic Systems Division. Introduced the
  reference monitor concept and defined the three properties: tamper resistant, always
  invoked, and small enough to verify. These properties later became the foundation for
  MLS systems, TCSEC, security kernels, and modern LSM enforcement models.

* *The Orange Book* (TCSEC) — DoD Trusted Computer System Evaluation Criteria. Explains
  that the reference monitor enforces access relationships between subjects and objects.
  Introduces the *security kernel*, defined as "the hardware and software that implements
  the reference monitor concept."

* *The Bell–LaPadula Model* (1973) — the formal model behind MLS systems, establishing
  the "no read up, no write down" rules that govern information flow enforcement.
