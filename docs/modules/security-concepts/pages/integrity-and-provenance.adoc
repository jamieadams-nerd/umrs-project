= Integrity and Provenance in UMRS
:toc: left
:description: Definitions and enforcement mechanisms for provenance, integrity, attestation, and chain of custody in the UMRS architecture.

// NIST 800-53 SI-7 — Software and information integrity
// NIST 800-53 AU-9 — Protection of audit information
// NIST 800-53 AU-10 — Non-repudiation
// NSA RTB RAIN — Non-bypassable enforcement

High-assurance systems fail most often not because individual mechanisms are wrong,
but because terms like "integrity" and "provenance" are used loosely enough to mean
different things to different people — until a failure makes the ambiguity visible.
UMRS treats these terms as engineering properties with specific definitions, each
with specific enforcement mechanisms.

== Core Definitions

=== Provenance

Provenance is the documented history of an artifact: where it originated, how it was
produced, and what transformations have been applied to it.

Provenance answers: "Where did this come from, and how did it get here?"

Provenance alone does not guarantee correctness or legitimacy. It establishes traceable
history. In high-assurance systems, provenance must be tamper-evident, attributable to
a specific identity (human, system, or device), and preserved across transformations.

In UMRS, provenance verification applies at the kernel attribute level. Before trusting
data from a kernel pseudo-filesystem — `/proc/`, `/sys/fs/selinux/` — UMRS verifies
the filesystem magic number via `statfs`. This prevents spoofing via bind-mounts or
malicious overlays (NIST 800-53 SI-7). See xref:architecture:kernel-files-tpi.adoc[Two-path
independence] for implementation details.

=== Integrity

Integrity is the assurance that data has not been altered in an unauthorized or
undetected manner.

Integrity answers: "Has this changed since it was last trusted?"

Integrity does not mean the data is correct, appropriate, or authorized for use. It
means the bits are the same as when they were last trusted, and any change is
detectable. UMRS enforces integrity at two layers:

* *IMA (Integrity Measurement Architecture)* — the Linux kernel measures file content
  hashes and records them in a tamper-visible measurement log at
  `/sys/kernel/security/ima/ascii_runtime_measurements`. IMA Appraisal blocks access
  to files whose hash no longer matches the stored policy value.

* *EVM (Extended Verification Module)* — protects file metadata (security labels,
  ownership, permissions) from offline tampering by computing an HMAC over the
  security-relevant xattrs. A tampered label breaks the HMAC and is rejected.

Together, IMA and EVM ensure that an object's content and its security label are
cryptographically bound and tamper-evident.

=== Integrity Assurance

Integrity assurance is the degree of confidence that integrity protections are
correctly implemented, enforced, and verifiable.

The distinction between "we hash files" and integrity assurance is substantial:

* The hashing mechanism must be correctly keyed (EVM HMAC key management)
* The algorithm must be FIPS-validated (NIST 800-53 SC-13)
* The enforcement boundary must be at the kernel — not in a daemon that can be stopped
* The measurement log must be integrity-protected (append-only, signed)
* The mechanism must resist bypass — loading a policy that does not cover a file is
  not the same as proving a file is trustworthy

Integrity assurance is a system property, not a property of a single mechanism.

=== Attestation

Attestation is a cryptographic or authoritative assertion by an entity that a specific
claim is true at a specific time.

Examples:

* "This file was received intact at time T."
* "This system was in configuration state Y when measured."
* "This data was produced by process Z."

Attestation answers: "I assert this is true, and I am accountable for that assertion."

Attestation is distinct from provenance. Provenance describes history. Attestation
asserts a specific fact about a specific artifact at a specific moment. In UMRS, the
IMA measurement log is an implicit attestation: it records what was seen, when, and in
what state, in a tamper-visible ledger.

=== Authenticity

Authenticity is the assurance that an artifact is genuinely what it claims to be — that
it comes from the identity it purports to come from and has not been substituted.

Authenticity answers: "Is this really from who or what it claims?"

Authenticity requires identity binding, provenance or attestation, and integrity. Courts
frequently reject digital evidence for failure to authenticate, even when integrity
is present. The distinction matters: an intact, unmodified file is not necessarily
authentic if its provenance cannot be established.

=== Non-repudiation

Non-repudiation is the ability to prevent an entity from credibly denying a prior action.

Non-repudiation is a legal and procedural property, enabled by cryptography but not
guaranteed by it alone. Requirements:

* Digital signatures with key custody controls (the signer cannot plausibly claim the
  key was not theirs)
* Trusted timestamps
* Audit records that cannot be silently altered

NIST 800-53 AU-10 (Non-repudiation) directly addresses this requirement. UMRS satisfies
it through signed audit logs and IMA/EVM measurements that tie actions to kernel-verified
identities.

=== Chain of Custody

Chain of custody is the documented and continuous control history of an artifact from
creation or acquisition to final disposition.

Chain of custody answers: "Who had control of this, when, and under what authority?"

In digital systems, chain of custody requires:

* Integrity protection of the artifact itself
* Integrity protection of the custody records
* Explicit transfer events — no silent handoffs
* A gap in the chain is a break in trust

This is where most unclassified systems fail. Logs exist, but custody records are not
themselves cryptographically protected. An attacker who can modify a log file can
reconstruct a plausible-looking custody history. IMA/EVM prevents this by making the
measurement log tamper-visible.

=== Custody Trust vs Content Trust

These concepts must be kept separate:

* *Custody trust* — "This artifact has not changed since we received it."
* *Content trust* — "This artifact is accurate, valid, or correct."

UMRS explicitly claims custody trust through IMA/EVM enforcement. It does not
automatically claim content trust. A CUI document whose IMA hash verifies is
demonstrably unaltered since it was last measured. Whether its content is accurate or
appropriate is a separate determination outside the scope of kernel enforcement.

Conflating these two properties is a source of audit failures. A system that claims
integrity without distinguishing what is actually being attested will fail under
forensic scrutiny.

=== Auditability

Auditability is the ability to reconstruct events and decisions after the fact with
sufficient fidelity to support accountability.

Auditability requires:

* Complete records (no gaps, no silent failures)
* Meaningful semantics (records must identify who, what, when, and why)
* Stable identifiers (subject and object identities must be traceable)
* Integrity protection of the audit records themselves

Audit logs that cannot answer "who, what, when, why" are compliance theater, not
auditability. NIST 800-53 AU-3 specifies the minimum content of audit records. UMRS
generates audit events at the kernel level (SELinux AVC denials, IMA measurements)
and at the application level (structured log output), all protected by signed log
chains.

== Enforcement Architecture

=== SELinux MLS: Access and Flow Control

SELinux MLS labeling controls which processes can access which objects and in which
direction information can flow. A process running at sensitivity level s4 (CUI) cannot
read objects labeled s5 (EXPORT_CONTROLLED), regardless of POSIX permissions. A process
at s4 cannot write to objects labeled s2 (CONTROLLED) — the no-write-down rule prevents
intentional or accidental downgrade of sensitive information.

This enforcement is non-bypassable (NSA RTB RAIN). The SELinux reference monitor runs
in the kernel. There is no userland path that circumvents it.

=== IMA/EVM: State and Provenance Control

IMA and EVM enforce that files are in their expected state before they are used in
any security-relevant decision. This addresses a class of failure that MLS labels
alone cannot prevent: a file with a correct label but tampered content.

*Measurement:* IMA computes a hash of file content on each access and records it in
the kernel measurement log. This log is append-only and not writable by userland.

*Appraisal:* In IMA appraisal mode, the kernel compares the current file hash against
a stored policy value. If the hash does not match, access is denied. This prevents
a tampered binary, configuration file, or policy definition from being executed.

*EVM:* EVM computes an HMAC over the security-relevant xattrs (SELinux label,
ownership, permissions) and stores it as a kernel-protected xattr. If the label is
altered offline (bypassing the running kernel), the HMAC breaks and the file is
rejected on next access.

Together, these mechanisms mean that both the label (what security attributes apply)
and the content (what the object actually contains) are cryptographically protected
and verifiable.

=== Evidence for Auditors

Administrators can generate observable evidence of enforcement at any time:

[source,shell]
----
# Confirm a file has an active IMA hash
getfattr -n security.ima <file>

# Show the tamper-visible kernel measurement log
cat /sys/kernel/security/ima/ascii_runtime_measurements

# Confirm MLS labels are applied to CUI artifacts
ls -Z /data/cui/
----

These commands produce evidence that enforcement is active and that specific objects
are covered by the integrity policy.

For a consolidated view of all security attributes on a single file — SELinux
security context, IMA/EVM flag state, encryption source, and typed findings —
use xref:umrs-tools:umrs-stat.adoc[umrs-stat].
It reads these attributes through a TOCTOU-safe fd-anchored path, surfaces them
in a structured audit card, and emits any findings as typed data that can be
counted and filtered without parsing log output.

For directory-level label inspection, xref:umrs-tools:umrs-ls.adoc[umrs-ls]
groups entries by SELinux type and security marking using the same fd-anchored
approach.

== Why This Matters in Practice

Unclassified systems fail because they treat important data like ordinary files. The
combination of MLS labeling and IMA/EVM enforcement moves security properties from
policy documents and application logic into the kernel, where they are not bypassable
by a compromised process, a misconfigured permission, or a user with elevated
privileges but insufficient clearance.

The goal is not secrecy. It is deterministic, auditable, provably enforced handling
of data whose integrity and access controls matter.
