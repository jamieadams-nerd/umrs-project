`umrs-uname` is a security overlay of the classic `uname` command.
Where `uname` reports the kernel version string, `umrs-uname` reports whether
the kernel is running in a hardened, integrity-verified state â€” and shows
the evidence behind that conclusion.

The tool runs the UMRS platform detection pipeline at startup, collects a live
snapshot of kernel security posture, and displays the results in an interactive
three-tab audit card.

.Primary audiences
* *Security operators* â€” verify system posture before CUI processing.
* *Security auditors* â€” review the evidence chain behind a trust classification.
* *Developers* â€” understand how platform detection and posture probing work in practice.

.Data source descriptions
[cols="2,4", options="header"]
|===
| Source | What `umrs-uname` reads from it

| `/etc/os-release`
| OS identity fields: `ID`, `NAME`, `VERSION_ID`, `PRETTY_NAME`, `CPE_NAME`.
  Read via `OsDetector` with fd-anchored I/O and ownership verification.
  The T4 trust tier requires this file to pass ownership and package digest verification.

| `/proc/sys/kernel/random/boot_id`
| Per-boot unique identifier.
  Displayed in Tab 0 for log correlation.

| Live sysctl nodes (`/proc/sys/â€¦`)
| Kernel hardening indicator values: ASLR level, kptr_restrict, unprivileged BPF,
  ptrace scope, perf counters, dmesg restriction, sysrq, user namespace limit, and others.
  Each node is read via a typed `StaticSource` with filesystem magic verification.
  These are the live values the running kernel is using.

| `/etc/sysctl.d/` and `/usr/lib/sysctl.d/`
| Intended configuration values.
  The posture engine compares each live indicator against its configured value.
  Disagreement produces a contradiction â€” either DRIFT, NOT PERSISTED, or UNVERIFIABLE.

| `/sys/fs/selinux/enforce` and `/sys/fs/selinux/mls`
| SELinux enforcement mode and MLS flag.
  Read via `SelinuxEnforce` and `SelinuxMls` â€” typed `StaticSource` implementations
  with sysfs magic verification.

| `/sys/kernel/security/lockdown`
| Kernel lockdown mode: none, integrity, or confidentiality.

| `/proc/sys/crypto/fips_enabled`
| FIPS mode state.

| Module-related sysctl nodes
| `kexec_load_disabled`, `modules_disabled`, `module.sig_enforce`.
  These gates determine whether the running kernel can be replaced or extended.

| RPM / dpkg database
| Platform identity facts used to compute the T3 and T4 trust tiers.
  At T3, two or more independent package database facts must agree on the platform identity.
  At T4, the os-release ownership and a package digest are both verified.

|===

NOTE: Every read from `/proc` and `/sys` passes through filesystem magic verification.
The tool cannot be deceived by a regular file mounted at a procfs or sysfs path.
This is the non-bypassability property required by NSA RTB RAIN.

[cols="1,2,4", options="header"]

| T1

== Security Design

*Non-bypassable source verification* â€” every read from `/proc` and `/sys` is preceded
by a filesystem magic number check.
The tool cannot be deceived by a bind-mounted regular file at a procfs path.
(NSA RTB RAIN, Non-Bypassability)

*Fail-closed trust* â€” if the detection pipeline fails a hard gate, the trust tier is
T0 (Untrusted).
The failure is shown in Tab 0; posture data is still collected and shown in Tab 1.

*Evidence, not assertion* â€” the trust tier is computed from the evidence chain,
not asserted by the tool.
The full evidence chain is available for review in Tab 2.

*Contradiction detection* â€” the posture engine compares every live sysctl value against
the configured value in `sysctl.d`.
Contradictions are typed data: DRIFT, NOT PERSISTED, or UNVERIFIABLE.
They can be counted, filtered, and matched programmatically.
(NIST SP 800-53 CM-6, CA-7)
