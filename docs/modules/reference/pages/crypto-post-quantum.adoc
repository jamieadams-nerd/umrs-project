= Post-Quantum Cryptography
:description: Post-quantum cryptographic algorithms — ML-KEM, ML-DSA, SLH-DSA — and their FIPS 203/204/205 standardization status.
:toc: left
:toclevels: 2

// NIST 800-53 SC-12 — Cryptographic key establishment and management
// NIST 800-53 SC-13 — Cryptographic protection
// FIPS 203, 204, 205 (2024)

Post-quantum cryptography (PQC) refers to cryptographic algorithms believed to be resistant
to attacks by sufficiently powerful quantum computers.

Classical public-key algorithms — RSA, ECDSA, ECDH — rely on mathematical problems
(integer factorization and discrete logarithm) that Shor's algorithm solves efficiently
on a quantum computer.
NIST completed standardization of three post-quantum algorithms in 2024.
These are the algorithms to use in new designs where post-quantum resistance is required.

[IMPORTANT]
====
Post-quantum algorithms are _additions_ to a classical baseline, not replacements for
existing algorithms in deployed systems.
Current guidance is to plan migration; not all environments have completed it.
For FIPS-compliant deployments today, classical algorithms remain the primary option
unless the specific FIPS 203/204/205 module is validated and available in the toolchain.
====

== Standardized Post-Quantum Algorithms

[cols="1,1,1,2", options="header"]
|===
| Algorithm | Standard | Type | Former name

| ML-KEM | FIPS 203 (2024) | Key encapsulation mechanism | CRYSTALS-Kyber
| ML-DSA | FIPS 204 (2024) | Digital signature | CRYSTALS-Dilithium
| SLH-DSA | FIPS 205 (2024) | Hash-based digital signature | SPHINCS+
|===

== ML-KEM — Key Encapsulation

ML-KEM (Module-Lattice Key Encapsulation Mechanism) establishes a shared secret between
two parties.
It replaces classical key agreement (ECDH, DH) in protocols where quantum resistance is required.

Security basis: learning-with-errors (LWE) problem over module lattices.

[cols="1,1,1,2", options="header"]
|===
| Parameter set | Security level | Public key size | Shared secret size

| ML-KEM-512 | ~128-bit (AES-128 equivalent) | 800 bytes | 32 bytes
| ML-KEM-768 | ~192-bit (AES-192 equivalent) | 1,184 bytes | 32 bytes
| ML-KEM-1024 | ~256-bit (AES-256 equivalent) | 1,568 bytes | 32 bytes
|===

*UMRS recommendation*: ML-KEM-1024 for high-assurance use.
ML-KEM-768 is the baseline for interoperability-constrained environments.

== ML-DSA — Digital Signatures

ML-DSA (Module-Lattice Digital Signature Algorithm) is the primary post-quantum digital
signature standard.
Use for signing artifacts, audit chains, and software delivery.

Security basis: module-lattice hard problems (MLWE, MSIS).

[cols="1,1,1,1", options="header"]
|===
| Parameter set | Security level | Public key size | Signature size

| ML-DSA-44 | ~128-bit | 1,312 bytes | 2,420 bytes
| ML-DSA-65 | ~192-bit | 1,952 bytes | 3,293 bytes
| ML-DSA-87 | ~256-bit | 2,592 bytes | 4,595 bytes
|===

*UMRS recommendation*: ML-DSA-87 for high-assurance signing.
ML-DSA-65 is the baseline.

== SLH-DSA — Hash-Based Signatures

SLH-DSA (Stateless Hash-Based Digital Signature Algorithm) is a conservative alternative
to ML-DSA.
Its security depends only on the underlying hash function's properties — not on
lattice-based hardness assumptions.

Use SLH-DSA when deploying environments require conservative assumptions or when
formal verification of the security basis is a requirement.

Note: SLH-DSA signatures are significantly larger than ML-DSA signatures, which may
constrain its use in size-sensitive protocols.

[cols="1,1,2", options="header"]
|===
| Parameter set | Security level | Characteristic

| SLH-DSA-SHAKE-128s / -128f | ~128-bit | Small signatures (s) or fast signing (f)
| SLH-DSA-SHAKE-192s / -192f | ~192-bit | Small signatures (s) or fast signing (f)
| SLH-DSA-SHAKE-256s / -256f | ~256-bit | Small signatures (s) or fast signing (f)
|===

== Migration Approach

For systems that must support both classical and post-quantum algorithms during transition:

. Identify the cryptographic primitives in each component (key agreement, signatures, KDF).
. For each primitive, confirm whether a FIPS 203/204/205-validated implementation is available
  in your toolchain.
. For key agreement: consider hybrid schemes that combine ECDH P-384 with ML-KEM-768 or
  ML-KEM-1024 — this preserves classical security while adding post-quantum resistance.
. For signatures: prefer ML-DSA-87 for new signing infrastructure.
  Keep existing ECDSA P-384 infrastructure active until all verifiers are updated.
. Document all migration decisions as control evidence under NIST 800-53 SC-12.

== Control Mapping

[cols="1,3", options="header"]
|===
| Control | Satisfaction

| NIST 800-53 SC-12
| Key establishment for ML-KEM follows FIPS 203.
ML-KEM-1024 provides key agreement meeting the highest NIST-defined security level.

| NIST 800-53 SC-13
| Cryptographic protection.
ML-DSA-87 and SLH-DSA are NIST-standardized algorithms under FIPS 204/205.
Use of standardized algorithms with documented parameter choices satisfies SC-13.
|===
