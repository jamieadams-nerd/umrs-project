= Sensitivity Levels
:description: SELinux sensitivity levels — hierarchy, Bell-LaPadula rules, and UMRS type model.
:toc: left

// NIST 800-53 AC-3 — Access Enforcement
// NIST 800-53 AC-4 — Information Flow Enforcement
// Kernel sources: security/selinux/ss/mls.c, mls.h, policydb.c

The sensitivity level is the hierarchical component of an SELinux security context.
It represents a classification tier within the Multi-Level Security (MLS) model.
Every subject (process) and object (file, socket, IPC resource) carries a sensitivity level.
Access decisions compare levels to enforce the Bell-LaPadula rules.

== Format

Sensitivity levels are written with an `s` prefix followed by a numeric ordinal:

```
s0   s1   s2   ...   s15
```

`s0` is the lowest level.
`s15` is the highest.
The exact hierarchy — which level corresponds to which classification tier — is defined in the policy.

In UMRS, sensitivity levels map to the unclassified data classification tiers.
See xref:reference:selinux/umrs-mls-registry.adoc[UMRS MLS registry] for the configured mapping.

== Bell-LaPadula Access Rules

Sensitivity levels enforce two mandatory access rules derived from the Bell-LaPadula model:

[cols="1,3", options="header"]
|===
| Rule | Definition

| *No-read-up*
| A subject at level `sN` cannot read objects at level `sN+1` or higher.
A process at `s2` cannot read a file labeled `s3`.

| *No-write-down*
| A subject at level `sN` cannot write to objects at level `sN-1` or lower.
A process at `s2` cannot write to a file labeled `s1`.
|===

These rules enforce confidentiality.
Read-up would allow information disclosure to an insufficiently cleared subject.
Write-down would allow information to flow from a higher classification tier to a lower one.

NOTE: SELinux enforces these rules at the kernel level through the reference monitor.
They are not advisory — a violation produces a denial, not a warning.

See xref:security-concepts:reference-monitor.adoc[Reference Monitor Design in UMRS].

== MLS Level Structure

When a sensitivity level is combined with a category set, the result is an MLS level:

```
sensitivity_level : category_set
```

Example: `s2:c0,c4` — sensitivity level 2 with categories 0 and 4.

The level combined with a high/low pair forms an MLS range.
See xref:glossary:index.adoc#_mls_range[Glossary: MLS Range].

== UMRS Type Model

UMRS models sensitivity levels as a distinct validated Rust type: `SensitivityLevel`.

[source,text]
----
SensitivityLevel(u16)

MAX_SENSITIVITY: u16 = 15
Valid range: s0 through s15
----

Validation rules enforced at construction time:

* Input must start with the character `s`
* Remainder must be a valid decimal integer
* Value must fall within `0..=15`

A `SensitivityLevel` cannot be constructed with an invalid string.
If validation fails, an error is returned at the construction boundary.
Downstream code receives a valid `SensitivityLevel` or nothing at all.

The type implements `PartialOrd` and `Ord`.
This enables direct comparisons in dominance checks and clearance evaluations
without converting back to strings.

[source,rust]
----
// Parse from the sensitivity component of an MLS context string
let level: SensitivityLevel = "s3".parse()?;

// Dominance comparison — used in MLS access decisions
assert!(s3 > s2);    // s3 dominates s2
assert!(s2 < s3);    // s2 does not dominate s3
----
