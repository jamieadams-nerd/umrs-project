= SELinux Role
:description: SELinux role identifier — RBAC layer, domain authorization, and UMRS type model.
:toc: left

// NIST 800-53 AC-3 — Access Enforcement
// NIST 800-53 AC-6 — Least Privilege
// Kernel sources: security/selinux/ss/policydb.c, security/selinux/include/security.h

The SELinux role is the second component of a security context.
It is the core of the Role-Based Access Control (RBAC) layer in SELinux.
Roles act as an intermediate between users and types:
a user can only access the types (domains) that their current role authorizes.

== Function

Roles serve two functions in the SELinux access model:

. *Domain authorization* — a role defines the set of process domains (types) that
  a user running in that role is permitted to enter.
  A user cannot transition into a domain that their role does not authorize.
. *Privilege escalation barrier* — a user cannot jump directly from a low-privilege domain
  to a sensitive domain unless their role permits that transition.
  Role transitions are explicit and policy-controlled.

== Naming Convention

SELinux role identifiers end with `_r`.

[cols="1,3", options="header"]
|===
| Identifier | Description

| `system_r`
| System services role.
Processes started at boot or by the init system run in this role.

| `staff_r`
| Confined staff role.
Permits transitions to more privileged roles such as `sysadm_r` via `sudo` or `newrole`.

| `sysadm_r`
| System administrator role.
Authorized for high-privilege domains including administrative tools.

| `user_r`
| Least-privileged user role.
Restricted to standard application domains.
Cannot transition to administrative domains.

| `object_r`
| Placeholder role for files and passive objects.
Files are not active principals; `object_r` satisfies the syntactic requirement
for a role field without granting any active privileges.
|===

== UMRS Type Model

UMRS models the SELinux role as a distinct validated Rust type: `SelinuxRole`.

[source,text]
----
SelinuxRole(String)

Valid identifiers: system_r, staff_r, object_r
MIN_ROLE_LEN: 3 bytes
MAX_ROLE_LEN: 255 bytes
----

Validation rules enforced at construction time:

* ASCII characters only
* No whitespace
* Character set: `[a-z0-9_]` (lowercase letters, digits, underscore)
* Must end with `_r`
* Stem (identifier minus the `_r` suffix) must be non-empty
* Length within bounds: 3–255 bytes

A `SelinuxRole` cannot be constructed with a malformed identifier.
Construction succeeds and returns the typed value, or it fails and returns an error.

[source,rust]
----
let role: SelinuxRole = "staff_r".parse()?;

// Invalid — wrong suffix
let err = SelinuxRole::new("staff_u");
assert!(err.is_err());

// Invalid — uppercase letters not permitted
let err = SelinuxRole::new("Staff_r");
assert!(err.is_err());
----

=== Policy Associations (not modeled in this type)

This type models only the identifier primitive.
The following are defined in policy, not in this type:

* Authorized domain (type) sets per role
* Role transition rules
* User-to-role authorization mappings
