= Compliance Frameworks Registry
:description: Authoritative list of governing standards, their versions, and their role in UMRS design and implementation.
:toc: left
:toclevels: 2

This page lists each standard or framework that governs UMRS design,
implementation, and deployment. For each, it states the version in use,
and the specific role that standard plays in the project.

Control citations in the source code (`// NIST 800-53 AC-4`,
`// NSA RTB RAIN`) refer to the frameworks listed here. See
xref:devel:compliance-annotations.adoc[Compliance annotations guide]
for how to read and write those citations.

== Governing Standards

[cols="1,1,3"]
|===
|Standard |Version |Role in UMRS

|NIST SP 800-53
|Revision 5
|The primary security and privacy control catalog. Drives control
 citations throughout the codebase. Key controls: AC-3 (access
 enforcement), AC-4 (information flow enforcement), AU-3 (audit record
 content), MP-4 (media protection / zeroize), SC-28 (data at rest),
 SI-7 (software and information integrity).

|NIST SP 800-171
|Revision 2
|CUI protection requirements for non-federal systems. Scopes 800-53
 to the CUI use case. Drives the CUI labeling model and handling
 requirements that UMRS enforces.

|NIST SP 800-218 (SSDF)
|Version 1.1
|Secure Software Development Framework. Drives high-assurance
 implementation patterns: TPI parsing (PW.4), provenance verification
 (PW.4), secure arithmetic (PW.4.1), and supply chain hygiene (PO.1.2,
 PW.4.3).

|CMMC 2.0
|Level 1–3
|Cybersecurity Maturity Model Certification. Operationalizes 800-171
 requirements as a maturity assessment. Level 2 maps directly to
 800-171; Level 3 adds select 800-53 controls. UMRS design targets
 Level 2 as the minimum baseline.

|NSA RTB VNSSA / RAIN
|Current NSA guidance
|NSA Requirements and Test Benchmarks. RAIN (Reference Architecture
 and Implementation Notes) provides the non-bypassability requirement:
 security checks must always be invoked. Drives private constructors,
 module boundaries, TOCTOU-safe I/O, and fail-closed error handling
 throughout the codebase.

|FIPS 140-2 / 140-3
|Active standard
|Cryptographic module validation standard. FIPS mode is assumed active
 in all production deployments. Any cryptographic primitive used in
 UMRS must be a FIPS-validated algorithm. Drives the prohibition on
 vendoring OpenSSL and the preference for FIPS-compatible Rust crates.

|RHEL 10 STIG / SCAP Security Guide
|scap-security-guide 2026-03-17
|The Security Technical Implementation Guide (STIG) for Red Hat
 Enterprise Linux 10, expressed as machine-readable SCAP content.
 SCAP (Security Content Automation Protocol) is a NIST-standardized
 framework for automated configuration assessment. The STIG provides
 human-readable security guidance; the SCAP profile translates that
 guidance into a set of automated checks, each identified by a
 Common Configuration Enumeration (CCE) identifier of the form
 `CCE-NNNNN-N`. UMRS uses CCE identifiers to link posture indicators
 in the catalog to the authoritative STIG check that governs the same
 setting. Where a direct mapping exists, the `IndicatorDescriptor.cce`
 field carries the CCE identifier, enabling automated compliance tooling
 to correlate UMRS assessment results with STIG findings. UMRS includes
 indicators that exceed STIG baseline coverage — those indicators
 carry `cce: None`. See xref:devel:compliance-annotations.adoc#_cce_citation_format[CCE Citation Format]
 for how CCE identifiers are used in source code and documentation.
|===

== CMMC Level Structure

CMMC 2.0 is not simply a subset of NIST SP 800-53. It is a certification mechanism that
enforces specific requirements tailored to defense contractors protecting FCI (Federal
Contract Information) and CUI.

[cols="1,2,3", options="header"]
|===
| Level | Requirement basis | Scope

| Level 1 — Foundational
| 15 practices from FAR 52.204-21
| Basic safeguarding of FCI. Self-assessment permitted.

| Level 2 — Advanced
| 110 practices aligned with NIST SP 800-171 Rev 2
| Protection of CUI. Third-party assessment required for prioritized programs.
  This is UMRS's minimum target baseline.

| Level 3 — Expert
| 110 practices from NIST SP 800-171 plus enhanced requirements from NIST SP 800-172
| Advanced persistent threat (APT) resistance for highest-priority programs.
  Government-led assessment required.

|===

CMMC does not mandate specific programming languages or implementation techniques.
It requires demonstrated process discipline and evidence of compliance — documented
code reviews, automated scanning, version-controlled configurations, and secure SDLC
practices are the categories assessors evaluate.

=== CMMC Coding-Level Focus Areas

The areas of CMMC Level 2 most directly relevant to software development are:

*Configuration Management (CM).* Code must be version-controlled, configurations
documented, and build processes auditable. Baselines must be established and drift
detected.

*System and Information Integrity (SI).* Known vulnerabilities must be patched.
The development environment must be scanned. Code must be reviewed for flaws before
promotion.

*Secure Development Life Cycle (SDLC) evidence.* Assessors expect evidence that
security is embedded in the development process: static analysis results, documented
code review records, and test outcomes.

The SSDF (NIST SP 800-218) is the DoD's preferred framework for specifying how
these CMMC obligations are met at the implementation level.

=== PW.4 Secure Coding Practices — UMRS Mapping

PW.4 is the practice with the most direct impact on source code. The sub-tasks map to
specific Rust mechanisms enforced in this codebase.

[cols="1,2,2,2", options="header"]
|===
| Sub-task | Goal | Rust mechanism | UMRS implementation

| PW.4.1
| Validate all input from untrusted sources
| Parse into typed structs at the boundary (`TryFrom`, `FromStr`).
  "Parse, don't validate" — if construction succeeds, the value is proven valid.
| `SecurityContext`, `CategorySet`, `SensitivityLevel` constructors all return `Result`.
  Downstream code receives a validated type, not a raw string.

| PW.4.2
| Encode output; do not reflect sensitive data to callers
| Separate what goes to audit logs from what is returned in errors.
  Variable data goes to `log::error!`; caller-visible errors describe the category only.
| `SecureReader::execute_read()` — path is logged, not returned in the error string.

| PW.4.3
| Prevent memory corruption: buffer overflows, use-after-free
| `#![forbid(unsafe_code)]` in every crate root — a compile-time proof, not a policy.
  Rust's ownership model and borrow checker enforce safety without runtime overhead.
| Every crate root carries `#![forbid(unsafe_code)]`. No unsafe blocks in the workspace.

| PW.4.4
| Handle integer overflows and underflows
| `.checked_add()`, `.checked_sub()`, `.checked_mul()` — return `Option`, forcing
  explicit handling. `.get(i)` instead of `buf[i]` for security-relevant indexing.
  `subtle::ConstantTimeEq` for credential comparisons.
| `CategorySet` bit operations; `SensitivityLevel` arithmetic.

| PW.4.5
| Use approved cryptographic modules and algorithms
| FIPS 140-2/3 validated implementations only. Algorithm choice is necessary but
  not sufficient — the implementing module must carry a CMVP certificate number.
| FIPS mode assumed active on all deployments. `ProcFips` is a first-class kernel
  attribute in `umrs-platform`. See xref:cryptography:fips-cryptography-cheat-sheet.adoc[FIPS cryptography reference].

|===

See xref:devel:compliance-annotations.adoc[Compliance annotations guide] for how these
sub-tasks translate to rustdoc annotation practice.

== How These Standards Interact

NIST 800-53 provides the control catalog — a comprehensive set of
security and privacy controls organized by family (AC, AU, SC, SI,
etc.). NIST 800-171 scopes that catalog to CUI environments, selecting
the subset of 800-53 controls that apply to non-federal systems handling
CUI. CMMC 2.0 takes the 800-171 requirements and structures them as
a tiered maturity model that organizations can be assessed against.

SSDF operates orthogonally to the control catalog. Where 800-53 and
800-171 define what security properties a system must have, SSDF defines
how software must be developed to achieve and maintain those properties.
SSDF practices (PW.4, PO.1.2) drive the high-assurance coding patterns
in this codebase — not the control citations, but the implementation
discipline that makes those controls verifiable.

NSA RTB adds constraints that go beyond what 800-53 requires. The RAIN
non-bypassability requirement, in particular, is stricter than AC-3 or
AC-4 alone: it requires that security enforcement be structurally
impossible to skip, not merely prohibited by policy. UMRS implements
this through Rust's type system — private constructors and module
visibility rules prevent callers from bypassing validation at the
language level.

FIPS 140-2/140-3 constrains the cryptographic primitive layer
independently of all other frameworks. A system can satisfy AC-4 and
800-171 while using non-FIPS cryptography; FIPS compliance is a
separate requirement imposed by the deployment environment.
