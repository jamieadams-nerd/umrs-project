= Cryptographic Key Recommendations
:toc: left
:description: Recommended cryptographic key types for a high-assurance MLS system, with artifact paths and purpose definitions.

// NIST 800-53 AU-9, AU-10 — Protection of audit information
// NIST 800-53 SC-12 — Cryptographic key establishment and management
// NIST 800-53 SI-7 — Software and information integrity

A high-assurance MLS system requires distinct cryptographic keys for distinct
purposes. Sharing keys across functions conflates security boundaries and complicates
audit accountability. The following key types are recommended as a defensible minimum.

== 1. Audit Log Signing Key

_Scope: auditd output only._

This key signs binary audit records generated by the Linux Audit Framework (`auditd`).
It does not cover journald, syslog, application logs, or kernel messages outside the
audit framework.

.Typical artifact paths
* `/var/log/audit/audit.log`
* `/var/log/audit/audit.log.*` (rotated)
* `/var/log/audit/*.log`
* Data streams produced by `audisp` modules before off-host transfer

.Purpose
* Ensure audit trail integrity
* Detect tampering after generation
* Preserve forensic reliability

.Why a dedicated key is required
AU-9 and AU-10 require audit data to be independently protected. Audit logs must not
share keys with application or system logs.

NOTE: "Audit signing keys are used exclusively to sign files produced by `auditd`
under `/var/log/audit` and are not shared with any other logging or system function."

== 2. System and Application Log Signing Key

_Scope: non-audit operational logs._

This key covers application logs, service logs, daemon logs, and custom application
logging — everything under `/var/log` that is not produced by `auditd`.

.Typical artifact paths
* `/var/log/messages`
* `/var/log/secure`
* `/var/log/maillog`
* `/var/log/httpd/*`
* `/var/log/nginx/*`
* `/var/log/<application_name>/*`
* `/var/log/journal/*` (only if exporting to files)

.Important distinction
journald internally seals its own journal files. That internal sealing is not
equivalent to external cryptographic signing. If journal files are exported,
archived, or moved off-host, this key applies.

.Excluded
* `/var/log/audit` (covered by the audit log signing key)
* Archived bundles (covered by the archive signing key below)

.Purpose
* Detect tampering with operational logs
* Provide integrity prior to archival or forwarding
* Support incident response and troubleshooting trust

NOTE: "System and application log signing keys protect non-audit operational logs
generated under `/var/log`, excluding files owned by `auditd`."

== 3. Log Archive Signing Key

_Scope: post-rotation and bundled artifacts._

This key signs bundles or compressed sets of logs prepared for long-term storage
or transfer. It signs the container artifact, not the individual log files within it.
Prior per-file signatures must remain verifiable alongside the archive signature.

.Typical artifact paths
* `/var/log/archive/*`
* `/var/log/archives/*`
* `/var/backups/logs/*`
* `/srv/log-archive/*`
* Temporary artifacts: `*.tar.gz`, `*.xz`, `*.zip` produced by logrotate hooks

.Purpose
* Preserve chain of custody
* Prove archive integrity after leaving the host
* Support long-term forensic retention

.Why this key is separate from per-log keys
Rotation cadence differs from per-file signing. Archive lifetime is often much longer.
Private key exposure risk differs between operational signing and archival signing.

NOTE: "Archive signing keys protect packaged log artifacts prepared for off-system
storage or retention and are distinct from per-log signing keys."

== 4. Integrity Baseline Signing Key

_Scope: file-integrity and MLS system trust._

This key signs integrity baselines, file hashes, and expected-good state definitions.
It does not cover logs or dynamic runtime data.

.Typical artifact paths for IMA/EVM
* `/etc/ima/keys/*`
* Measurement data under `/sys/kernel/security/ima/`
* Hashes of `/bin`, `/sbin`, `/usr/bin`, `/usr/sbin`, `/etc`

.Typical artifact paths for custom baseline tools
* `/var/lib/integrity-baseline/*`
* `/etc/integrity/*`
* `/var/lib/aide/*`

.Purpose
* Detect unauthorized modification of binaries or configuration
* Satisfy NIST 800-53 SI-7 and CM integrity requirements
* Anchor MLS trust decisions to a verified baseline

NOTE: "Integrity signing keys protect cryptographic baselines used to validate the
system's executable and configuration state and are not used for logging or archival
purposes."

== 5. Secure Boot and Platform Trust Keys

_Scope: bootloaders, kernel images, kernel modules. Usually out of scope for
application-level key tooling._

.Typical artifact locations
* UEFI firmware databases (`db`, `dbx`)
* `/boot/efi/*`
* `/boot/vmlinuz-*`
* `/boot/initramfs-*`

These keys are usually manufacturer- or organization-managed. They establish the
chain of trust from firmware through kernel boot and are managed separately from
OS-level signing keys.

== 6. TLS and Service Authentication Keys

_Scope: inventoried cryptographic material, not log-signing._

.Typical artifact paths
* `/etc/pki/tls/private/*.key`
* `/etc/pki/tls/certs/*.crt`
* `/etc/ssl/private/*`

TLS private keys provide confidentiality and authentication for service sessions.
They do not protect log integrity or audit integrity and must not be reused for
those purposes.

== 7. Package and Artifact Verification Keys

_Scope: supply-chain integrity, consumer-side verification._

.Typical artifact paths
* `/etc/pki/rpm-gpg/*`
* `/etc/yum.repos.d/*.repo`

These keys validate external software artifacts. They are not used for
system-originated signing.

== Minimum Required Key Set

For a defensible high-assurance MLS deployment, the following four keys are
the minimum:

[cols="1,3"]
|===
| Key | Scope

| Audit log signing key | `/var/log/audit/*`
| System/application log signing key | `/var/log/*` (non-audit)
| Log archive signing key | `/var/log/archive/*`, `/srv/log-archive/*`
| Integrity baseline signing key | Required if integrity monitoring is enabled
|===

Platform trust, TLS, and package verification keys are inventoried but not
necessarily generated by the UMRS key management tooling.
