Boot id

The Linux Boot ID is one of the more elegant pieces of identity metadata in the modern Linux runtime environment. It plays a particularly important role in systemd-based logging, especially within systemd-journald, because it provides a unique identifier for each boot session of the kernel. That lets journald reliably group log entries belonging to a specific boot even when logs are persistent across reboots.

Let’s walk through it systematically.

⸻

Linux Boot ID

1. What the Boot ID Is

The Boot ID is a 128-bit UUID that uniquely identifies a single boot instance of the Linux kernel.

Example:

$ cat /proc/sys/kernel/random/boot_id
3c0e9b4c-9c2e-4e2b-9b6e-5b9c61c49c0f

Key properties:
	•	UUID (128 bits)
	•	Generated once per boot
	•	Constant for the entire runtime of that boot
	•	Changes on every reboot

So if a system boots five times, there will be five different boot IDs.

⸻

2. Where It Is Stored

Primary kernel interface:

/proc/sys/kernel/random/boot_id

This is a procfs pseudo-file generated by the kernel.

Important characteristics:

Property	Description
Filesystem	procfs
Generated by	Linux kernel
Lifetime	Valid only for current boot
Mutability	Read-only
Format	UUID

It is not stored on disk anywhere by default.

Instead:
	•	It is generated during kernel initialization
	•	It remains in kernel memory for the entire runtime
	•	It disappears on reboot

⸻

3. Kernel Origins

The boot ID comes from the kernel’s random subsystem.

Implementation area:

kernel/random.c

During kernel startup:
	1.	The random subsystem initializes
	2.	The kernel generates a UUID
	3.	The value is stored in a kernel variable
	4.	It is exposed via:

/proc/sys/kernel/random/boot_id

This is handled through the sysctl interface.

⸻

4. Why Boot ID Exists

Before boot IDs, logs had a major problem:

If logs were persistent across boots, it was hard to tell which messages belonged to which boot.

Example problem with classic syslog:

Apr 10 10:10 kernel: network up
Apr 10 10:12 sshd started
Apr 10 10:15 system crash
Apr 10 10:16 kernel: network up

You can’t reliably determine:
	•	where one boot ended
	•	where the next began

Boot IDs solve this.

⸻

5. How journald Uses Boot ID

systemd-journald stores the boot ID in the field:

_BOOT_ID

Example:

journalctl -o verbose

You will see:

_BOOT_ID=3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f

Every log entry produced during a boot carries this value.

This allows journald to:

Group logs by boot

journalctl --list-boots

Example output:

-2  3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f  Tue 10:00
-1  7a9b8c4a3d2e4c6a9d5b7f1a2c4e5f6d  Tue 10:45
 0  1a2b3c4d5e6f7081920a0b0c0d0e0f10  Tue 11:12

Each entry corresponds to a unique boot ID.

You can inspect a specific boot:

journalctl -b -1

or

journalctl _BOOT_ID=3c0e9b4c9c2e4e2b9b6e5b9c61c49c0f


⸻

6. Relationship to Machine ID

The boot ID is often confused with the machine ID, but they serve different purposes.

Identifier	Lifetime	Purpose
Machine ID	Persistent	System identity
Boot ID	Per boot	Runtime session identity

Machine ID location:

/etc/machine-id

Example:

8de277067b3544d4b65c267d0edab928

This remains constant across boots.

A journald log entry usually contains both:

_MACHINE_ID
_BOOT_ID

So journald can answer:
	•	which machine produced the event
	•	during which boot session

⸻

7. Why This Matters Operationally

Boot IDs are extremely valuable for forensics and debugging.

Example use cases:

Kernel crash analysis

You can isolate the previous boot:

journalctl -b -1

Incident investigation

Determine events before reboot:

journalctl _BOOT_ID=<previous_boot>

System monitoring

Identify reboot frequency.

Correlation

You can correlate:
	•	kernel messages
	•	systemd unit starts
	•	application logs

within a single boot context.

⸻

8. Boot ID in Other Tools

Many components read the boot ID.

Examples:

systemd

sd_id128_get_boot()

C API.

⸻

Rust (systemd crate / manual read)

You can retrieve it simply:

let boot_id = std::fs::read_to_string(
    "/proc/sys/kernel/random/boot_id"
)?;


⸻

Kubernetes / containers

Boot ID is sometimes used to:
	•	detect host reboot
	•	reset runtime state

⸻

9. Security Properties

Boot ID is not cryptographic.

Important to understand:

Property	True
Unique per boot	Yes
Random	Yes
Cryptographically secure	No guarantee
Persistent	No

It is not suitable as a secret or authentication token.

However it is very useful as a runtime correlation identifier.

⸻

10. Interesting Edge Cases

Containers

Containers share the host boot ID because /proc is usually shared.

So inside Docker:

cat /proc/sys/kernel/random/boot_id

returns the host boot ID.

⸻

Kexec

If a system reboots via kexec, a new boot ID is generated.

⸻

Virtual machines

A reboot inside the VM generates a new boot ID just like physical hardware.

⸻

11. Why This Is Elegant

The design is clever because:
	•	kernel generates it once
	•	extremely cheap
	•	globally accessible
	•	immutable for the boot
	•	automatically integrated into journald

This makes it an ideal session identifier for the operating system runtime.

⸻

12. Why You Should Care (for UMRS)

Given the architecture you’re building around journald ingestion and event correlation, Boot ID is extremely valuable.

You can treat it as:

Runtime Event Epoch

It naturally partitions logs into:

machine_id
    └── boot_id
            └── events

Which is a very clean hierarchy for log correlation engines.
