The Reference Monitor Model

The Reference Monitor is a formal concept introduced in:
	•	the Anderson Report (1972)
	•	later formalized in Orange Book (TCSEC) security architecture
	•	used heavily in MLS systems

It defines the minimal enforcement mechanism that mediates all access to objects.

Formal definition

A Reference Monitor is a mechanism that enforces an access control policy and must satisfy three properties:

Property	Meaning
Complete mediation (non-bypassable)	Every access request must go through it
Tamper resistance	Subjects cannot modify it
Verifiability	It must be small enough to analyze and validate

These three properties are fundamental.

⸻

What Is the Reference Monitor in Linux?

In a Linux system the reference monitor is inside the kernel.

Specifically:

Linux Kernel
   ↓
Linux Security Module (LSM) framework
   ↓
SELinux / AppArmor / Smack

The LSM hooks inside the kernel act as the reference monitor implementation.

When a process tries to:

open()
read()
write()
exec()
mount()
ptrace()

the kernel calls security hooks such as:

security_inode_permission()
security_file_open()
security_task_create()

Those hooks enforce the security policy.

Therefore:

The kernel security subsystem is the reference monitor.

What UMRS Is (More Precisely)

UMRS is not the reference monitor.

UMRS is a security verification and observation system.

More precisely, UMRS functions as:

Runtime Security Inspection System

or

Security Enforcement Validation System

It inspects the behavior and integrity of the reference monitor.

How You Should Describe UMRS

The safest and most accurate language would be something like:

UMRS does not implement a reference monitor.
The Linux kernel security subsystem provides the reference monitor implementation.

UMRS validates that the reference monitor is correctly configured, active, and enforcing policy.

Why This Distinction Matters

Calling UMRS a reference monitor would imply that it:
	•	mediates every access request
	•	enforces security policy directly
	•	sits in the kernel access path

That would not be accurate.

UMRS is instead performing:

security posture validation
runtime inspection
enforcement verification
policy drift detection

If You Want a Strong Architecture Statement

You could include something like this:

Reference Monitor Integrity

The Linux kernel security subsystem implements the system’s reference monitor.
This mechanism mediates access to system resources and enforces security policy.

UMRS does not replace or duplicate this enforcement mechanism.
Instead, UMRS verifies that the reference monitor is:
	•	active
	•	enforcing policy
	•	correctly configured
	•	resistant to bypass conditions

By validating these properties, UMRS provides assurance that the system’s security enforcement mechanisms remain trustworthy.

1. The Anderson Report (1972) — The Origin of the Reference Monitor

This is the original document that introduced the reference monitor concept.

Direct PDF download

Download the Anderson Report (Computer Security Technology Planning Study)

This report was commissioned by the U.S. Air Force Electronic Systems Division and is widely known as “The Anderson Report.”

It introduced the concept that:

a mechanism must validate each reference to programs or data against authorized access rules.  

And it defined the three famous properties:
	1.	Tamper resistant
	2.	Always invoked
	3.	Small enough to verify  

Those properties later became the foundation for:
	•	MLS systems
	•	TCSEC
	•	security kernels
	•	modern LSM enforcement models.

⸻

2. The Orange Book (TCSEC)

This is the famous DoD Trusted Computer System Evaluation Criteria.

Direct PDF

Download the Trusted Computer System Evaluation Criteria (Orange Book)

The Orange Book explains that the reference monitor enforces access relationships between subjects and objects and became the central concept for secure system design.  

It also introduces the concept of the security kernel, defined as:

the hardware and software that implements the reference monitor concept.  

This is the exact terminology used throughout MLS system design.
