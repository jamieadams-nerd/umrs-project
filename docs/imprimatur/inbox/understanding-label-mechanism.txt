Many years ago, people realized something important about computer systems: security couldn’t rely entirely on humans making the right decisions. System owners could define rules about who should be allowed to see certain data, but if enforcement was left up to users, mistakes were inevitable. Files would be shared incorrectly, access would be granted too broadly, and sensitive information could be exposed—either unintentionally due to errors and oversight, or intentionally through misuse or malicious activity.

So the idea emerged that instead of trusting people to enforce the rules, the system itself should enforce the policy defined by its owners. The operating system—specifically the kernel, the core component that controls everything)—would take responsibility for making sure those rules were followed every single time, without exception.

They didn’t replace existing systems to do this. Instead, they added a security subsystem that could be enabled to strengthen enforcement. That’s where the name comes from: Security-Enhanced Linux. It’s not a separate operating system—it’s an enhancement that allows the system to enforce security policy in a consistent, non-bypassable way. Not every system adopted it, but where it is used, it provides a much stronger model of control.

Now, to understand how this works, you have to think about how both data and users are described inside the system. Every piece of data—like a file—has a security label. Every user has a clearance, and processes—like a web server or application—also run with a clearance because they act on behalf of users. The system compares these to decide what is allowed.

A good way to picture this is a family and a refrigerator.

Imagine a mom, a dad, and a couple of kids. Inside the refrigerator are several shelves. Each shelf represents a different sensitivity level. The bottom shelf is level zero (s0), the next one up is level one (s1), and the top shelf is level two (s2).

On each shelf are Tupperware containers. Each container has a label on it—just like a filename—and you can see that label from the outside. You don’t know what’s inside unless you’re allowed to open it, but the label itself is visible. In a real system, it’s the same way: when you list files in a directory, you can actually see the security label attached to each file right alongside its name.

Now each person in the family has a clearance. defines the shelves they’re allowed to access.

The kids can only access the bottom shelf (s0). They cannot see anything above it.
Dad can access the middle shelf (s1), and because he’s higher, he can also see down to the kids’ shelf (s0)—but he cannot see up to mom’s shelf (s2).
Mom, at the top level (s2), can access her shelf and see everything below it—both dad’s (s1) and the kids’ shelves (s0).

So access flows downward, not upward. You can always see down, but you can’t see up. So far, that’s just sensitivity levels.

But now we add another dimension.

Each person is also assigned certain categories, this collection of assigned  categories is their category set, which is part of their clearance. 

Let’s say the containers are color-coded:

Red lids mean chicken (c0).
Blue lids mean vegan (c1).
Green lids mean vegetarian (c2).

These colors represent categories, and in the system they’re stored as simple identifiers like c0, c1, c2—often as compact bit sets so many categories can be tracked efficiently.

Now instead of thinking about food preferences, think of it this way: each child is assigned to specific categories they are allowed to eat.

The first child can eat most things, but not allowed to eat his brother’s vegan food.  So he is assigned the chicken (c0) and veggie (c2) categories.

The next child is assigned to the vegan (c1) and veggie (c2) categories because he does not eat meat.

Those category memberships define what they are allowed to access. That’s their category set, and it’s part of their overall clearance.

So even though both kids are at the same sensitivity level—both can access the bottom shelf (s0)—they don’t have access to the same containers.

The first child can open red (c0) and green (c2) containers, but not blue (c1).
The second child can open blue (c1) and green (c2) containers, but not red (c0).

They can both access the green containers (c2) because they both belong to the veggie category (c2).

Now every container also has a security label. That label includes:
•	Its sensitivity level (which shelf it’s on, like s0, s1, s2)
•	Its category (its color, like c0, c1, c2)

And just like in a real system, that label is visible—you can see it, but you still need the right clearance to access what’s inside.

So when someone tries to open a container, the system does two checks.

First, it checks the sensitivity level:
Are you allowed to access this shelf (sensitivity level). Is the shelf’s level <= Person’s clearance level. Dad wants steal from the kids:  is s0 <= s1, yes. 

That’s the key idea:
Clearance (on the person or process) is compared to the security label (on the data).

And that model is completely flexible.

You can define s0 through s15 sensitivity levels you want. An individual categories are from c0 to c1024.  If a category set only had one member there is a possibility of 1024 category sets.  

You can assign users however you want. But careful planning is imperative site security policy personnel clearances all should be taken account and have proper procedures before assigning people access to the system.

The system doesn’t care what what textual name was assigned to the categories or study level they mean—it just enforces the rules based on the sensitivity level, S number and the combination of C numbers. 

When you bring this into the real world, especially in systems like Security-Enhanced Linux, those shelves and colors become real information domains.

Instead of food, you’re dealing with things like law enforcement data, agriculture data, investigative records, or other controlled information. A file is assigned a security label. A user—or a process acting on behalf of that user—is given a clearance.

In many unclassified environments, you might effectively operate at a single sensitivity level—one shelf—but you still use categories to separate different kinds of information with different handling requirements. Handling retirement could be something silly like do not do not share this with a dog

For example, a police officer might be able to access general department data, but not investigative files. Those investigative files are simply a different category for which he is not a member. Unless the officer’s clearance includes that category (c1), they can see the label, but they cannot access the contents.

What makes this powerful is the scale. As said earlier, the system  can be configure with up to 15 sensitivity levels and  1,024 individual categories.  Internally, the system stores those categories efficiently (called the access vector cache) so the Kernel can make fast, reliable decisions every time access is attempted.

But the hard part isn’t the mechanism—it’s the design. You have to carefully decide what your categories are, how they map to real-world data like Controlled Unclassified Information, and who should be assigned to which clearances.

Once that’s done, the system takes over.

A file is labeled.
A user attempts access.
The system compares clearance to label.

If they match the rules, access is allowed.
If they don’t, it’s denied.

No guessing. No exceptions. No relying on someone to do the right thing.

And that’s really the whole idea.

You don’t need to get lost in all the terminology around Security-Enhanced Linux to understand it. At its core, it’s just this:

Give users a clearance.
Give data a label.
Let the system enforce the rules.

Enforcement aside,  just knowing what’s in the containers by the label is helpful enough.
