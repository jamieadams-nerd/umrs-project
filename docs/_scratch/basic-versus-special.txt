## CUI Basic

- Definition: Information for which no specific law or regulation dictates unique handling controls.
Controls: Follows the baseline uniform set of safeguards defined by the National Archives (NARA) in 32 CFR Part 2002 and NIST SP 800-171 for non-federal systems.
Marking: Mandatory to use "CUI" or "CONTROLLED" in the banner; category markings (e.g., //PRVCY) are optional unless required by agency policy. 

## CUI Specified
- Definition: Information where the underlying authority (law or regulation) mandates specific controls that differ from or exceed the Basic baseline.
- Controls: The authority dictates unique requirements, such as enhanced physical security, specific encryption, or restricted dissemination.
- Marking: Mandatory to include the category in the banner, and it must be preceded by the SP- prefix (e.g., CUI//SP-CTI).

In the real world of CUI, there is no such thing as a "Basic Clearance" that grants access to all Basic categories. Access is always governed by Lawful Government Purpose (LGP).
If you grant a user c0 ("CUI Basic") and allow that to "read-down" or cover all other Basic categories (like PRVCY, LEI, or PROCURE), you are technically violating the Need-to-Know principle required by [32 CFR Part 2002](https://www.archives.gov/cui/registry/cui-glossary.html).
Why "c0" as a Catch-All is Dangerous
In an SELinux/MCS environment, here is how you should handle the distinction:

   1. CUI Basic is a Handling Standard, Not an Access Level:
   An HR specialist needs Privacy (PRVCY) data (Basic) but has no business seeing Contract (PROCURE) data (also Basic). If your SELinux policy allows c0 to read both, you've created a "flat" security model that CUI is specifically designed to prevent.
   2. The "Purpose" Gate:
   In CUI, a person's "Clearance" is their background check, but their "Access" is their Attribute. You should use specific category memberships (e.g., c5 for PRVCY, c10 for PROCURE) to mimic the real-world requirement that users only see what their specific job requires.

The Correct SELinux Strategy
Instead of using c0 as a "Basic Clearance" level, use it strictly for un-categorized legacy data. For everything else:

* Assign Category Memberships: Users should have a set of categories (e.g., s0:c5,c10) that matches their job functions.
* Enforce MCS Multi-Category Logic: If a file is labeled with both c5 (PRVCY) and c10 (PROCURE), the user must have both in their process context to read it.
* Basic vs. Specified: The only difference for your SELinux policy is that Specified (SP-) categories often have stricter Type Enforcement (TE) rules (e.g., only certain trusted applications can open SP-CTI files), whereas Basic categories might share a common "cui_client_t" domain.

Recommendation: Treat c0 as "General/Miscellaneous CUI Basic" and require specific category memberships for everything else to ensure your system actually enforces Lawful Government Purpose.

In an SELinux MCS (Multi-Category Security) setup—which is what you are using if you are at s0—there is no natural "read-down" between categories. To the kernel, c10 and c11 are just two different buckets.
However, you have two architectural paths to handle the SP-EXPT vs. EXPT relationship:
Path A: The "Clearance Set" Approach (Recommended)
You treat them as two distinct attributes. If a person is cleared for the "harder" stuff (SP-EXPT), you simply assign them both categories in their SELinux user mapping.

* User Clearance: s0:c10,c11 (Can see both)
* User Clearance: s0:c10 (Can only see Basic)
* File A (EXPT): s0:c10
* File B (SP-EXPT): s0:c11

This mirrors the real world: just because you can handle "Specified" data doesn't mean you automatically get a "pass" to see every "Basic" file in the system. You still need Category Membership for each.
Path B: The MLS "Dominance" Approach (Hierarchical)
If you move to MLS (Multi-Level Security), you can use the Sensitivity levels (s0, s1, s2) to create a hierarchy.

* s0: CUI Basic
* s1: CUI Specified
* The Logic: An s1 user can "dominate" (read) s0 files.

The Catch: MLS is much harder to maintain. If an s1 process reads an s0 file and then tries to write to an s0 directory, the kernel will block it (to prevent "writing down" and leaking info). Most people find this too restrictive for a standard CUI environment.
Using SELinux "Types" for Refinement
This is where the real power is. You can use Type Enforcement (TE) to say:

* Type cui_basic_t: Any standard office app (LibreOffice, etc.) can open this.
* Type cui_specified_t: Only a "hardened" or "vetted" application (like a specific encrypted viewer) is allowed to transition to this type and read the data.

This way, even if a user has the category c11 (SP-EXPT) in their clearance, they can't just use cat or vim to leak it—they have to use the authorized Type-restricted application.

While there are over 125 categories, there are only about 15-20 official LDCs used across the government. Most organizations only use the "Big 6":

* NOFORN: No Foreign Dissemination.
* FED ONLY: Federal Employees Only.
* FEDCON: Federal Employees and Contractors Only.
* DL ONLY: Distribution List Only (rare).
* REL TO: Authorized for Release to [Specific Country/Group].
* DISPLAY ONLY: For viewing only, no copying/distribution.

2. Do categories ALWAYS have an LDC?
No. Most CUI Basic is shared freely across the "Lawful Government Purpose" boundary without an LDC.
However, some categories are frequently or strictly paired with specific LDCs due to their underlying laws:

* Export Controlled (EXPT): Almost always paired with NOFORN or REL TO.
* Intelligence (INTEL): Frequently paired with NOFORN.
* Tax (TAX): Often paired with FED ONLY.

3. The "Specified" Exception (The "Wait, what?" moment)
For DoD Technical Data (like your SP-CTI files), the "LDC" isn't actually an LDC—it is the Distribution Statement (B-F).

* In your SELinux policy, you should treat Distribution Statement C exactly like an LDC (a unique category tag), even though it doesn't look like NOFORN.

4. Mutually Exclusive LDCs (Critical for your Logic)
In your SELinux code, you must ensure a file doesn't have "clashing" LDC categories. For example:

* A file cannot be both NOFORN (No foreigners) and REL TO GER (Release to Germany).
* Your system should flag or block a label that tries to include both.

Summary for your SELinux Mapping:

* CUI Category = What the data is (e.g., CTI).
* LDC = Who can see it (e.g., NOFORN).
* Your Policy: A user needs the Category Tag AND must meet the LDC Attribute (e.g., Nationality: US) to get read access.

In SELinux MCS, you should treat LDCs and Distribution Statements as distinct "attribute tags" within your category set. This allows you to enforce access based on both the content (What is it?) and the dissemination (Who can see it?).
To answer your first question: Yes, a file can have more than one LDC, provided they aren't logically contradictory (e.g., you can have FEDCON and PRIOR together, but not NOFORN and REL TO [Country]).
Here is how you configure your SELinux contexts for both scenarios:
Scenario 1: Data with an LDC (e.g., CUI//INV//FED ONLY)
This file contains Investigative info restricted to Federal employees only.

   1. Map Categories:
   * c10 = INV (Category)
      * c50 = FED_ONLY (LDC)
   2. File Context: s0:c10,c50
   3. User Access:
   * Federal Investigator: Cleared for s0:c0-c1023 (or specifically c10,c50). Access GRANTED.
      * Contractor Investigator: Cleared for c10 (INV) but NOT c50. Access DENIED by the kernel.
   
Scenario 2: Data with a Distribution Statement (e.g., CUI//SP-CTI with Statement C)
This file contains Technical Info restricted to Government and Contractors.

   1. Map Categories:
   * c20 = SP-CTI (Category)
      * c60 = DIST_STMT_C (Dissemination Control)
   2. File Context: s0:c20,c60
   3. User Access:
   * DoD Contractor: Cleared for c20 and c60. Access GRANTED.
      * Foreign Partner: Even if they have a "Technical" clearance, if they lack the c60 attribute in their SELinux user mapping, the kernel blocks the open() syscall.
   
Configuration Best Practices

* Contradiction Check: In your user-space management tool, write a wrapper that prevents labeling a file with c50 (FED ONLY) and c60 (DIST STMT C), as they represent different sharing tiers.
* The "All-Must-Match" Rule: Remember that in MCS, if a file has c10,c20,c30, the user must have all three in their set. If they are missing even one, they are out.
* Type Enforcement (TE) Overlay: Use a specific SELinux type (e.g., cui_technical_data_t) for the Distribution Statement files to ensure they can only be opened by "vetted" applications, while using a more generic type for LDC files.

To make your SELinux labels exactly match the official CUI banner syntax (including the double slashes), you can define those specific combinations in your setrans.conf.
The mcstrans daemon is quite flexible—it simply looks for a bitmask match (like c10,c50) and replaces it with whatever string you provide.
The "Banner-Compliant" setrans.conf

# Basic Categories
s0:c10=CUI//INV
s0:c20=CUI//SP-CTI
# Limited Dissemination Controls (LDCs)
s0:c50=//FED ONLY
s0:c100=//NOFORN
# Distribution Statements (Note: No // suffix per DoD rules, but you can add it for your system clarity)
s0:c60= (Dist Stmt C)
s0:c61= (Dist Stmt D)
# --- COMBINED BANNER STRINGS ---# This is the "Magic" that makes ls -Z look like a CUI Banner
# Example: CUI//INV//FED ONLY
s0:c10,c50=CUI//INV//FED ONLY
# Example: CUI//SP-CTI//NOFORN
s0:c20,c100=CUI//SP-CTI//NOFORN
# Example: CTI with a Distribution Statement
s0:c20,c60=CUI//SP-CTI (Dist Stmt C)

Why this is the "Ultimate" Setup for You:

   1. Visual Alignment: When you run ls -Z, the output will look nearly identical to the physical marking on the document. This reduces human error during audits.
   2. Double-Slash Logic: By defining the combined entry s0:c10,c50 with the double slashes (//), you bypass the default behavior where SELinux might just put a comma between them.
   3. Audit Logs: Your /var/log/audit/audit.log will now record denials using these exact strings if the mcstrans daemon is active, making it much easier for a Security Officer to understand why access was blocked.

A Quick Implementation Warning:
In SELinux, the order of categories in the bitmask matters for translation. Usually, it's numerical (c10,c50). If you label a file as c50,c10, the translator might not catch it unless you define both directions or stick to a strict labeling script that always sorts categories numerically.

In older versions of mcstrans (and still supported in many implementations), you could define "modifiers" or "suffixes" that the daemon would dynamically append to any base category. This prevents you from having to manually write every single possible combination (like INV//NOFORN, CTI//NOFORN, NUC//NOFORN) in the config file.
How to set up "Auto-Suffixing" for CUI
You define your Categories and your Modifiers (LDCs/Distribution Statements) separately using the bit-masking logic.

# 1. Define the "Base" Categories (The "What")# These are the first part of the banner
s0:c10=CUI//INV
s0:c20=CUI//SP-CTI
s0:c30=CUI//NUC
# 2. Define the "Modifiers" (The "Handling")# Note the leading double slashes in the string!
s0:c100=//NOFORN
s0:c101=//FED ONLY
s0:c102=//FEDCON
# 3. Define the "Distribution" Modifiers# (No slashes per standard, but you can add them for style)
s0:c200= (Dist Stmt C)
s0:c201= (Dist Stmt D)

How the "Auto-Suffix" Works:
When mcstrans sees a file labeled s0:c10,c100, it looks up c10 and c100 independently. Because they are both present in the MCS set, it concatenates them in the order they appear in the bitmask.

* File 1 (s0:c10,c100) → Translates to: CUI//INV//NOFORN
* File 2 (s0:c20,c101) → Translates to: CUI//SP-CTI//FED ONLY
* File 3 (s0:c30,c201) → Translates to: CUI//NUC (Dist Stmt D)

Why this is better for your Access Control:

   1. Scalability: If you have 100 categories and 10 LDCs, you only need 110 lines in your config instead of 1,000 combinations.
   2. Order of Operations: SELinux processes categories numerically (c0 before c100). By putting your Categories in the low range (c0-c99) and your Handling/LDCs in a high range (c100+), the "Banner" will always be built in the correct order: Category + Handling.
   3. Validation: If a user tries to add a category that isn't defined, the system just shows the raw c number, tipping you off that the label is non-compliant.

The "Mutually Exclusive" Logic
While setrans.conf will happily concatenate //NOFORN//REL TO GER if you apply both bits, your Access Control logic is what matters. You should ensure your user-provisioning scripts (the ones that run semanage user) never assign conflicting LDC bits to a single user.
