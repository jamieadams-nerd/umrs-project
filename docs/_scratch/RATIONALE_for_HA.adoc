== Overview: The High-Assurance Needle
High assurance is defined by the **integrity of execution** and **deterministic provenance**. This
document provides the technical and operational rationale for the **Unclassified MLS Reference
System (UMRS)**, focusing on how SELinux MLS labeling and IMA/EVM integrity enforcement prevent 
systemic failures that traditional security models cannot stop.

== 1. Technical Enforcements: SELinux MLS and IMA/EVM
The UMRS architecture moves beyond simple permissions by implementing two core kernel guards:

* **SELinux MLS Labeling:** Controls **access and flow**. It prevents "too much legitimate access" 
by ensuring a process can only interact with data matching its security level and category 
(e.g., CUI//WATER).
* **IMA/EVM (Integrity/Verification):** Controls **state and provenance**. IMA calculates a 
cryptographic hash of files (Measurement) and blocks access if the hash doesn't match the "good" 
value (Appraisal). EVM protects metadata (labels, ownership) from offline tampering.

== 2. New Real-World Evidence: Technical Provenance & Integrity

=== The "Deepwater Horizon" Alarm Bypass (Industrial Safety)
* **The Incident:** Testimony revealed that the rig's general alarm was set to "bypass" mode to 
prevent false alarms from waking workers. When methane gas hit the drill floor, the alarm did not 
trigger, leaving workers with no warning until the explosion.
* **The Failure:** A critical safety configuration (the alarm state) was modified for convenience 
without enforced integrity.
* **UMRS Solution:** **IMA/EVM Appraisal** would have treated the alarm configuration file as a 
**controlled object**. Any unauthorized modification to the "bypass" state would have been 
detected or blocked by the kernel, ensuring the safety-critical state remained authoritative.

=== The Mars Climate Orbiter (Unit Discrepancy)
* **The Incident:** A $125M spacecraft was lost because one engineering team used metric units 
while another used English units in software code. 
* **The Failure:** A data "transformation" error where data was reused outside its original 
context without verified provenance or type enforcement.
* **UMRS Solution:** **SELinux MLS labels** can be used to tag data by its "context" or "unit 
standard." If a process expecting "Metric" data attempted to ingest a file labeled "English," the 
kernel would deny the flow, preventing the calculation error at the source.

=== OPM & Equifax (The "One Big Database" Problem)
* **The Incident:** Millions of PII records were exfiltrated because attackers (or insiders) gained 
access to centralized, unsegmented repositories.
* **The Failure:** "Too much legitimate access" once inside the network.
* **UMRS Solution:** **SELinux MLS labeling** segments data into "cells" (e.g., Member ID vs. 
Clinical Data). This prevents an attacker with a single set of stolen credentials from performing a 
bulk export of the entire database.

== 3. Sector-Specific Failure Maps (Expanded)

=== CUI//WATER & CUI//ENV (Water Quality)
* **Incident (Flint Water Crisis):** Reports were altered or withheld. Internal EPA memos 
documenting concerns were not authoritative enough to force action.
* **UMRS Solution:** **IMA/EVM** provides an immutable record. A "Water Quality Memo" becomes a 
signed, measured artifact. If a supervisor attempts to "silent edit" a concern before it is sent to 
regulators, the kernel-level audit log and appraisal failure provide immediate forensic proof.

=== CUI//TRAN & CUI//CRIT (Infrastructure Safety)
* **Incident (Rail/Pipeline Falsification):** Cases like WMATA and PG&E where inspectors faked 
safety reports over years.
* **UMRS Solution:** Using **IMA/EVM**, an "Inspection Event" is cryptographically bound to the 
user, device, and timestamp. A report cannot be created for a date in the past or modified after 
the fact without breaking the integrity chain.

=== CUI//PRIVACY (The "Saudi Twitter" Case)
* **Incident:** Twitter employees misused legitimate access to user data for foreign intelligence.
* **UMRS Solution:** **Purpose-based labels**. An operator may have "Read" access for support 
tickets, but "Export" access for the same data would require a different security context, 
preventing bulk exfiltration by an individual.

== 4. Why UMRS/MLS Thinking Matters
Unclassified systems fail because they treat important data like ordinary files. The UMRS 
integration of SELinux and IMA/EVM provides:

1. **Constrained Access:** Prevents the "too much access" failure via MLS labeling.
2. **Proven Provenance:** Proves via **IMA hashes** exactly what was seen and when.
3. **Integrity Enforcement:** **IMA/EVM** blocks tampered files from being used in decisions.
4. **Purpose Separation:** Segregates operational workflows from regulatory reporting.

[IMPORTANT]
====
**Controlled ≠ Secret.** In UMRS, **Controlled = Decision-Critical.**
====

== 5. Evidence Collection Guidance for Auditors
Administrators generate observable evidence of enforcement via:
* **`getfattr -n security.ima`**: Proving files have active integrity hashes.
* **`/sys/kernel/security/ima/ascii_runtime_measurements`**: A tamper-proof history of file access.
* **`ls -Z`**: Confirming MLS security labels are applied to all CUI artifacts.
