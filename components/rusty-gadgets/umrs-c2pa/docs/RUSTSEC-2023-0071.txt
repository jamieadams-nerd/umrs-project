RUSTSEC-2023-0071 — RSA Marvin Attack Assessment
=================================================

Advisory:     RUSTSEC-2023-0071
Crate:        rsa 0.9.10
CVSS:         5.9 (Medium)
Type:         Timing side-channel in PKCS#1 v1.5 decryption
Patched:      No patched version available as of 2026-04
Reference:    https://rustsec.org/advisories/RUSTSEC-2023-0071
Assessed:     2026-04-02
Assessed by:  Knox (security-engineer), Rusty (rust-developer), Herb (security-auditor)

Dependency Path
---------------

  rsa 0.9.10 <- c2pa 0.78.7 <- umrs-c2pa

The rsa crate is a transitive dependency. UMRS does not declare a direct
dependency on the rsa crate.


What Is the Marvin Attack?
--------------------------

The Marvin Attack is a modernized variant of Bleichenbacher's 1998 adaptive
chosen-ciphertext attack against RSA PKCS#1 v1.5.

The attack works like this: an attacker sends crafted ciphertexts to a system
that performs RSA decryption and measures how long each decryption takes.
The PKCS#1 v1.5 padding check in the rsa crate is not constant-time —
valid and invalid padding take slightly different amounts of time to process.
These timing differences are small (nanoseconds to microseconds) but
measurable, especially over a network or shared system.

By collecting enough timing measurements across many crafted inputs, the
attacker can determine whether the padding was valid for each attempt.
This leaks information about the plaintext one bit at a time. With
sufficient queries, the attacker can recover the full plaintext or even
the private key.

The fundamental issue is that the rsa crate's decryption code has
data-dependent branches in the padding validation path. Constant-time
implementations (like those in OpenSSL's FIPS module) avoid this by
ensuring the same code path executes regardless of whether the padding
is valid.


UMRS Impact Assessment: NOT AFFECTED
-------------------------------------

UMRS does not exercise the vulnerable code path. Four independent reasons:

1. UMRS NEVER PERFORMS RSA SIGNING OR DECRYPTION.

   All UMRS signing operations use ECDSA exclusively (P-256 or P-384).
   The parse_algorithm() function in signer.rs enforces an algorithm
   allow-list. RSA is not on the list. This gate is the single entry
   point for all signing operations — it cannot be bypassed.
   (NSA RTB RAIN — non-bypassability.)

   Allowed algorithms: es256 (ECDSA P-256), es384 (ECDSA P-384).
   Explicitly excluded: ed25519 (not FIPS 186-5 approved), all RSA variants.

2. THE MARVIN ATTACK TARGETS DECRYPTION, NOT SIGNATURE VERIFICATION.

   When UMRS encounters a third-party C2PA manifest that was signed with
   RSA, the c2pa SDK performs signature VERIFICATION. Signature verification
   is a public-key operation — it uses the signer's public key to check
   the signature. The Marvin Attack exploits timing differences in the
   DECRYPTION padding check, which is a private-key operation. These are
   fundamentally different code paths. Signature verification does not
   exercise the vulnerable PKCS#1 v1.5 decryption logic.

3. ON FIPS-ACTIVE SYSTEMS, RSA ROUTES THROUGH SYSTEM OPENSSL.

   When /proc/sys/crypto/fips_enabled is 1 (which it is on all compliant
   RHEL 10 CUI deployments), the openssl crate's FIPS provider handles
   RSA operations. The pure-Rust rsa crate is not exercised. The system
   OpenSSL FIPS module implements constant-time RSA and is FIPS 140-3
   validated (certificate #4282 for RHEL 10).

4. UMRS HOLDS NO RSA PRIVATE KEYS.

   The Marvin Attack requires a decryption oracle — a service that will
   decrypt attacker-supplied ciphertexts using a private key. UMRS does
   not possess, generate, store, or use RSA private keys. There is no
   oracle to attack.

Monitoring Plan
---------------

- Track upstream c2pa crate releases for an update that removes or replaces
  the rsa dependency.
- If a future UMRS feature requires RSA key operations (none are planned),
  this assessment must be revisited before implementation.
- Re-assess if the rsa crate publishes a patched version — the transitive
  dependency should be updated even though UMRS is not affected, to
  eliminate the advisory from cargo audit output.
