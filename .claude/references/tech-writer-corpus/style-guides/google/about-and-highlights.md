## About the Guide

This guide provides editorial style guidelines for public-facing Google developer documentation.
It contains guidelines, not rules — you should depart from it when doing so improves your content.

**Priority hierarchy for style decisions:**
1. Project-specific style — follow guidance specific to your project or product
2. This style guide — when project-specific guidelines don't provide explicit guidance
3. Third-party references: Merriam-Webster, the Chicago Manual of Style, and the Microsoft Writing
   Style Guide

If you're new to the guide, start with **Highlights**, **Voice and Tone**, and
**Text-Formatting Summary**.

## Core Style Highlights

### Voice and tone

- Be conversational and friendly without being frivolous.
- Use active voice and second person ("you"), not "we."
- Write in present tense.
- Use standard American spelling.
- Avoid phrases like *simply*, *It's that simple*, *It's easy*, or *quickly* in a procedure.
- Avoid internet slang or abbreviations such as *tl;dr* or *ymmv*.
- Read your draft aloud to check for natural flow.
- Use transitions between sentences — phrases like *Though* or *This way* can make paragraphs
  less stilted.

### Formatting rules

- Use sentence case for document titles and section headings.
- Put UI elements in bold.
- Put code-related text in code font.
- Use serial commas (Oxford comma).
- Use unambiguous date formatting.
- Don't use ampersands (&) as conjunctions or shorthand for *and* — use the word *and* instead.

### Structural rules

- Put conditions before instructions.
- Use numbered lists for sequences.
- Use bulleted lists for most other lists.
- Use description lists for pairs of related pieces of data.
- Introduce lists with a complete sentence.
- Maintain parallel structure across all list items.
- Don't use a list to show only one item.

### Inclusive writing

Write inclusively: use gender-neutral language, address a diverse audience, and avoid offensive,
triggering, or ableist language. Clarify person-first and identity-first language per guide
guidance.

### Writing for a global audience

Use simple verbs. Avoid idioms, cultural references, and expressions that may not translate.
Standard American spelling and clear sentence structure support machine translation.
