# Efficient and Robust Approximate Nearest Neighbor Search Using Hierarchical Navigable Small World Graphs

**Authors:** Yury A. Malkov, Dmitry A. Yashunin
**Year:** 2016 (arXiv preprint); 2020 (IEEE TPAMI publication)
**Published in:** IEEE Transactions on Pattern Analysis and Machine Intelligence, 42(4), 824–836
**Source URL:** https://arxiv.org/abs/1603.09320

---

## Overview

HNSW (Hierarchical Navigable Small World) is the dominant algorithm for approximate
nearest neighbor (ANN) search in high-dimensional vector spaces. It is the algorithm
underlying most modern vector database systems including ChromaDB, Pinecone, Weaviate,
Qdrant, Milvus, and FAISS (with HNSW index type). Understanding HNSW is essential for
any system that uses dense vector retrieval — including UMRS's RAG pipeline.

HNSW combines two key insights:
1. **Skip list analogy:** A hierarchy of graphs at multiple scales, allowing long-range
   "highway" connections at upper layers and fine-grained local connections at lower layers
2. **Small world graphs:** The NSW (Navigable Small World) property — graphs where any two
   nodes can be reached in O(log N) hops — enables efficient greedy traversal

The result is an index structure that achieves sub-linear query time (O(log N)) with
high recall (>95% at top-10 retrieval) and O(N log N) construction time.

## Background: Navigable Small World Graphs

### Small World Phenomenon

The "small world" phenomenon (Milgram 1967, Watts-Strogatz 1998): in many real-world
graphs (social networks, the web, biological networks), the average shortest path length
scales as O(log N) even for very large N. This is because nodes have both local clustering
and a small number of long-range connections.

### NSW for ANN Search

Malkov et al.'s earlier NSW paper (2014) showed that greedy search on a navigable small
world graph achieves approximate nearest neighbor search with polylog complexity:
- Build graph by inserting vectors one by one; connect each new vector to its k nearest
  neighbors among existing vectors
- Query: greedy traversal starting from an entry point, always moving to the neighbor
  closest to the query vector

NSW has a problem: the query complexity is O(log² N) but the constant is large due to
routing through "hub" nodes at varying scales.

## HNSW Algorithm

### Structure

HNSW builds a multi-layer graph H = {L_0, L_1, ..., L_{max}} where:
- L_0 is the bottom layer: contains ALL vectors, with short-range connections
- L_i for i > 0: contains a random subset of vectors (each vector enters layer i
  with probability proportional to e^{-i/mL}, an exponentially decaying density)
- Connections at layer i are to M nearest neighbors at that layer

This is directly analogous to a **skip list**: the upper layers provide long-range
"express" connections while lower layers provide precise local resolution.

### Construction

```
procedure insert(q, M, mL):
  L = floor(-ln(uniform(0,1)) * mL)  // assign layer
  ep = entry_point  // current graph entry point
  for lc = max_layer down to L+1:
    W = search_layer(q, ep, ef=1, lc)  // greedy search, single candidate
    ep = nearest(W, q)
  for lc = min(L, max_layer) down to 0:
    W = search_layer(q, ep, ef=efConstruction, lc)  // beam search
    neighbors = select_neighbors(q, W, M, lc)  // heuristic selection
    connect(q, neighbors, lc)
    for each e in neighbors:
      if degree(e, lc) > Mmax:
        shrink(e, lc, Mmax)  // prune to maintain degree bound
    ep = W
  if L > max_layer:
    entry_point = q; max_layer = L
```

### Query

```
procedure search(q, K, ef):
  ep = entry_point
  for lc = max_layer down to 1:
    W = search_layer(q, ep, ef=1, lc)  // upper layers: ef=1 (greedy)
    ep = nearest(W, q)
  W = search_layer(q, ep, ef, lc=0)  // bottom layer: ef candidates
  return nearest K elements of W
```

### search_layer (Beam Search)

```
procedure search_layer(q, ep, ef, lc):
  visited = {ep}
  candidates = priority_queue({ep})  // min-heap by distance to q
  W = priority_queue({ep})  // max-heap: current best ef results
  while candidates not empty:
    c = extract_nearest(candidates)
    f = furthest(W)
    if dist(c,q) > dist(f,q): break  // all candidates worse than known results
    for each e in neighbors(c, lc):
      if e not in visited:
        visited.add(e)
        f = furthest(W)
        if dist(e,q) < dist(f,q) or |W| < ef:
          candidates.add(e)
          W.add(e)
          if |W| > ef: remove_furthest(W)
  return W
```

### Neighbor Selection Heuristic

Standard selection: take M nearest neighbors from candidates.
HNSW heuristic (better in practice): select neighbors that improve coverage of the
local Voronoi cell:
- For each candidate e, add it to neighbors only if no existing neighbor is closer to e than q
- This ensures the neighbor set "covers" the region around q rather than clustering near one side

The heuristic improves recall by creating a more diverse neighborhood — a direct application
of the diversity criterion from information theory (diverse neighbors span more of the space).

## Complexity Analysis

### Construction
- Each insertion: O(M² · ef_construction · log N) — beam search at each layer
- Total: O(N · M² · ef_construction · log N)
- Space: O(N · M · log N) — each node has M connections at each of its layers

### Query
- Upper layers: O(log N) greedy steps per layer
- Layer 0: O(ef · M) neighbor checks
- Total: O(ef · M · log N) — effectively O(log N) for fixed ef and M

### Empirical Performance (from paper)

On SIFT-1M (1M 128-dim float vectors):
- HNSW achieves 0.995 recall@10 in ~0.4ms per query
- FAISS IVF achieves same recall in ~2ms (5× slower)
- Brute force: ~10ms (25× slower than HNSW)

Memory: ~100 bytes per vector with M=16 — much less than hash-based methods.

## Key Parameters

- **M:** Number of connections per node per layer (typ. 16–64); controls graph density,
  recall, and memory
- **ef_construction:** Beam width during construction (typ. 200–400); more is slower to
  build but better recall
- **ef (query):** Beam width during query (runtime parameter); tune for recall vs. speed
- **mL = 1/ln(M):** Layer assignment probability normalization; controls hierarchy depth
- **Mmax:** Maximum connections per node at layer 0 (typ. 2M); prevents hub formation

## Information-Theoretic Analysis

### HNSW as Approximate Search

HNSW is an approximate algorithm — it returns near-neighbors, not exact nearest neighbors.
The recall-speed trade-off is the graph-search analog of rate-distortion:
- Higher ef → higher recall (lower distortion) → slower query (higher "cost")
- The Pareto frontier of recall vs. speed is analogous to the rate-distortion curve R(D)

### Entropy of the Navigable Graph

The graph structure encodes implicit knowledge about the local geometry of the embedding
space. Nodes with many long-range connections function as "hubs" that reduce the graph's
diameter. The entropy of the degree distribution determines how efficiently the graph
routes queries:
- Scale-free degree distributions (few very-high-degree hubs) minimize diameter
  but create routing bottlenecks
- HNSW's degree bound (Mmax) prevents pathological hub formation while maintaining
  navigability

### Connection to Network Information Theory

HNSW's layer structure is analogous to a communication network with different bandwidth
layers:
- Top layers: low-bandwidth, long-range routing (few nodes, coarse resolution)
- Bottom layers: high-bandwidth, local delivery (all nodes, fine resolution)

This is exactly the hierarchical coding structure analyzed in network information theory
(Slepian-Wolf, Wyner-Ziv coding). The multi-scale structure achieves near-optimal
information routing.

## UMRS Relevance

### ChromaDB Backend

UMRS's RAG pipeline uses ChromaDB with HNSW indices. Understanding HNSW means:
- **Parameter tuning:** M=16, ef_construction=200, ef=50 are common defaults; increasing
  M improves recall but increases memory and construction time
- **Collection size limits:** Above ~10M vectors, HNSW construction time becomes a concern;
  multi-segment approaches or DiskANN (disk-based HNSW variant) address this
- **Embedding dimension:** HNSW recall degrades gracefully with dimension (unlike k-d trees
  which become unusable above ~20 dims); 1536-dim OpenAI embeddings work well

### Retrieval Quality Analysis

When RAG retrieval quality is poor (wrong documents returned), the HNSW layer can be
examined by:
- Checking ef (query beam width) — increasing ef improves recall at latency cost
- Checking embedding quality — if embeddings don't capture semantic similarity, HNSW
  retrieves correctly by its metric but incorrectly by semantic meaning
- Checking collection fragmentation — small collections have fewer navigable paths

### Multi-level Security and Embedding Partitioning

For UMRS's MLS use case, documents at different security levels should be in separate
HNSW collections (different ChromaDB collections). HNSW doesn't support per-document
access control natively — the security boundary must be enforced at the collection level:
- One collection per security level
- Queries filtered to collections the querying subject is cleared for
- Cross-level retrieval is explicitly prohibited by collection separation

This matches UMRS's trust model: SELinux labels govern which collection a query can access.

### Index Refresh Strategy

HNSW supports online insertion but not deletion (efficiently). When documents are updated:
- Mark the old document as deleted (tombstone)
- Insert the new document
- Periodically rebuild the index to remove tombstones (compaction)

This has implications for the RAG pipeline refresh workflow.

## Key Terminology

- **ANN (Approximate Nearest Neighbor):** Finding close-enough neighbors in sub-exact time
- **Small world graph:** Graph with O(log N) diameter and local clustering
- **Skip list:** Hierarchical linked-list structure; HNSW is its graph analog
- **Beam search:** Best-first search maintaining ef candidates; used in search_layer
- **M:** Maximum connections per node; core HNSW construction parameter
- **ef_construction / ef:** Beam width at build / query time
- **Recall@K:** Fraction of true top-K nearest neighbors returned by ANN query
- **Voronoi cell:** Region of space closest to a given vector; HNSW heuristic approximates coverage
- **Hub:** Node with very high degree; HNSW bounds degree to prevent bottlenecks
- **SIFT-1M:** Standard ANN benchmark (1M 128-dim SIFT descriptors)
