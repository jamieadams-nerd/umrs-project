## Overview: AVX-512 for Security Engineers

AVX-512 is a family of Intel instruction set extensions centered on 512-bit (ZMM) vector
registers. The name is misleading: "AVX-512" is not a single feature but a collection of
independent subsets, each with its own CPUID bit. A CPU may support AVX-512 Foundation
(`avx512f`) but not the crypto-specific subsets (VAES, VPCLMULQDQ). Detection requires
checking multiple independent flags.

### Security relevance

AVX-512 matters for security engineers for three distinct reasons:

1. **Crypto throughput multiplier:** VAES and VPCLMULQDQ on ZMM registers process 4 AES blocks
   per instruction (vs 1 for AES-NI on XMM, 2 for VAES on YMM). This enables AES-GCM at
   5–8 GB/s per core — relevant for high-bandwidth encrypted storage (dm-crypt, kernel TLS).

2. **Frequency scaling penalty:** On legacy Skylake-X server CPUs, executing AVX-512 "heavy"
   instructions triggers a downclocking of the entire socket (or cluster). A small fraction of
   AVX-512 operations can reduce base frequency by 200–400 MHz across all cores. Cloud providers
   discovered this in 2018 and many now mask AVX-512 from VMs by default to avoid cross-tenant
   frequency interference. This is the most operationally significant AVX-512 caveat.

3. **Cloud VM masking:** Because of the frequency penalty, AWS, GCP, and others mask AVX-512
   from virtual machines on certain instance families. A deployed system may have `/proc/cpuinfo`
   reporting no `avx512f` flag even when running on AVX-512-capable physical hardware. This is
   not a security finding but must be understood when evaluating the crypto acceleration stack.

### Frequency Scaling: The Technical Detail

Intel introduced per-core frequency scaling for wide SIMD operations. There are two categories:

- **Light AVX-512 instructions:** Integer operations, `VMOVAPS`, comparisons — minimal or no
  frequency impact on newer microarchitectures (Ice Lake and later).
- **Heavy AVX-512 instructions:** Floating-point multiply-add (FMA), VAES, VPCLMULQDQ — trigger
  downclocking on Skylake-X and some Cascade Lake SKUs.

Travis Downs' detailed analysis (2020) showed that Skylake-X enters a "license 2" frequency
state (reduced by ~200–400 MHz) after executing a heavy 512-bit instruction, with a transition
period of approximately 20 microseconds during which instruction throughput is reduced to 1/4
normal dispatch rate.

On Ice Lake (2019) and later Intel server CPUs, the frequency penalty was significantly reduced.
AMD Zen 4 and Zen 5 processors handle AVX-512 without frequency reduction.

**Cloudflare's production finding (2018):** Cloudflare running Xeon Silver 4116 (Skylake) at 2.1
GHz base measured: single-core AVX-512 execution reduced frequency to 1.8 GHz; all-core AVX-512
reduced to 1.4 GHz. Even when AVX-512 consumed only 2.5% of CPU time (ChaCha20-Poly1305
workload), the frequency reduction affected all threads. This caused measurable latency
regressions in non-AVX-512 work.

**Implication for UMRS:** On Skylake-class bare metal, verify that OpenSSL and the kernel crypto
stack are not inadvertently enabling AVX-512 paths for workloads where the frequency penalty
outweighs the throughput gain. On Ice Lake and AMD Zen 4+, this concern is substantially reduced.

## Feature #19: AVX-512 (Foundation + Crypto-Relevant Subsets)

### AVX-512 Subset Map (crypto-relevant)

AVX-512 is a family, not a single feature. The researcher must treat each subset as a
separately detectable CPUID flag. The full AVX-512 family has over 20 subsets; the
crypto-relevant subset map:

| Subset | CPUID Leaf | EBX/ECX bit | `/proc/cpuinfo` flag | Purpose |
|--------|-----------|------------|---------------------|---------|
| AVX-512F (Foundation) | 07H EAX=0 | EBX bit 16 | `avx512f` | Required by all other subsets; 512-bit ZMM registers |
| AVX-512BW (Byte/Word) | 07H EAX=0 | EBX bit 30 | `avx512bw` | 8/16-bit lane ops; needed for VPSHUFB on ZMM |
| AVX-512VL (Vector Length) | 07H EAX=0 | EBX bit 31 | `avx512vl` | 128/256-bit operation modes on ZMM-capable CPUs |
| AVX-512DQ (Double/Quad) | 07H EAX=0 | EBX bit 17 | `avx512dq` | 32/64-bit lane instructions; used in some ECC paths |
| AVX-512CD (Conflict Detection) | 07H EAX=0 | EBX bit 28 | `avx512cd` | Conflict detection for gather/scatter; not crypto-critical |
| VAES (on ZMM/YMM) | 07H EAX=0 | ECX bit 9 | `vaes` | Vectorized AES on 256/512-bit registers; **most crypto-critical** |
| VPCLMULQDQ | 07H EAX=0 | ECX bit 10 | `vpclmulqdq` | Vectorized carryless multiply for AES-GCM on ZMM/YMM; **most crypto-critical** |
| AVX-512VNNI | 07H EAX=0 | ECX bit 11 | `avx512_vnni` | Neural network integer ops; low crypto relevance |
| GFNI | 07H EAX=0 | ECX bit 8 | `gfni` | Galois Field instructions; used in advanced AES-GCM implementations |

**Detection rule:** Do NOT check only for `avx512f`. Check each crypto-relevant subset
independently. A VM may have `avx512f` but not `vaes` or `vpclmulqdq`. The crypto benefit
of AVX-512 without VAES+VPCLMULQDQ is marginal for the algorithms that matter.

### VAES and VPCLMULQDQ on AVX-512 — Crypto Acceleration Detail

The combination of VAES + VPCLMULQDQ on ZMM registers is the critical crypto performance
feature in AVX-512. Understanding their interaction with the AES-GCM construction:

**AES-GCM = AES-CTR (encryption) + GHASH (authentication)**

- **AES-CTR with VAES (ZMM):** 4 AES blocks (64 bytes) encrypted per `VAESENC` instruction.
  With a pipeline of 8 VAES operations in flight (managing latency), throughput approaches
  4 blocks × 8 pipeline stages = 32 blocks per 14-cycle AES-256 latency window. This gives
  the 5–8 GB/s throughput figures cited above.

- **GHASH with VPCLMULQDQ (ZMM):** 4 independent GHASH multiplications per instruction. GHASH
  is the bottleneck in AES-GCM when PCLMULQDQ (scalar) is the only available path. VPCLMULQDQ
  eliminates this bottleneck, making AES-GCM throughput limited by AES speed rather than GHASH.

- **Combined:** `aesni_intel` kernel module implements the pipelined VAES+VPCLMULQDQ path when
  both flags are present. This is the path used by kernel TLS (`CONFIG_TLS`) and IPsec for bulk
  encryption of network traffic.

**GFNI (Galois Field NI):** Also introduced with Ice Lake. Provides direct GF(2^8) arithmetic
instructions. Used in alternative AES-GCM implementations (notably in the intel-ipsec-mb
library) and in some SNOW-3G / ZUC cipher implementations (3GPP). Not directly used in the
standard Linux kernel crypto stack as of kernel 6.12, but relevant for specialized workloads.

### KVM CPU Model and AVX-512 Flag Availability

When KVM virtual machines are configured with named CPU models rather than `cpu host`, the
model name determines which flags are exposed. Critical model boundaries for AVX-512:

| KVM CPU model | `avx512f` | `vaes` | `vpclmulqdq` |
|---|---|---|---|
| `Haswell` | No | No | No |
| `Broadwell` | No | No | No |
| `Skylake-Server` (base) | Yes | No | No |
| `Skylake-Server-v4` | Yes | No | No |
| `Cascadelake-Server` | Yes | Yes | Yes |
| `Icelake-Server` | Yes | Yes | Yes |
| `SapphireRapids` | Yes | Yes | Yes |
| `cpu host` | Passes all host flags | Passes if host has it | Passes if host has it |

On UMRS deployments using KVM with named CPU models (common in OpenStack/Satellite-managed
clouds), verify that the model version is at least `Cascadelake-Server` or `Icelake-Server`
to expose VAES+VPCLMULQDQ. Using `cpu host` is the most straightforward path to exposing all
available hardware features to the guest.

### VMware EVC Masking Behavior

VMware EVC (Enhanced vMotion Compatibility) masks CPU flags to the lowest common denominator
of all hosts in a cluster. AVX-512 implications:

- EVC baseline `Intel Skylake EP (Xeon Scalable)` — includes `avx512f` and core subsets;
  VAES and VPCLMULQDQ depend on whether all cluster hosts are Ice Lake or later.
- EVC baseline `Intel Broadwell (Xeon E5/E7 v4)` — AVX-512 entirely masked.
- EVC baseline `Intel Sandy Bridge (Xeon E5/E7 v1)` — AVX2 masked; no AVX-512.

In a production cluster with a mix of Skylake and Ice Lake CPUs, EVC will mask AVX-512 to
the Skylake baseline, losing VAES+VPCLMULQDQ even on Ice Lake nodes. This is a significant
crypto acceleration loss that should be flagged in a UMRS deployment review.

## CVE / Vulnerability Table (AVX-512)

| CVE | Year | Issue | Impact | Resolution |
|-----|------|-------|--------|------------|
| No dedicated CVEs against AVX-512 as a hardware primitive | — | — | — | — |
| Frequency scaling behavior (not a CVE) | 2018 | Skylake AVX-512 all-core frequency reduction | Performance regression, potential latency SLA violations | Use Ice Lake or later; or disable AVX-512 in OpenSSL via `OPENSSL_ia32cap` |
| GFNI carry-less multiply timing research | 2020+ | Theoretical: GFNI misuse could reintroduce timing channels in non-standard GCM implementations | Theoretical — not a CVE; proper GFNI usage is constant-time | Use GFNI only in reviewed crypto implementations (intel-ipsec-mb) |

Note: The primary risk associated with AVX-512 in production is the **frequency scaling penalty**
on pre-Ice Lake CPUs, not a security vulnerability. This risk is operational (latency regressions,
SLA violations) rather than cryptographic.

## Posture Check Specifications

### AVX-512 Capability Assessment (Layer 1)
Check the full subset flag set, not just `avx512f`:
```
/proc/cpuinfo flags to verify for crypto posture:
  avx512f       — Foundation (required for all subsets)
  avx512bw      — Byte/Word ops (required for VPSHUFB on ZMM)
  avx512vl      — Vector Length extensions (128/256-bit modes)
  vaes           — Vectorized AES (primary crypto flag)
  vpclmulqdq    — Vectorized carryless multiply (AES-GCM auth)
```

If `avx512f` present but `vaes`/`vpclmulqdq` absent: note that AVX-512 foundation is present
but the crypto-critical subsets are not available. Likely a cloud VM with partial AVX-512 exposure.

### Frequency Risk Assessment (Skylake-class hardware only)
1. Determine CPU microarchitecture from `/proc/cpuinfo` model name
2. If Skylake-X or Cascade Lake (pre-v2): flag potential frequency penalty risk
3. If Ice Lake, Sapphire Rapids, or AMD Zen 4+: no frequency concern
4. If bare metal RHEL 10 on known-safe microarchitecture: full AVX-512 recommended

### Cloud VM Masking Advisory
If `/proc/cpuinfo` shows no `avx512f` on a known AVX-512-capable instance family:
1. Verify instance type and confirm whether AVX-512 is expected to be exposed
2. Check if `cpu host` or `Cascadelake-Server` (or later) model is configured in KVM/QEMU
3. Document as "AVX-512 masked by hypervisor — expected behavior for this deployment"
4. Note highest available crypto tier: AVX2+AES-NI or AVX2+VAES(YMM) as fallback
