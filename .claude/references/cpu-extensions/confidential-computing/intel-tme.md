# Intel TME (Total Memory Encryption) / MKTME (Multi-Key TME)

## 23-Column Matrix Profile

| # | Column | Value |
|---|--------|-------|
| 1 | Feature name | Intel TME (Total Memory Encryption) / MKTME (Multi-Key Total Memory Encryption) |
| 2 | Vendor | Intel |
| 3 | CPUID detection | **TME:** CPUID leaf 7, subleaf 0, ECX bit 13 (`TME` -- TME supported). **MKTME:** CPUID leaf 7, subleaf 0, EDX bit 18 (`PCONFIG` -- PCONFIG instruction for MKTME key management). CPUID leaf 0x1B (PCONFIG_LEAF) returns PCONFIG target information. **TME activation:** IA32_TME_ACTIVATE MSR (0x982) -- read after boot to determine TME/MKTME state. Bits [31:0] indicate number of KeyIDs available for MKTME. |
| 4 | Linux `/proc/cpuinfo` flag | `tme` (TME capability). MKTME enumerated via PCONFIG leaf but not a separate cpuinfo flag. |
| 5 | Key instructions | PCONFIG (ring-0: key programming for MKTME, subleaf KEY_PROGRAM). No user-visible instructions -- TME is transparent. Memory encryption/decryption occurs in the memory controller. |
| 6 | Introduced | TME: Intel Ice Lake (10th gen client, 2019); 3rd Gen Xeon Scalable (Ice Lake-SP, 2021). MKTME: 3rd Gen Xeon Scalable and later. |
| 7 | Security relevance | TME encrypts all DRAM using AES-XTS-128 with an ephemeral platform key generated by the CPU at each boot. Protects against physical memory attacks (cold boot, bus snooping, DRAM removal). MKTME extends TME with per-VM or per-process keys, enabling memory isolation with hardware-enforced encryption domains. MKTME is the underlying mechanism for TDX memory encryption. |
| 8 | Performance benefit | TME: near-zero overhead (hardware AES engine in memory controller, inline encryption). MKTME: minimal additional overhead for key selection per memory access. Encryption is pipelined and does not add observable latency for most workloads. |
| 9 | Known vulnerabilities | No TME-specific CVEs. TME protects against physical attacks only -- it does NOT protect against software attacks (the running OS/hypervisor has access to all plaintext). MKTME key management via PCONFIG is privileged (ring-0) -- any kernel compromise exposes all keys. TME ephemeral key is generated from CPU hardware RNG (RDRAND-derived); concerns about RDRAND quality apply transitively. |
| 10 | Compliance mapping | NIST SP 800-53 SC-28 (Protection of Information at Rest), MP-5 (Media Transport); CMMC SC.L2-3.13.16; FIPS 140-3 (AES-XTS-128 is a FIPS-approved algorithm, but TME's implementation is part of the CPU silicon, not a separately validated module). |
| 11 | Classification | **Important** |
| 12 | Classification rationale | TME provides transparent memory encryption that protects against physical DRAM attacks. Classification is Important rather than Critical because: (1) TME does not protect against software-based attacks, (2) on systems where physical security is assured, TME's benefit is marginal, (3) MKTME's multi-key capability is primarily relevant for cloud/virtualization providers rather than single-tenant UMRS deployments. For confidential computing deployments, TME/MKTME is a prerequisite for TDX (which IS Critical/Operational). |
| 13 | Linux kernel support | **TME:** Kernel reads IA32_TME_ACTIVATE MSR to determine if TME is active. No kernel configuration required -- TME is transparent once BIOS enables it. **MKTME:** `CONFIG_X86_KEYLOCKER` and MKTME kernel patches (in development). Linux 6.x simplified MKTME detection by reading KeyID bits directly from IA32_TME_ACTIVATE MSR. MKTME KeyIDs are used internally by KVM for TDX memory encryption. |
| 14 | Detection method (safe Rust) | Check `/proc/cpuinfo` for `tme` flag (Layer 1 -- CPU support). Check `dmesg` for "x86/tme: enabled" messages (BIOS activation). Read `/sys/devices/system/cpu/cpu0/` topology for TME/MKTME KeyID information if exposed. No dedicated sysfs interface for TME status -- primary detection is CPUID + MSR (ring-0). From userspace, `dmesg` is the most reliable indicator. |
| 15 | Virtualization confidence | **TRANSPARENT** -- TME encrypts all memory regardless of virtualization. Guest VMs benefit from TME without any awareness. MKTME provides per-VM keys: KVM assigns KeyIDs to VMs. Guest cannot directly verify TME is active (encryption is below the memory controller). Cloud attestation or trust in the cloud provider is required. |
| 16 | ARM/AArch64 equivalent | ARM TrustZone Address Space Controller provides memory encryption regions. ARM Realm Management Extension (RME) provides per-realm memory encryption (closer to MKTME). No direct 1:1 equivalent for transparent full-memory encryption. |
| 17 | References | Intel Architecture Memory Encryption Technologies Specification (doc 336907); Intel SDM; WikiChip TME reference; Linux kernel TME detection code |
| 18 | Disposition when unused | **LEAVE ENABLED** -- TME has no meaningful attack surface. It provides defense-in-depth against physical attacks with zero performance cost. There is no reason to disable TME. MKTME can be left dormant (no KeyIDs allocated) without risk. |
| 19 | Software utilization detection | TME is transparent -- no software opt-in required. Detection is: "Is TME active?" (BIOS/firmware enabled it at boot). `dmesg | grep -i tme` for kernel messages. `/proc/crypto` is NOT relevant (TME uses hardware memory controller encryption, not kernel crypto subsystem). MKTME utilization: check KVM for active KeyID allocations. |
| 20 | FIPS utilization requirement | TME uses AES-XTS-128, which is a FIPS-approved algorithm. However, TME's AES engine is part of the CPU silicon and would need to be covered by a FIPS 140-3 validation certificate for the CPU (not a standalone module). Intel's FIPS validation status for TME specifically is unclear -- verify against Intel's CMVP submissions. |
| 21 | Active mitigation status | No entry in `/sys/devices/system/cpu/vulnerabilities/`. TME is not a vulnerability mitigation -- it is a data-at-rest encryption technology. |
| 22 | Feature accessible vs advertised | **BIOS-GATED.** CPUID reports TME capability, but BIOS must enable it via IA32_TME_ACTIVATE MSR. Some BIOS implementations expose a "Total Memory Encryption" option; others enable it by default on server platforms. MKTME requires additional BIOS configuration for KeyID allocation. If BIOS does not enable TME, the feature is dormant despite CPUID indication. |
| 23 | Guest-vs-host discrepancy risk | **LOW** -- TME is transparent to guests. A guest does not need to know TME is active. MKTME KeyID assignment is host-side only. No guest-visible discrepancy risk for TME. For MKTME/TDX, the discrepancy is handled by TDX attestation. |

## TME vs AMD SME Comparison

| Aspect | Intel TME | AMD SME |
|--------|-----------|---------|
| Encryption algorithm | AES-XTS-128 | AES-128 |
| Key management | Ephemeral (CPU-generated each boot) | Ephemeral or OS-managed via C-bit |
| Granularity | All DRAM (transparent) | Per-page via encryption bit (C-bit) |
| Multi-key | MKTME (KeyIDs) | SME does not have multi-key; SEV provides per-VM keys |
| Detection | CPUID leaf 7 + MSR | CPUID 0x8000001f + MSR |
| Kernel parameter | None required (BIOS-driven) | `mem_encrypt=on` (if BIOS enables but does not activate) |

## Trust Model

**What must be trusted:**
- Intel CPU hardware (memory controller encryption engine)
- CPU hardware RNG (ephemeral key generation)
- BIOS/firmware (TME activation at boot)

**What is NOT trusted (by TME):**
- Physical DRAM (encrypted)
- DRAM bus (encrypted)
- Cold boot attacks (key is ephemeral, lost on power cycle)

**What TME does NOT protect against:**
- Software attacks (running OS has plaintext access to all memory)
- DMA attacks from PCIe devices (unless IOMMU is active)
- Speculative execution side channels
- Kernel memory corruption
