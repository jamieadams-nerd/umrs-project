# Reference Documentation — Divio Documentation System

## Core Purpose

Reference guides serve as "technical descriptions of the machinery" and focus exclusively on describing how to operate software. Their singular objective is to describe code-determined elements like classes, functions, APIs, and methods.

## Key Characteristics

**Information-Oriented Approach**
Reference material prioritizes factual information presentation. While examples may illustrate usage, the content should avoid explaining foundational concepts or demonstrating how to accomplish specific tasks—that belongs in how-to guides.

**Simplicity and Directness**
The documentation should be "simple and to the point." However, basic usage descriptions—such as how to instantiate classes or invoke methods—constitute legitimate reference content as part of technical accuracy, distinct from instructional how-to documentation.

## Writing Best Practices

**1. Mirror Code Structure**
"Give reference documentation the same structure as the codebase," enabling users to navigate code and documentation simultaneously while helping maintainers identify gaps.

**2. Maintain Consistency**
Structure, tone, and format must remain uniform throughout, resembling encyclopedic or dictionary-style presentation.

**3. Describe Only**
"The only job of technical reference is to describe, as clearly and completely as possible." Avoid explanations, instructions, or discussions; instead link to appropriate how-to guides and tutorials.

**4. Ensure Accuracy**
Keep descriptions current and precise. Discrepancies between actual functionality and documentation will mislead users inevitably.

## The Cooking Analogy

Reference material resembles an encyclopedia entry—presenting information about ingredients (or software components) in standardized formats, including provenance, behavior, chemical properties, and usage warnings.
