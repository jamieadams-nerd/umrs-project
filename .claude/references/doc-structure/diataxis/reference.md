# Reference Documentation — Diátaxis Framework

Source: https://diataxis.fr/reference/
Retrieved: 2026-03-12

## Overview

Reference guides function as **technical descriptions** of machinery and operations. They are **information-oriented** in nature, providing propositional and theoretical knowledge that users consult during their work.

Key distinction: "Whereas the content of tutorials and how-to guides are led by needs of the user, reference material is led by the product it describes."

## Purpose and Characteristics

Reference material serves to describe software components—APIs, classes, functions—and their usage. Users depend on such documentation for reliability and certainty, establishing "firm platforms on which to stand while they work."

The material functions similarly to a map: it conveys necessary information about territory without requiring direct exploration. Reference guides should be "austere" with minimal narrative; users consult rather than read them thoroughly.

## Key Principles

### Neutral Description
Focus exclusively on descriptive content with "austere and uncompromising" style emphasizing "neutrality, objectivity, factuality." The documentation should follow the machinery's structural organization rather than user workflows.

### Consistent Patterns
"Standard patterns are what allow us to use reference material effectively." Users expect familiar formats and predictable organization, enabling quick information location.

### Structural Alignment
"The structure of the documentation should mirror the structure of the product" so users navigate them simultaneously without forcing unnatural arrangements.

### Illustrative Examples
Examples provide understanding while maintaining descriptive focus, demonstrating usage without instructional explanation.

## Language Standards

Reference guides employ factual statements about machinery behavior, enumerate commands and options, list features and limitations, and include necessary warnings—always avoiding instructional tone or opinion.
