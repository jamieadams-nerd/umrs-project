## Canonical "translate for [crate]" Workflow

This is the confirmed, user-approved workflow. Execute these steps in order when asked to
"translate for [crate]" or to onboard a new crate to the i18n pipeline.

1. **Check prerequisites**
   - Run `which xtr` or `xtr --version` to confirm it is installed.
   - If missing: report `cargo install xtr` as the install command.
   - Do not run the extraction step, but continue with all other steps.
   - A hand-crafted `.pot` is acceptable as a placeholder when `xtr` is unavailable,
     provided the wrapped strings are already known from source inspection.
   - The hand-crafted `.pot` must be regenerated with `xtr` once it is installed.

2. **Confirm domain assignment**
   - Read `resources/i18n/domains.md`; create it if it does not exist.
   - Binary crates get one domain named after the crate.
   - Library crates each get their own domain (for reuse across binaries).
   - Add the new crate row to the domain table before any extraction work begins.

3. **Run french-lookup skill**
   - Invoke the `french-lookup` Skill tool for each primary technical term.
   - Do this before proposing any translation — corpus result is authoritative.
   - If no corpus match: consult `resources/i18n/vocabulary-fr_CA.md`.
   - If neither source covers the term: retain English and add a `# TRANSLATOR:` comment.

4. **Extract strings**
   - Run `xtr` to produce `resources/i18n/<domain>/<domain>.pot`:
     ```
     xtr --package-name <domain> \
         --output resources/i18n/<domain>/<domain>.pot \
         <source files>
     ```
   - Run from repo root. Re-run after any new `gettext()` wrap in source.
   - If `xtr` unavailable: hand-craft the `.pot` from confirmed source locations.

5. **Initialize locale**
   - If `resources/i18n/<domain>/fr_CA.po` does not already exist:
     ```
     msginit --input=resources/i18n/<domain>/<domain>.pot \
             --locale=fr_CA \
             --output=resources/i18n/<domain>/fr_CA.po \
             --no-translator
     ```

6. **Translate**
   - Replace the `msginit`-generated header with the UMRS standard `.po` header
     (product name = crate name; see header template in system prompt).
   - Fill in `msgstr` values using corpus terms or vocabulary-fr_CA.md.
   - Add `# TRANSLATOR:` comments for any term requiring explanation or judgment.

7. **Validate and compile**
   - `msgfmt --check resources/i18n/<domain>/fr_CA.po`
   - `msgfmt -o resources/i18n/<domain>/fr_CA.mo resources/i18n/<domain>/fr_CA.po`
   - Both must succeed before proceeding.

8. **Update vocabulary file**
   - Read `resources/i18n/vocabulary-fr_CA.md` before the session; update it after.
   - Add any new term decisions with Source and Notes columns.
   - Also update `.claude/agent-memory/umrs-translator/vocabulary.md`.

9. **Update top-level Makefile**
   - Read `/media/psf/repos/umrs-project/Makefile` first (required before any edit).
   - Add `<domain>` to `I18N_TEXT_DOMAINS`.
   - Add `I18N_SOURCES_<domain>` variable mapping domain to source files.
   - Add `I18N_ACTIVE_LOCALES_<domain>` variable listing translated locales.
   - Add per-domain targets: `i18n-extract-<domain>`, `i18n-merge-<domain>`,
     `i18n-compile-<domain>`.
   - Extend `i18n` target to depend on the three new per-domain targets.
   - Extend `i18n-check` to validate the new domain's `.po` files.
   - Add new phony targets to the top-level `.PHONY` declaration.
   - Verify with `make i18n-check` after editing.
