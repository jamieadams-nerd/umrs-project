# Setrans and Catalog Access Control Review\n\n```\nAudit date: 2026-03-27\nDepth: in-depth\nScope: MLS-setrans.conf, TARGETED-setrans.conf, US-CUI-LABELS.json, CANADIAN-PROTECTED.json, LEVELS.json, five-eyes-mcs-allocation-table.md\n```\n\n## Executive Summary\n\nThis review covers the MCS category allocation scheme that forms the foundation of UMRS\nFive Eyes access control. The setrans.conf files translate raw SELinux MCS labels into\nhuman-readable CUI/Protected markings; correctness here is load-bearing for every access\ncontrol decision the system makes.\n\n**Critical finding count: 4 CRITICAL, 4 HIGH, 5 MEDIUM, 4 LOW**\n\nThe most severe findings are MCS category collisions where subcategory numbers from one\nCUI group overflow into the block allocated to another group, creating cross-domain\ndominance paths in the SELinux MCS subsystem. A process cleared for one CUI group would\nautomatically dominate objects in an unrelated group.\n\n---\n\n## Finding 1 — CRITICAL: PRIV/STUD (c10) collides with PROCURE group base (c10)\n\n```\nFile: data/MLS-setrans.conf (line 108), data/TARGETED-setrans.conf (line 84)\nLocation: PRIV block, subcategory STUD\nSeverity: CRITICAL\nControl reference: NIST SP 800-53 AC-4 (Information Flow Enforcement)\nRemediation owner: coder (JSON schema) + security-engineer (setrans)\n```\n\n**The collision:**\n- `s1:c1,c10` = `CUI//PRIV/STUD` (Student Records) — line 108 in MLS\n- `s1:c10` = `CUI//PROCURE` (Procurement and Acquisition) — line 113 in MLS\n\n**The access control consequence:**\nA process with clearance `s1:c1,c10` (cleared for Privacy/Student Records) holds\ncategories {c1, c10}. In MCS dominance logic, this process dominates any object labeled\nwith a subset of its categories. An object at `s1:c10` (Procurement) has category set\n{c10}, which is a subset of {c1, c10}.\n\n**Result: A Privacy/Student Records user can read all Procurement data.**\n\nThis is not theoretical — it is how MCS set dominance works in the kernel. The process\ncategory set must be a superset of the object category set for read access. {c1, c10}\nis a superset of {c10}.\n\nSimilarly, any PROCURE subcategory using c10 as its base (e.g., `s1:c10,c11` for SBIZ)\nwould be dominated by any PRIV subcategory that includes c10.\n\nThe PRIV block is defined as c1-c9 (9 subcategories), but STUD is subcategory #10,\noverflowing into c10 which is the first category of the next block.\n\n**Recommended action:**\nOption A (minimal): Expand PRIV to c1-c10 and shift PROCURE to c11-c19 (cascading\nshifts required for all subsequent groups).\nOption B (clean): Redesign subcategory numbering to use relative offsets within the\ngroup base (e.g., PRIV subcategories at c2-c10, PROCURE base at c20, etc.), with\nwider block spacing.\nOption C (recommended): Switch to a compound-category scheme where subcategories use\nthe group base + a subcategory number from a dedicated subcategory range (e.g., c1 +\nc200..c299 for PRIV subcategories), completely separating group bases from subcategory\nnumbers. This eliminates overflow by construction.\n\n---\n\n## Finding 2 — CRITICAL: CRIT/WATER (c50) collides with INTEL group base (c50)\n\n```\nFile: data/MLS-setrans.conf (line 146), data/TARGETED-setrans.conf (line 122)\nLocation: CRIT block, subcategory WATER\nSeverity: CRITICAL\nControl reference: NIST SP 800-53 AC-4 (Information Flow Enforcement)\nRemediation owner: coder (JSON schema) + security-engineer (setrans)\n```\n\n**The collision:**\n- `s1:c40,c50` = `CUI//CRIT/WATER` (Water Assessments) — line 146 in MLS\n- `s1:c50` = `CUI//INTEL` (Intelligence) — line 151 in MLS\n\n**The access control consequence:**\nA process with clearance `s1:c40,c50` (CRIT/WATER) holds {c40, c50}. Until the\ncategory allocation is redesigned to prevent overflow, the MCS layout cannot provide\ncorrect access isolation between CUI groups.\n\n### Are the sensitivity level assignments correct for the Five Eyes model?\n\n**Mostly yes, with one critical ambiguity.** The US CUI model (all at s1) is correct.\nThe Canadian model (PA=s1, PB=s2, PC=s3) is correct in concept but the setrans\nencoding (different categories per tier) may break BLP dominance hierarchy if the intent\nis that higher tiers dominate lower tiers (Finding 7).\n\n### Concerns about the compound category approach for subcategories?\n\nThe compound category approach (group_base + subcategory_number) is sound in principle\nbut fragile in practice because:\n\n1. **No enforcement of block boundaries.** Nothing prevents a subcategory number from\n   exceeding the block size. This is the root cause of Findings 1-4.\n\n2. **Subcategory numbers overlap group bases by construction.** The subcategory for\n   PRIV/STUD is c10, which is PROCURE's group base. This is not a coincidence — it is\n   inherent in the sequential numbering scheme where subcategory numbers within a group\n   are drawn from the same global category namespace as group bases.\n\n3. **The fix requires either:** (a) wider block spacing (wasteful of the 1024 category\n   limit), (b) a two-namespace scheme where subcategory numbers are drawn from a\n   different range than group bases, or (c) accepting that the compound scheme limits\n   the number of subcategories per group to (next_group_base - current_group_base - 1).\n\n**Recommendation:** Document the maximum subcategory count per group as a hard constraint.\nFor the current 10-wide blocks, each group can have at most 9 subcategories (c1-c9 within\nthe block). Groups that exceed this (CRIT with 10, LGL with 12) must be allocated wider\nblocks.
