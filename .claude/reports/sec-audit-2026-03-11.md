**Finding SEC-01**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: lines 109–130 (SealingKey::derive)
Severity: HIGH
Finding: The sealing key derivation deviates materially from the pattern specification.
  The pattern spec (pattern-sec.adoc, "Sealing Key" section) defines two required
  entropy sources:
    (1) boot_id — the kernel-generated UUID at /proc/sys/kernel/random/boot_id
    (2) Process start time — the monotonic start time of the current process, from
        /proc/self/stat

  The implementation uses only (1) boot_id, combined with wall-clock subsecond nanos
  (SystemTime::now().duration_since(UNIX_EPOCH).subsec_nanos()). The process start
  time from /proc/self/stat is not used.

  This has two concrete security consequences:

  a) The timestamp used (subsecond nanos of wall-clock at construction time) is NOT
     the same as the process start time. It is the construction-time wall-clock, which
     could be identical for two processes constructed at the same nanosecond on the
     same boot. The process start time from /proc/self/stat is monotonically unique
     per process under the same kernel session.

  b) If SystemTime is unavailable, the timestamp contribution falls back to zero
     (line 119: .unwrap_or(0)). In that case the key degrades to SHA-256(boot_id ‖
     0x00 ‖ 0x0000000000000000), which is reproducible by any process that knows
     the boot_id and knows the clock was unavailable. While the attack surface
     is narrow (requires knowing clock failure occurred), the pattern spec does not
     contemplate this fallback.

  The doc comment on SealingKey::derive (line 101) correctly documents what the
  code DOES (wall-clock construction timestamp), but the module-level doc comment
  (line 31) claims "construction timestamp" without clarifying the deviation from the
  spec. The pattern spec is unambiguous: "Process start time — the monotonic start
  time of the current process, read from /proc/self/stat."

Recommended citation: NIST SP 800-53 SC-12 (key derivation strength)
Recommended remediation: Either (a) use the process start time from
  /proc/self/stat field 22 (start time in clock ticks) as the second entropy
  source per the spec, OR (b) update the pattern spec to formally document the
  deviation and justify it. If the spec is updated, the module-level doc comment
  must also reflect the accepted deviation.
Remediation owner: coder (if changing the implementation) / tech-writer (if
  updating the spec to match); decision required from owner first.
```

**Finding SEC-02**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: lines 607–648 (read_fips_at_init)
Severity: HIGH
Finding: The FIPS gate fails open, which is the wrong behavior for a FIPS-required
  deployment. Lines 625–629 and 632–637 both return `false` (caching allowed) on
  any read failure of /proc/sys/kernel/fips_enabled.

  The module-level doc comment (lines 22–26) accurately documents this:
  "If FIPS is active, the cache is disabled entirely." But the in-function comment
  at line 611 provides the justification: "fail-open on the FIPS check means we
  allow caching on systems where procfs is temporarily unavailable."

  The justification is internally consistent, but it conflicts with NIST SP 800-53
  SC-13, which requires using only FIPS-validated cryptographic modules on FIPS
  systems. If procfs is unreadable and FIPS is actually active, the cache will
  silently enable non-FIPS-validated HMAC-SHA-256 for the session.

  The pattern spec (pattern-sec.adoc, "FIPS Posture" section) states: "If no
  validated provider is available, the cache must be disabled." It does not address
  the procfs-read-failure case, leaving ambiguity.

  The practical risk on a real RHEL10 deployment is low (procfs is always present),
  but the current code violates the letter of SC-13 in a theoretical failure mode.
  The more conservative behavior — fail-closed on FIPS read failure (return true,
  disable caching) — is correct for a FIPS deployment. Fail-open on FIPS detection
  is the wrong direction for a high-assurance system.

Recommended citation: NIST SP 800-53 SC-13
Recommended remediation: Change read_fips_at_init to return `true` (FIPS active,
  disable caching) on any read failure. The log message on failure should be changed
  from debug to warn so the anomaly is auditable. Add a comment explaining the
  fail-closed rationale. Update the module-level doc comment accordingly.
Remediation owner: coder
```

**Finding SEC-03**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: lines 559–586 (decode_cached_result)
Severity: MEDIUM
Finding: The cache-on-hit design defeats the performance goal of the SEC pattern
  and introduces a correctness inconsistency.

  On a verified cache hit, decode_cached_result() re-runs the full pipeline (line
  585: self.detector.detect()) and clears the entry (line 584). The doc comment on
  this function acknowledges the design and calls it "intentionally conservative."

  From a security standpoint, the seal verification is correct — tampering IS
  detected. However, the design consequence is:

  1. Every query re-runs the pipeline, making the cache semantically equivalent
     to no-cache except that it detects tampering between queries.

  2. The entry is cleared after each hit (line 584), so the pipeline runs N times
     for N queries. The "cache" never actually reduces pipeline invocations.

  3. The pattern specification's flowchart (pattern-sec.adoc, "VERIFY → Valid →
     RET") shows the cached result being returned directly on a valid seal — NOT a
     pipeline re-run. The implementation contradicts the spec's documented behavior.

  The doc comment describes this as a "future iteration" improvement. This is
  acceptable as a documented deviation, but the pattern spec must be updated to
  reflect the current implementation's behavior so auditors and future developers
  understand what the SEC pattern actually does in this codebase.

  This is not a security regression (the seal still detects substitution), but
  it is a code-vs-documentation inconsistency at the specification level.

Recommended citation: (no new citation needed — existing SI-7 covers the tamper
  detection goal; the gap is design/doc consistency)
Recommended remediation: Update pattern-sec.adoc to document the current conservative
  implementation variant: "On cache hit, the pipeline re-runs to produce a typed
  result; the seal provides substitution detection, not I/O avoidance." Mark the
  full-serialization path as a planned future enhancement. The code comment in
  decode_cached_result is adequate — the spec must catch up to it.
Remediation owner: tech-writer
```

**Finding SEC-04**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: line 597 (ttl() accessor)
Severity: LOW
Finding: The `ttl()` method carries a bare `#[must_use]` without a message string.
  Per the Must-Use Contract Rule (CLAUDE.md), all #[must_use] annotations must
  include a message string explaining why the return value matters.

  Current annotation:
      #[must_use]
      pub const fn ttl(&self) -> Duration { ... }

  Required form:
      #[must_use = "TTL is a security parameter; ignoring it may mask misconfiguration"]
      pub const fn ttl(&self) -> Duration { ... }

  This is a policy violation, not a security gap — the Duration return is not
  security-critical on its own. The caching_enabled() method at line 591 has a
  correct message-bearing annotation; ttl() should match the style.

Recommended citation: NIST SP 800-53 SI-10 (per the Must-Use Contract Rule mapping)
Remediation owner: coder
```

**Finding SEC-05**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: lines 199–244 (build_payload / digest_evidence)
Severity: LOW
Finding: The evidence digest (digest_evidence) covers only three fields per record:
  path_requested, path_resolved, and parse_ok. The EvidenceRecord also contains
  source_kind, fs_magic, sha256, pkg_digest, opened_by_fd, and stat. An adversary
  who substitutes a record with the same paths and parse_ok outcome but a different
  fs_magic or sha256 would not be detected by the seal.

  The module doc comment (lines 257–264) accurately describes which fields ARE
  covered. The question is whether the omitted fields represent a meaningful
  tampering vector.

  Assessment: For the threat model defined (substituting TrustLevel, injecting a
  false OsRelease, persisting stale data), the three covered fields are the most
  load-bearing. The sha256 field in EvidenceRecord is already present in the
  payload indirectly through the evidence digest (if sha256 changes, the parsed
  content changes, which changes parse_ok or the os_release field in the payload).
  The fs_magic field is a weaker omission — an adversary could substitute a record
  claiming procfs magic without altering paths, and the seal would not detect it.

  This is a LOW finding because: (a) the adversary must already control in-process
  memory to substitute an EvidenceRecord; (b) the resulting TrustLevel would still
  be re-verified by the pipeline re-run on cache hit (due to SEC-03); (c) the doc
  comment is transparent about what is and is not covered.

Recommended citation: NIST SP 800-53 SI-7, AU-10
Recommended remediation: Consider extending digest_evidence to include fs_magic
  (encode as 8-byte LE, or 0xFF * 8 if None) and opened_by_fd (1 byte). This closes
  the substitution gap for those fields at negligible cost.
Remediation owner: coder
```

**Finding SEC-06**

```
File: components/rusty-gadgets/umrs-platform/src/sealed_cache.rs
Location: lines 494–498 (generic SealError arm in query_inner)
Severity: LOW
Finding: The catch-all error log at line 495 emits the SealError variant via {e:?}:
      log::warn!("SEC: seal verification error ({e:?}); discarding cache");

  SealError is a private enum with two variants: HmacInit and TagMismatch. Neither
  contains sensitive data today. However, the pattern of forwarding internal error
  details to log output should be scrutinized against NIST SP 800-53 SI-12
  (information management). The current variants are safe, but if SealError is ever
  extended with a variant that includes a payload excerpt (e.g., for diagnostics),
  this log line would immediately leak it.

  The TagMismatch arm (lines 486–491) handles the security-critical path
  separately with a pre-written message — that is correct. The generic arm should
  be similarly explicit rather than forwarding the Debug repr verbatim.

Recommended citation: NIST SP 800-53 SI-12
Recommended remediation: Replace the generic arm's log with a fixed-text message
  that does not forward the error Debug repr:
      log::warn!("SEC: seal error during verification (non-mismatch); discarding cache");
  Add a comment that SealError must never contain payload or key material.
Remediation owner: coder
```

**Finding SEC-07**

```
File: components/rusty-gadgets/umrs-platform/tests/sealed_cache_tests.rs
Location: module level (entire file)
Severity: MEDIUM
Finding: The test suite lacks a direct tamper-detection test. No test injects a
  modified payload or corrupted tag and verifies that the cache discards the entry
  and re-runs the pipeline. The comment at lines 26–31 acknowledges this and states
  that seal integrity is "verified through CacheStatus and caching_enabled paths."

  This reasoning is indirect: the tests confirm the structural properties
  (ZeroizeOnDrop compiles, TTL ceiling enforced, status coherence) but do not
  exercise the most critical runtime behavior — that a HMAC tag mismatch triggers
  discard and re-run rather than returning a stale result.

  The comment at line 24 states "the SEC module is tested structurally, not via
  fn-level test hooks." This is an acceptable justification IF the module exposes
  no test-hook API, which is confirmed (no pub(crate) test functions are declared
  in sealed_cache.rs). However, the absence of a tamper test should be documented
  as a known gap, not rationalized as adequate.

  A tamper test would require either:
  (a) A pub(crate) or #[cfg(test)]-gated method to corrupt the stored entry, OR
  (b) A test-only wrapper type that allows injecting a SealedCache with a known
      bad entry.

  The current design makes this impossible without modifying the module's privacy
  boundary — which is itself a security property. This creates a genuine testing
  tension that should be explicitly acknowledged and tracked.

Recommended citation: NIST 800-218 SSDF PW.4 (testing covers security properties)
Recommended remediation: Either (a) add a #[cfg(test)] method on SealedCache that
  corrupts the stored tag byte and expose it exclusively to the test binary, enabling
  a direct tamper test, OR (b) create a tracking task for the rust-developer to
  implement this test hook. Document the gap explicitly in the test file's module
  doc comment: "KNOWN GAP: no direct tamper-detection test exists due to SealedCache
  privacy boundaries. Tracked in task <N>."
Remediation owner: coder
```

Location: lines 201–210 (read_fips_from_proc helper)
Severity: LOW
Finding: The FIPS cross-check helper (read_fips_from_proc) uses
  std::fs::read_to_string, a path-based open with no provenance verification.
  The SEC module itself uses ProcfsText + SecureReader for the same read.

  For a test that is cross-checking the FIPS gate behavior of the SUT, using
  std::fs::read_to_string is acceptable — it reads the same path. However, the
  inconsistency means the test would not detect a scenario where ProcfsText rejects
  the path for a provenance reason (wrong filesystem magic) while std::fs::read_to_string
  succeeds. In that scenario, fips_gate_coherent would see the raw file content
  and expect caching_enabled to match, but SealedCache::new would have failed the
  FIPS read and defaulted to caching_enabled=true.

  This is a test-only concern; it does not affect production code.

Recommended citation: NIST SP 800-53 SI-7 (tests should verify the same trust
  boundary as production code)
Recommended remediation: Replace std::fs::read_to_string with a ProcfsText +
  SecureReader call in read_fips_from_proc, or add a comment explaining why the
  simpler path-based read is sufficient for cross-check purposes.
Remediation owner: coder

## Design Assessment: decode_cached_result

The current design re-runs the full pipeline on every cache hit (SEC-03). From a
security standpoint, this is strictly safer than the alternative — it means the
pipeline's trust-building logic runs on every call, so no stale trust tier can be
returned. The trade-off is that the performance benefit of caching is not realized.

This is an acceptable intermediate state provided:
1. The behavior is explicitly documented in the code (it is — lines 562–585).
2. The pattern spec is updated to reflect it (currently contradicted — see SEC-03
   and SEC-11).
3. A future task exists to implement true serialization/deserialization of
   DetectionResult so cached results can be returned directly.

The seal still provides value in the current design: it detects in-memory
substitution of payload bytes between queries, providing an additional tamper-
detection layer even though the "performance cache" benefit is deferred.
