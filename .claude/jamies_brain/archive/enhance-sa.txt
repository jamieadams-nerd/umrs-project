Security auditors

If you want your security-auditor agent to behave like a real on-site assessor, the key is to give it the same source material and mental model used by professional auditors and accreditation bodies.

A real auditor does not just read controls. They follow a formal assessment lifecycle:
	1.	Understand the system boundary and architecture
	2.	Identify the applicable control baseline
	3.	Review documentation and evidence
	4.	Perform technical verification
	5.	Record findings and residual risk
	6.	Support authorization/accreditation decisions

The best training corpus therefore includes three classes of material:
	1.	Security control frameworks
	2.	Audit methodology / assessment guides
	3.	Real accreditation process documentation

Below are the most valuable resources to feed into the agent’s knowledge base.

1. How Real Security Audit Findings Are Structured

Professional assessors typically record findings in a structured format like this:

Field	Meaning
Control ID	The control being evaluated
Control Description	Short summary of the control
Assessment Method	Examine / Interview / Test
Evidence Reviewed	What artifacts were evaluated
Finding	What was discovered
Severity	Risk level
Impact	What risk it creates
Recommendation	How to fix it
Status	Open / Closed / Mitigated

Example:

Control ID: AC-6
Control Name: Least Privilege

Assessment Method: Test

Evidence Reviewed:
- sudo configuration
- SELinux policy
- systemd service unit privileges

Finding:
Several services run as root where privilege separation is possible.

Severity: Moderate

Impact:
Increases attack surface for privilege escalation.

Recommendation:
Run services under dedicated service accounts.

Your AI auditor should generate exactly this style of record.

Excellent. Here is a UMRS-friendly model for a high-assurance automated inspection system that behaves much more like a real assessor than a generic scanner.

The core idea is this:

do not build “a scanner.”
Build an assessment engine that produces:
	1.	claims
	2.	evidence
	3.	findings
	4.	risk statements
	5.	exportable artifacts

That mirrors how RMF and real assessments work: assessors gather evidence using methods like examine, interview, and test, then roll the results into assessment artifacts and authorization decisions.  

1. High-level operating model

I would structure the auditor as five layers:

Layer A — Asset and boundary discovery

This determines what is in scope.

Examples:
	•	OS identity
	•	kernel version
	•	packages
	•	services
	•	listening sockets
	•	storage/mount layout
	•	trust boundaries
	•	management interfaces
	•	identities/roles
	•	security subsystems present

This aligns with RMF’s emphasis on system categorization, boundary definition, and control selection before assessment starts.  

Layer B — Evidence collection

This gathers raw, attributable facts.

Examples:
	•	/proc and /sys reads
	•	SELinux mode and loaded policy facts
	•	package provenance
	•	crypto policy / OpenSSL config
	•	journald / audit configuration
	•	systemd unit data
	•	file permissions and labels
	•	boot state
	•	immutable flags, mount flags, lockdown state

This is the “examine” and “test” substrate. In assessor language, you are collecting assessment objects and the evidence needed to support a conclusion.  

Layer C — Assertions and control mapping

This converts raw facts into security-relevant statements.

Examples:
	•	“Kernel lockdown is active.”
	•	“SELinux is enforcing.”
	•	“System-wide crypto policy prohibits legacy algorithms.”
	•	“journald is configured for persistent logging.”
	•	“Root login is disabled for SSH.”

Then map those assertions to controls such as AU, AC, CM, SC, SI, IA families. RMF and 800-53/53A are designed around exactly this kind of control-oriented evaluation.  

Layer D — Findings and risk logic

This is where the auditor becomes useful.

Examples:
	•	Assertion failed
	•	Assertion partially satisfied
	•	Assertion contradicted by another source
	•	Runtime state differs from configured/persisted state
	•	Control is present but weak
	•	Control claim exists in docs but no supporting evidence exists

That produces findings, severity, rationale, and remediation.

Layer E — Reporting and export

This produces both:
	•	human-readable reports
	•	machine-ingestible packages

For long-term interoperability, OSCAL is the strongest target because it is expressly designed for machine-readable SSP, SAP, assessment results, and POA&M content, and NIST/FedRAMP have been pushing that direction for several years.

2. The most important design principle: separate facts from judgments

Your engine should never jump directly from “I read a file” to “the system is secure.”

Use this chain:

source -> evidence -> assertion -> control evaluation -> finding -> risk

Example:

source:
  /sys/kernel/security/lockdown

evidence:
  "integrity"

assertion:
  kernel lockdown is enabled in integrity mode

control relevance:
  hardening / runtime attack surface reduction

finding:
  satisfactory for baseline X

This gives you traceability. It also makes contradiction handling much cleaner.

That is especially aligned with your assurance mindset: evidence provenance first, conclusions second.

3. Dual-truth model: runtime vs persisted configuration

This fits your existing thinking extremely well.

For many settings, the engine should collect two distinct views:

Ground truth

What is true right now
	•	actual runtime kernel state
	•	actual loaded SELinux mode
	•	actual mount options
	•	actual service state
	•	actual crypto provider state

Intended truth

What the system is configured to become
	•	kernel cmdline
	•	sysctl files
	•	config files
	•	policy modules
	•	unit files
	•	package policy
	•	bootloader config

Then compare them.

This is incredibly valuable because many audit failures are not “feature absent.” They are:
	•	configured but not active
	•	active but not persisted
	•	persisted but overridden at runtime
	•	documented but unimplemented

That kind of mismatch is exactly what real assessors care about when validating system claims against reality.

Useful fields:

5. Assertion model

Assertions should be distinct from evidence:

{
  "assertion_id": "as-lockdown-enabled",
  "statement": "Kernel lockdown is enabled",
  "derived_from": ["ev-kernel-lockdown-001"],
  "result": "true",
  "confidence": "high",
  "scope": "host",
  "control_refs": ["CM-6", "SI-7"],
  "rationale": "Kernel reports lockdown mode as integrity"
}

Why this matters:
	•	one assertion may rely on multiple evidence records
	•	one evidence record may support several controls
	•	assertions can be re-evaluated later as policies evolve

6. Finding model

This is where you make it assessor-like.

{
  "finding_id": "fd-ssh-root-login-001",
  "title": "SSH root login remains enabled",
  "status": "open",
  "severity": "high",
  "likelihood": "medium",
  "impact": "high",
  "affected_scope": "host:ssh",
  "control_refs": ["AC-6", "IA-2", "CM-6"],
  "evidence_refs": ["ev-sshd-config-001", "ev-sshd-runtime-002"],
  "summary": "SSH permits direct root login.",
  "recommendation": "Disable PermitRootLogin and validate effective daemon state."
}

The important part is that the finding references the evidence directly.

That gives you a path to:
	•	SAR-style outputs
	•	POA&M generation
	•	later re-test and closure

FedRAMP templates and OSCAL workflows are explicitly structured around SAP/SAR/POA&M style handoff.

7. Inspection passes that mirror a real assessor

I would make the auditor run in four passes.

Pass 1 — Boundary and inventory

Questions:
	•	What system is this?
	•	What is in scope?
	•	What security subsystems are present?
	•	What services and trust zones exist?
	•	What data paths matter?

Outputs:
	•	asset inventory
	•	boundary summary
	•	subsystem map
	•	interface map

Pass 2 — Control evidence collection

Questions:
	•	What direct evidence exists for each control family?
	•	What can be examined non-invasively?
	•	What is runtime truth vs persisted truth?

Outputs:
	•	evidence bundle
	•	assertion bundle
	•	contradiction list

Pass 3 — Evaluation and scoring

Questions:
	•	Is each assertion satisfied, failed, partial, or unverifiable?

	•	What claims are only documentary?

Outputs:
	•	findings
	•	severity/risk ratings
	•	control coverage summary

Pass 4 — Remediation and accreditation view

Questions:
	•	What blocks accreditation readiness?
	•	What is residual risk?
	•	What can be accepted temporarily?
	•	What belongs on a POA&M?

Outputs:
	•	POA&M candidates
	•	readiness summary
	•	export package

This lines up naturally with RMF’s assess / authorize / monitor flow.

10. Where UMRS can be genuinely differentiated

This is where your architecture can surpass ordinary scanners.

A. Provenance-aware evidence

Most tools record “result: pass/fail.”
You can record:
	•	where evidence came from
	•	how it was parsed
	•	what trust tier it has
	•	whether contradictory evidence exists

That is much closer to real assurance work.

B. Contradiction detection

Examples:
	•	config says enforcing, runtime says permissive
	•	bootloader says FIPS, userspace crypto posture says otherwise
	•	docs claim audit retention, journald storage says volatile
	•	service unit claims sandboxing, runtime privileges still broad

These contradictions are gold for auditors.

C. Mission-profile overlays

You could evaluate the same host against:
	•	baseline Linux hardening
	•	FIPS posture
	•	SELinux enforced posture
	•	UMRS high-assurance posture
	•	compartmented vault posture

That gives you layered accreditation views instead of one flat scan.

D. Evidence receipts

This is especially compatible with your receipt/provenance instincts.
Each evidence record or bundle could be signed, hashed, and chained.

That makes later review much stronger.
