Performance corpus

What you are describing is essentially the difference between passive reference material and an active knowledge index. Right now your agent has documents available, but unless it already knows what to look for, it won’t discover the new concepts inside them.

The solution is to introduce a knowledge extraction pass whenever new material is added. This turns raw documents into a structured capability index that the agent can consult quickly.

Below is a practical approach that works very well for agent-driven development teams.

⸻

1. The Core Idea: Pre-Digest the Corpus

When you add new material, run a task that:
	1.	scans the document
	2.	extracts important concepts
	3.	builds a small structured summary
	4.	stores that summary in a directory your agents can quickly consult

Instead of searching entire books, the agent first consults this knowledge digest.

Think of it like a table of contents written for the agent.

⸻

2. Example Workflow

When new material appears:

performance-corpus/new-docs/
    rust-perf-book.md
    drepper-memory.pdf

You trigger an agent task:

scan-new-knowledge

The Rust developer agent then:
	1.	reads the new document
	2.	extracts important ideas
	3.	generates a structured knowledge summary

⸻

3. What the Extracted Knowledge Should Look Like

The key is structured output.

Example:

developer-intelligence/performance-index/

Example file:

rust-perf-book.index.json

Example contents:

{
  "topics": [
    "allocation reduction",
    "memory locality",
    "branch prediction",
    "iterator fusion",
    "SIMD optimization"
  ],
  "key_techniques": [
    "Vec::with_capacity",
    "SmallVec",
    "bump allocation",
    "avoid bounds checks",
    "loop unrolling"
  ],
  "relevant_for": [
    "high-performance Rust",
    "low-latency systems",
    "CPU-bound workloads"
  ]
}

Now the agent has explicit awareness of the concepts contained in the material.

⸻

4. Build a Global Knowledge Map

Next, combine these extracted summaries into a master index.

Example:

developer-intelligence/
    knowledge-map.json

Example:

{
  "performance_topics": [
    "cache locality",
    "allocation strategies",
    "lock-free concurrency",
    "SIMD vectorization",
    "branch prediction"
  ],
  "algorithm_domains": [
    "graph traversal",
    "shortest path algorithms",
    "parallel search",
    "bloom filters"
  ]
}

This file becomes the awareness layer for the agent.

When solving problems it can think:

Does this relate to any known performance domains?

2. RAG Layer (Semantic Retrieval)

The second layer is your standard retrieval system.

When the agent asks a question like:

How do we reduce allocations in Rust loops?

The RAG system retrieves passages from:
	•	Rust performance book
	•	Rust compiler docs
	•	Tokio source comments

The benefit:

• natural language search
• contextual retrieval
• semantic matching

But RAG has a limitation:

The agent must ask the right question.

If the agent doesn’t know the concept exists, RAG may never retrieve it.

That’s the gap you correctly identified.

The layer you’re proposing sits between RAG and the agent reasoning system.
