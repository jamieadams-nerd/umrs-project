Good question. Here's the clean approach — do this right the first time.

## Plan

Create a proper confined SELinux user for `umrs-agent` without touching `jadams`, and give Claude Code a way to switch into that context for testing.

## Step 1: Fix `umrs-agent` shell temporarily

```bash
sudo usermod -s /bin/bash umrs-agent
```

## Step 2: Set the fcontext for home first

```bash
sudo semanage fcontext -a -t user_home_dir_t "/home/umrs-agent(/.*)?"
sudo restorecon -RFv /home/umrs-agent
```

## Step 3: Map to `user_u`

```bash
sudo semanage login -m -s user_u umrs-agent
# or -a if not already there
```

## Step 4: Fix the sudo rule for jadams

```bash
# /etc/sudoers.d/umrs-agent
jadams ALL=(umrs-agent) NOPASSWD: /bin/bash
```

## Step 5: Test transition via su

```bash
su - umrs-agent
id -Z
# Should show: user_u:user_r:user_t:SystemLow
```

## Step 6: Lock it back down

```bash
sudo usermod -s /sbin/nologin umrs-agent
```

---

Good instinct. On RHEL10 with targeted policy, regular users land in `unconfined_t` by default. You want `umrs-agent` mapped to a confined SELinux user.

## Step 1: Choose an SELinux user

Targeted policy ships these confined options:

| SELinux User | Good for |
|---|---|
| `user_u` | Minimal rights, no sudo, no su |
| `staff_u` | Can transition to `sysadm_r` via sudo |
| `sysadm_u` | Full admin, still confined |
| `guest_u` | Very restricted, no networking |

For a service/agent account, `user_u` is the right starting point — no privilege escalation paths.

## Step 2: Map the Linux user to an SELinux user

```bash
semanage login -a -s user_u umrs-agent
```

Verify:
```bash
semanage login -l | grep umrs-agent
```

## Step 3: Relabel the home directory

```bash
restorecon -RFv /home/umrs-agent
```

## Step 4: Confirm at login

```bash
su - umrs-agent -c "id -Z"
# Should show: user_u:user_r:user_t:s0
```

---

**If you eventually move to MLS policy** (which is your target), `user_u` maps cleanly — you'd add `-r s0` or a range like `-r s0-s0:c0.c1023` when you do `semanage login -a`. The groundwork you lay now carries forward.

What role does `umrs-agent` actually need to perform? That may refine the SELinux user choice.
